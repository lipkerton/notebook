Что такое `volume` в Docker? #flashcard
`volume` - это том (="хранилище"), который может независимо от контейнера сохранять какие-то данные. Например, если у нас есть БД в контейнере, то неплохо было бы хранить данные БД отдельно от контейнера с ней на случай, если контейнер понадобится переместить или перезапустить.

Что такое `network` в контексте Docker? #flashcard 
`network` - это сеть, в которой состоит контейнер. Сети нужны, чтобы контейнеры могли общаться друг с другом. Например, контейнер с БД и контейнер с приложением на FastAPI можно заставить общаться между собой, если объединить их в одну сеть - так они будут обмениваться сообщениями, будто стоят в одной системе.

Почему при вводе команды `git remote -v` отображаются именно пары ссылок?  #flashcard 
Потому что Git разделяет две операции - `fetch` и `push`. Это означает, что существует ссылка для чтения из репозитория и ссылка на запись в репозиторий. По умолчанию они одинаковые для каждой ссылки, но их можно настроить так, чтобы, например, изменения можно было записывать на один сервер, а читать можно было бы только с другого сервера.

Как мне сделать так, чтобы мои команды `git push` и `git pull` перестали просить указать ссылку на репозиторий, с которым я пытаюсь выполнить эти операции?  #flashcard 
Нужно установить "поток", т.е. указать локальному репозиторию место, которое он будет использовать по умолчанию, если нет других указаний:
```Bash
git pull --set-upstream <ссылка на репозиторий> <имя ветки>
git push --set-upstream <ссылка на репозиторий> <имя ветки>
```
При успешных операциях, Git запишет репозиторий, на который была указана ссылка, как репозиторий по умолчанию для всех будущих операций.

Что такое "индекс" в контексте Git? #flashcard 
Индекс - это промежуточная область между удаленным репозиторием и локальным репозиторием. Там хранятся все коммиты перед тем как они будут запушены в `main`. Благодаря индексу мы можем вносить изменения в историю коммитов и удалять из закоммиченых файлов какие-то вещи, которые нам не нужны на удаленном репозитории.

Как устроен Dockerfile? Как устроена механика создания образа из Dockerfile? #flashcard 
В Dockerfile прописывается последовательность команд. При создании образа результат каждого слоя (каждой отдельной команды) кэшируется. Если в слой будут внесены изменения (например, если изменятся файлы в директории на хосте, которые я копирую в контейнер), то весь слой и все слои после него будут перестроены.

В чем отличие между командами `RUN` и `CMD` в Dockerfile? #flashcard 
`RUN` запускается только единожды при инициализации контейнера, а `CMD` запускается только внутри контейнера, когда он уже запущен.
