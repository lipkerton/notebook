Почему запрос `select * from table` вреден? Назови основные причины. #flashcard

Как считается стоимость запроса в PostgreSQL? #flashcard 
По примерной формуле, которая заложена в оптимизатор: считаем I/O (input-output операции) и складываем с оценкой стоимости расходуемых ресурсов компьютера. Например:
```SQL
SELECT
	relpages, --количество страниц в таблице
	current_setting('seq_page_cost'), --стоимость чтения одной страницы
	relpages * current_setting('seq_page_cost')::int as total --итог
FROM pg_class where relname='species';
```
Так мы посчитаем стоимость прочтения всех страниц. Теперь нужно посчитать затраты процессора:
```SQL
SELECT
	reltuples, --количество строк
	current_setting('cpu_tuple_cost'), --стоимость чтения одной строки
	reltuples * current_setting('cpu_tuple_cost')::float as total
FROM pg_class where relname='species';
```

Что такое hash join? #flashcard

Тогда, чтобы соединить большую табличку с маленькой табличкой, нам нужно будет использовать хэш-функцию для перевода данных большой таблички в хэш, а затем сравнивать эти данные с данными маленькой таблички в оперативной памяти.

Что такое merge join? #flashcard

Используется реже, чем другие типы соединений, потому что оно предназначено только для тех случаев, в которых данные обеих таблиц отсортированы или могут быть очень просто отсортированы. Сравнение происходит по размеру столбцов соединяемых табличек: если столбцы одинаковы, то мы их соединяем, а если один или другой меньше, то идем по строчке дальше.

Что делает `INNER JOIN`? #flashcard 
Он объединяет два набора данных по какому-то признаку. В результирующем наборе данных будут только такие данные, которые соответствуют признаку с обеих сторон. Например, я хочу объединить наборы родов птиц и птиц: есть две таблицы - в одной птицы, а в другой их роды; они связаны между собой связью многие к одному (много птиц может соответствовать одному роду). Объединять буду по идентификатору рода с помощью `inner join`.
```SQL
select genus_name as "Род", species_name as "Птица"
from genus
join species on genus.genus_id = species.genus_id
order by genus_name, species_name;
```
Из двух наборов будут выбраны только те строки в которых есть совпадающие `genus_id` с двух сторон.

Что делает `LEFT JOIN`? #flashcard 
Он объединяет два набора данных по признаку. В результирующем наборе при этом будут все данные из левого набора данных и только некоторые данные из правого. Фишка в том, что из правого набора будут взяты только данные для которых нашлось совпадение в левом. Для тех данных из левой, для которых не нашлось совпадения в правой будут проставлены `null`.
```SQL
select genus_name as "Род", species_name as "Птица"
from genus
left join species on genus.genus_id = species.genus_id
order by genus_name, species_name;
```
<!--ID: 1748606029859-->

Он объединяет два набора данных по признаку. В результирующем наборе при этом будут все данные из правого набора данных и только некоторые данные из левого. Фишка в том, что из левого набора будут взяты только данные для которых нашлось совпадение в правом. Для тех данных из правой, для которых не нашлось совпадения в левой будут проставлены `null`.


Аномалия - это неожиданный результат, который возникает при попытке выполнить несколько одновременных запросов к одним и тем же данным.

Как называется самый "мягкий" уровень изоляции? Что на нем происходит? #flashcard 
`read uncommitted`. На нем, когда транзакция_1 вносит изменения в БД, транзакция_2 может видеть эти изменения до выполнения коммита транзакцией_1. Этот уровень отсутствует в PostgreSQL. Даже если буквально указать этот уровень при открытии транзакции, то ее поведение все равно будет соответствовать поведению транзакции с уровнем изоляции `read committed`.
<!--ID: 1750430415575-->


Как называется второй после самого мягкого уровня изоляции уровень? Что на нем происходит? #flashcard 
`read committed`. На нем транзакция_1 вносит изменения в БД, а транзакция_2 их не видит до коммита + она сама не может вносить изменения в те же данные, пока не будет закоммичена или откачена транзакция_1.
