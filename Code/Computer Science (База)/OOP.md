OOP = Object-Oriented Programming
на самом базовом уровне OOP - это о моделировании системы, которая состоит из объектов; каждый объект репрезентирует какой-нибудь конкретный аспект системы.
объекты могут содержать методы (функции) и данные. объект предоставляет публичный интерфейс для остального кода, но при этом имеет собственное приватное состояние, т.е. имеет какие-то внутренности, которые недоступны никакому коду, кроме того, что находится внутри класса.
перед тем как создавать объекты, мне нужно сначала написать свои классы. 
например, у меня есть задача описать учебную организацию. чтобы перечислить преподавательский состав, я сделаю класс `Professor`:
```Python
class Professor:

    def __init__(
        self,
        first_name: str,
        second_name: str,
        age: int,
        subject: str
    ) -> None:
        self.first_name = first_name.capitalize()
        self.second_name = second_name.capitalize()
        self.age = age
        self.subject = subject
    
    def __str__(self):
        return f"{self.first_name} {self.second_name}"
```
мой класс имеет конструктор класса - `__init__()`. конструктор класса будет задавать каждому новому объекту набор свойств - `first_name, second_name, age, subject`; и так же класс предоставляет каждому объекту один метод - `__str__()`.
на основании этого класса я могу создать конкретного профессора (конкретный объект):
```Python
professor = Professor(
	first_name="Олег",
	second_name="Колотыркин",
	age=35,
	subject="Математика"
) 
```
здесь понятно, что конкретный профессор является объектом, потому что он "объект класса `Professor`", но является ли сам класс объектом? я сейчас не говорю о перспективе Python, где "все является объектом", я говорю о перспективе OOP.
*в классическом OOP классы не являются объектами - это абстракции и шаблоны, которые нужны для создания объектов.*
# inheritence
один из принципов OOP - наследование (*inheritence*).
допустим, в моей модели обучения я хочу еще создать класс, который будет репрезентировать студентов:
```Python
class Student:

    def __init__(
        self,
        first_name: str,
        second_name: str,
        age: int,
        major: str
    ):
        self.first_name = first_name.capitalize()
        self.second_name = second_name.capitalize()
        self.age = age
        self.major = major

    def __str__(self):
        return f"{self.first_name} {self.second_name}"
```
теперь я могу создать новый объект от нового класса:
```Python
student = Student(
	first_name="Иван",
	second_name="Иванов",
	age=21,
	major="Компьютерные технологии"
)
```
на этом этапе у меня в скрипте написано уже два класса и это довольно много строк. сразу следует заметить, что между классами много общего: оба содержат в своих конструкторах три одинаковых атрибута, оба содержат одинаковый метод `__str__()`. как я уже сказал - класс в OOP является абстракцией для объектов; в данном случае у меня уже есть две абстракции (студент и профессор), теперь можно подумать: "А что есть общего у моих двух абстракций?"
именно в этом контексте появляется механизм наследования: в реальности абстракции *наследуются* друг от друга, например, есть понятие "машина" и понятие "седан" - любой седан является машиной и обладает свойствами машины, но сверху у него еще могут быть какие-то свойства, которые отличают его от других типов машин (например, от пикапа).
студент и профессор являются абстракциями кучи понятий: "персона", "человек", "существо" и прч. для кода лучше всего выбрать такое понятие, которое будет объединять все общие свойства между студентом и профессором. например, "персона":
```Python
class Person:

    def __init__(
        self,
        first_name: str,
        second_name: str,
        age: int
    ):
        self.first_name = first_name
        self.second_name = second_name
        self.age = age

    def __str__(self):
        return f"{self.first_name} {self.second_name}"
```
так как я поместил все, что является общим для студента и профессора, в один класс, теперь я могу выполнить наследование:
```Python
class Professor(Person):

    def __init__(
        self,
        first_name: str,
        second_name: str,
        age: int,
        subject: str
    ) -> None:
        super().__init__(
            first_name=first_name, second_name=second_name, age=age
        )
        self.subject = subject
    
class Student(Person):

    def __init__(
        self,
        first_name: str,
        second_name: str,
        age: int,
        major: str
    ):
        super().__init__(
            first_name=first_name, second_name=second_name, age=age
        )
        self.major = major
```
суть наследования в том, что каждый класс-наследник наследует все методы и может пользоваться всеми свойствами класса-родителя. в данном случае в конструкторе функция `super()` ссылается на родительский класс `Student/Professor` и использует конструктор класса `Person`, а метод `__str__()` и вовсе по умолчанию открыт для использования классами-наследниками, поэтому неявно присутствует в каждом.
другими словами, при наследовании у нас есть класс-родитель и класс-наследник. класс-наследник имеет доступ ко всем свойствам и методам класса-родителя, следовательно, все экземпляры класса-наследника будут иметь все свойства класса-родителя. это не работает в обратную сторону - класс-родитель не имеет доступа к методам и свойствам, которые являются уникальными для класса-наследника.
основной принцип наследования - хорошо проведенное наследование можно определить по тому, что экземпляр класса-наследника может заменить экземпляр класса-родителя в коде и при этом работоспособность программы не изменится.
# polymorphism
допустим у нас есть классы-наследники, а в классе-родителе написан метод `cool()`. вдруг мне понадобилось в одном классе-наследнике добавить к логике метода `cool()` какую-то фишку, которой нет ни в классе родителе, ни в другом классе-наследнике.
при том что я добавляю к логике `cool()` новую фишку, сама его функциональность не слишком отличается от функциональности аналогичного метода `cool()` в родительском классе и в другом классе-наследнике. т.к. функциональность не изменилась катастрофически, я не хочу делать из этого метода `cool()` другой метод - я просто переоформляю наследуемый метод `cool()`, т.е. оставляю один интерфейс для двух разновидностей метода.
такая махинация называется **полиморфизм**.
это когда у меня есть два связанных класса, у этих классов есть реализация одной логики в наследуемом от третьего класса методе. и при этом интерфейс один - метод называется одинаково в обоих классах:
```Python
class Person:
	...
	def cool(self):
		return [i for i in range(10)]

class Student(Person):
	...
	def cool(self):
		return [i + 1 for i in range(10)]

class Professor(Person):
	...
```
в примере, я переопределяю в классе `Student()` метод родительского класса `cool()`, но как видно разница не драматическая, поэтому я оставляю один интерфейс.
# encapsulation
как я сказал выше - объекты должны предоставлять внешний публичный интерфейс для того, чтобы сними общался весь остальной код, но еще у них должен быть своя приватная область, которая недоступна *снаружи*.
такая область содержит методы и свойства, которые доступны только внутри нее (поэтому она приватная).
создание приватных свойств у объектов - это **инкапсуляция**
в разных языках программирования есть разные способы реализации инкапсуляции. в общем случае мне просто нужно сделать так, чтобы те атрибуты и те методы, которые мне нужны только *внутри* данного класса не могли быть так просто изменены снаружи.
например:
```Python
class Person:
	
	def __init__(
		self,
		first_name: str,
		second_name: str,
		age: int
	):
		self.first_name = first_name
		self.second_name = second_name
		self.age = age
		
		self.sex = 'm'
```
допустим у класса `Person()` в конструкторе появился шовинистский атрибут `self.sex = 'm'`. теперь я сделаю экземпляр класса `Person()` и попробую изменить его атрибут через точечную нотацию:
```Python
person = Person(first_name="Юрий", second_name="Понамарев", age=45)
person.sex = 'w'
```
то у меня все успешно получится:
```Python
print(person.sex)  # => 'w'
```
что если я захочу запретить менять этот атрибут?
в Python невозможно полностью запретить менять атрибут извне, однако способы следовать принципу инкапсуляции есть, например, я могу "скрыть" атрибут выдав ему нижнее подчеркивание:
```Python
class Person:

    def __init__(
        self,
        first_name: str,
        second_name: str,
        age: int
    ):
		...
        self._sex = 'm'
```
то есть я просто поменял имя атрибута на другое. однако такое имя уже сложнее получить, потому что сложнее его угадать.
я могу отдавать атрибут по нормальному имени, если использую механизм `@property`:
```Python
class Person:
	
	def __init__(
        self,
        first_name: str,
        second_name: str,
        age: int
    ):
	    ...
	    self._sex = 'm'
	
	@property
	def sex(self):
		return self._sex
```
смысл такого подхода в том, что я отдаю "скрытый" атрибут по нормальному имени и при этом запрещаю менять этот атрибут по этому нормальному имени:
```Python
person = Person(first_name="Иван", second_name="Иванов", age=45)
print(person.sex)  # => 'm'
person.sex = 'w'  # => AttributeError: property 'sex' of 'Person' object has no setter.
```
ошибка `AttributeError` указывает на то, что не установлен некий `setter`.
`setter` - это механизм, который позволяет присваивать приватным атрибутам новые значения. в процессе присваивания мы так же можем настроить собственную валидацию.
например, я хочу, чтобы возраст не мог быть изменен на значение меньше нуля:
```Python
class Person:
	
	def __init__(
        self,
        first_name: str,
        second_name: str,
        age: int
    ):
	    
	    ...
	    self._age = age
	
	@property
	def age(self):
		return self._age
		
	@age.setter
	def age(self, value):
		if value > 0:
			self._age = value
			return self._age
```
если я попытаюсь заменить значение `age` на ноль, то `setter` просто пропустит эту попытку мимо, как будто ее не было.
