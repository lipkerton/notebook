# база

HTTP - Hyper Text Transfer Protocol, который изначально использовался для отправки текстовых страниц по сети. Почему он называется “Hyper Text”, а не просто “Text”? Потому что отправляются не просто странички, а странички, которые могут содержать в себе ссылки на другие странички - такие ссылки называются “гипер текст”.

HTTP - это клиент-серверный протокол. Это означает, что он помогает доставить данные на сервер и вернуть с него. В качестве клиента чаще всего выступает конкретный web-браузер. Клиент шлет запрос на сервер, сервер отвечает HTML страницей или каким-нибудь ошибочным статус-кодом.

HTML - Hyper Text Markup Language. Это язык разметки интернет-страничек, которые отправляются по протоколу HTTP. Раньше это был просто текст в формате XML, но теперь в этот же документ можно вставить CSS-стили, JavaScript код или разместить ссылки на отдельные документы с ними.

Три основных компонента в изначальной задумке HTTP:

- HTTP
- HTML
- URL
- (((сервер)))
- (((браузер)))

Последние два написаны так, потому что иногда они не упоминаются, т.к. очевидно, что они нужны, потому что без клиента нельзя отправить запрос на сервер.

А что такое URL? Uniform Resource Locator. Мы привыкли, что это просто какая-нибудь ссылка на какой-нибудь ресурс, например:

```HTML
https://habr.com/ru/articles/813395/
```

На самом деле в этом адресе есть несколько частей, которые относятся вообще к разным сферам. `https://` - это имя используемого протокола, для обращения к серверу. `habr.com` - доменное имя ресурса, по нему мы получаем доступ к самому серверу. Но мы же знаем, что на сервере есть еще куча мест куда можно зайти, такие места условно называются _ресурсы_. Чтобы сервер понял куда конкретно мы хотим попасть, после адреса сервера пишется еще куча информации, которая указывает на конкретный ресурс на сервере, т.е. это адрес нужных данных на сервере.

Кстати, URL может ссылаться не только на страницу формата HTML - он может ссылаться на любые данные, которые есть на сервере, например, на текстовый файл или даже на картинку.

HTTP/1.1 и HTTP/2 используют протокол TCP для обмена сообщениями. HTTP/3 использует протокол QUIC, чтобы быстрее доставлять сообщения. Так же первые две версии обменивались сообщениями, которые парсились в нормальные читаемые строки, когда приходили на сервер - это было нужно для упрощения чтения содержимого запросов; но эта ситуация изменилась, когда вышла третья версия протокола - там все строки зашифрованы в бинарный формат, т.е. они доставляются и отправляются быстрее.

# что конкретно он делает?

HTTP пакует сообщение клиента и кучу дополнительной информации в бинарный формат. Этот пакет отправляется по сети и расшифровывается обратно из бинарного формата в человеческий. Что в этом пакете? Это обычный текстовый файл - в нем, как в словаре даны заголовки HTTP-ответа или заголовки HTTP-запроса. Заголовки содержат всю-всю информацию о пакете. Какие-то из заголовков являются нужными, а какие-то нет. Например, помимо текста самого сообщения там храниться информация о типе сообщения, статус-код ответа и прч.

## как написать запрос?

Запрос от клиента к серверу состоит из трех частей:

- Запрос
- Заголовки
- Тело сообщения

Из них обязательна только первая часть - запрос. Теперь нужно задать крутой вопрос: а где мне писать сырой запрос? Понятно, что, когда я пишу из браузера я отправляю запросы, но я хочу сам разобраться - отправить “сырой” запрос.

### telnet

Это консольная утилита и с ее помощью можно написать самый простой запрос:

```Bash
telnet www.google.com 80
HEAD / HTTP/1.1
```

`HEAD` - это HTTP-метод. Когда мы пользуемся таким, то ждем, что сервер в ответ вернет специфическое сообщение. В данном случае мы ожидаем только заголовки, без самой страницы.

`/` - этот символ означает, что мы запрашиваем с сервера информацию о его корневой папке. По сути здесь вставляется путь к нужному нам ресурсу на сервере.

`HTTP/1.1` - это используемая версия протокола.

Что вернется? Вернется список всяких заголовков.

В самом верху вернулось имя протокола `HTTP/1.1` и статус-код `200 OK` .

### curl

В telnet я написал запрос HEAD к ресурсу, а вот консольная утилита curl может отправлять автоматически запрос GET.

```Bash
curl http://www.example.com
```

Такая команда выполнит запрос к ресурсу и вернет полный HTML-документ ресурса. Метод GET в отличие от метода HEAD возвращает не заголовки, а тело запроса - саму страницу.

Можно воспользоваться методом HEAD.

```Bash
curl --head http://www.example.com
```

Ответ будет примерно таким же, как в случае с telnet.

```Bash
HTTP/1.1 200 OK
Accept-Ranges: bytes
Content-Type: text/html
ETag: "84238dfc8092e5d9c0dac8ef93371a07:1736799080.121134"
Last-Modified: Mon, 13 Jan 2025 20:11:20 GMT
Content-Length: 1256
Cache-Control: max-age=600
Date: Sun, 23 Mar 2025 06:49:34 GMT
Connection: keep-alive
```

>[!important]На питоне так же реализованы библиотеки для общения с клиент-серверной архитектурой и HTTP [[веб-клиенты и веб-серверы|веб-клиенты и веб-серверы]].

## статус-код

Статус-код - это механизм, который сигнализирует о том, что соединение состоялось или о том, что произошла ошибка. Таких кодов много, но они все делятся на пять групп.

- 1XX - у сервера есть дополнительная информация для клиента
- 2XX - все прошло хорошо
- 3XX - переадресация (сервер переехал на другой ресурс и перекидывает все запросы туда)
- 4XX - ошибки на стороне клиента
- 5XX - ошибки сервера

Статус-коды приходят всегда вместе с заголовками и всегда подписаны фразами капсом, например, `404 BAD REQUEST` .
