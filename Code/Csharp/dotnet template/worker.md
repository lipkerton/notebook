я хочу создать сервис, который будет постоянно включен. демон, другими словами.
чтобы реализовать такую штуку в .NET уже есть шаблон - `worker`:
```C#
dotnet new worker -o MyWorker
```
эта команда создаст директорию сервиса и сразу напишет два основных файла - `Program.cs` и `Worker.cs`.
# Program.cs
внутри `Program.cs` такой текст:
```C#
using InterfaxTalker;

var builder = Host.CreateApplicationBuilder(args);
builder.Services.AddHostedService<Worker>();

var host = builder.Build();
host.Run();
```
для начала я хочу изменить `var` на типы из документации (https://learn.microsoft.com/en-us/dotnet/core/extensions/workers):
```C#
HostApplicationBuilder builder = Host.CreateApplicationBuilder(args);
...
IHost host = builder.Build();
```
так будет понятно, что я использую класс:
- `HostApplicationBuilder` - позволяет строить сервисы, смысл которых - работать фоново в системе (что идеально подходит для моей задачи). именно через него я буду настраивать свой процесс.
- `IHost` - это обертка, которая заворачивает конфигурацию в готовый хост. проверяются зависимости и фиксируется конфигурация.
что вообще происходит сейчас в таком коде?
для начала инструкция:
```C#
using InterfaxTalker;
```
сообщает, что я подключаю пространство имен `InterfaxTalker` - директорию текущего проекта. делаю я это, чтобы рабочий файл `Program.cs` видел другой рабочий файл `Worker.cs`.
далее:
```C#
HostApplicationBuilder builder = Host.CreateApplicationBuilder(args);
```
когда я пишу `CreateApplicationBuilder(args)` на самом деле я инициирую ряд автоматических действий. класс `HostApplicationBuilder` должен хранить внутри себя конфигурацию приложения, а `CreateApplicationBuilder(args)` должен собрать и подготовить конфигурацию приложения. настройки лежат в трех местах - переменные окружения, файл `appsettings.json` и аргументы командной строки `args`; `CreateApplicationBuilder` соберет их все в одном месте и передаст в `builder`, которая является экземпляром класса `HostApplicationBuilder`.
помимо сборки конфигурации, я так же получаю логгирование, настройки для которого по умолчанию уже прописаны в `appsetttings.json`:
```JSON
"Logging": {
	"LogLevel": {
		"Default": "Information",
		"Microsoft.Hosting.Lifetime": "Information"
	}
}
```
и так же я получаю контейнер зависимостей *DI*, куда буду складывать свои сервисы (свои классы), где выполняется основная работа. фишка этого контейнера в том, что он будет автоматически создавать экземпляры моих сервисов и запускать внутри них метод `ExecuteAsync` в фоновом потоке. другими словами, мне нужно запихнуть внутрь такого контейнера свой класс, где прописана логика программы, и контейнер далее сам выполнит все рутинные действия - создаст экземпляр и выполнит нужный метод.
регистрация сервиса происходит так:
```C#
builder.Services.AddHostedService<Worker>();
```
когда подготовка завершена (конфигурация собрана и сервисы добавлены) я могу сделать экземпляр хоста:

# Worker.cs
этот файл будет выполнять основную работу.
первая строка в нем (кроме указания на имя пространства):
```C#
public class Worker : BackgroundService {...}
```
реализация наследования - класс `Worker` наследуется от класса `BackgroundService`. класс `BackgroundService` - это основной класс для создания фоновых процессов в C# и он берет на себя всю логику, которую подразумевает фоновая программа (например, создание потоков и т.д.).
далее нужно понять как у классов в C# выглядят конструкторы. а они обычно выглядят так:
```C#
public class Worker : BackgroundService
{
	public Worker(ILogger<Worker> logger)
	{
		...
	}
}
```
фишка *DI (Dependency Injection)*, который я обсуждал ранее, в том, что такая механика как бы отдает зависимости самостоятельно.
например, в данном случае, я объявляю конструктор класса с требованием передачи ему на вход логгера, который был бы экземпляром класса `ILogger`. так как, класс `Worker` является частью *DI* контейнера, то *DI* контейнер при объявлении экземпляра класса должен подготовить и передать зависимость на вход конструктору класса.
как и в Python конструкторе, внутри конструктора C# я должен присвоить входящий аргумент какому-нибудь атрибуту объекта класса. но перед этим я должен объявить атрибут класса, поэтому перед конструктором поставлена строка:
```C#
private readonly ILogger<Worker> _logger;
```
кстати, приставка `<Worker>` нужна классу `ILogger`, чтобы было понятно к какой категории принадлежит сообщение из лога (в данном случае сразу будет понятно, что лог возвращает итог работы класса `Worker`).
когда я запускаю программу она должна вызвать метод `Run()` метод `Run()` должен создать экземпляр класса `Worker()` и вызвать метод `ExecuteAsync()`. внутри метода `ExecuteAsync()` прописан цикл, который и будет крутиться, пока пользователь не прервет выполнение программы.
для начала нужно посмотреть на объявление цикла:
```C#
protected override async Task ExecuteAsync(CancellationToken stoppingToken) {...}
```
слово `protected` - это разновидность директивы `private`. если `private` говорит, что метод может быть использован, только если он вызван из текущего класса, то `protected` говорит, что метод может быть использован текущим классом и всеми его наследниками.
`override` - директива, которая переписывает логику `ExecuteAsync`. она нужна, потому что такой метод объявлен в родительском классе, а я пытаюсь переделать его под себя.
что такое `async`?
это волшебное слово объявляет метод асинхронным.
у компилятора C# есть пул потоков. когда начинает выполнятся метод `async`, он занимает один из этих потоков и идет выполнять логику метода. когда компилятор натыкается на слово `await` он понимает, что нужно подождать пока данная строка вернет данные. однако в данном случае он не ждет буквально - он запоминает место, на котором остановил выполнение метода и уходит из него заниматься другими задачами.
а что такое `Task`?
`Task` - это объект, который возвращает метод, вызванный внутри метода `async`. фишка такая, что при вызове другого метода мы можем не получить ответ сразу, поэтому вместо ответа нам сразу вернется объект `Task`, как "номерок" запущенной задачи.
когда я вызову несколько методов и все они вернут `Task`, я могу собрать все `Task` в один список и попросить компилятор подождать их выполнения с помощью `await`.
здесь:
```C#
if (_logger.IsEnabled(LogLevel.Information))
{
	_logger.LogInformation("Worker running at: {time}", DateTimeOffset.Now);
}
```
включены ли логи уровня `INFO` и если включены, то прописываю в лог строку, а если нет, то ничего не делаю. 
и, наконец, инструкция:
```C#
await Task.Delay(1000, stoppingToken);
```
она запускает ожидание всех задач и ставит таймаут на каждую $1000$, по истечению таймаута программа будет остановлена.
