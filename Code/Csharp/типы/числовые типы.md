# база
наиболее часто используемый тип - `int`. он занимает 4 байта.
наиболее используемый он потому, что наиболее универсальный среди всех систем.
оказывается у типов большого размера есть проблемы с интеграцией на разных платформах. большим размером считается размер от 32 бит. главной же проблемой является чтение из памяти и запись в памяти - мы не можем атомарно выполнять эти операции с типами, вес которых больше 32 бит.

```C#
using System;
...

```
здесь следует обратить внимание на то, что `UInt16` это *имя в CLR*.
здесь нужно помнить, что CLR создана не только для одного C#. внутри нее существует собственный интерфейс типов, который просто упрощен в C#, чтобы код C# был больше похож на то, что разработчики привыкли видеть в C и C++.


но можно использовать и то, и другое - между `ushort` и `System.UInt16` нет разницы.

---
*CTS (common type system)* - это общая система типов среды выполнения CLR. она содержит в себе полный набор поддерживаемых типов данных, правил их создания и управления ими.
*CLS (common language specification)* - это ограниченное подмножество типов, правил и т.д. внутри *CTS*. такое подмножество требуется поддерживать всем языкам, которые пользуются средой выполнения CLR.
например, не все языки .NET поддерживают числа без знака и CLR не заставляет каждый язык реализовывать сложную систему типов - CLR требует лишь поддержки ограниченного подмножества (*CLS*).
>[!important] целые числа со знаком входят в множество *CLS*, а целые числа без знака уже нет, поэтому иногда в реализации некоторых фишек в C# могут встречаться странности. например, свойство массива `Length` является `int`, хотя длина массива никогда не может быть отрицательной.

---
С# так же поддерживает числа с плавающей точкой. Существуют два типа - `float` и `double`. они являются 32-битные и 64-битные числа стандартного формата IEEE 754.
числа `float` и `double` соответствуют числам с одинарной и двойной точностью.
числа с плавающей точкой работают не так, как целые числа, поэтому таблица типов будет отличаться:
![[floating_point_type.PNG]]
такие числа достаточно крупные (размером с обычный `int` в случае `float` и размером с `long` в случае `double`), потому что память, которая под них выделяется хранит не только сами значения, но и их точность.
как можно заметить сразу, не бывает чисел с плавающей точкой без знака. следовательно, первый бит из $32$ или $64$ всегда является указателем на знак. указатель на знак - это первая часть числа с плавающей точкой в памяти, далее идет экспонента (порядок) и мантисса (значащая часть).
*экспонента* - это смещенная  степень двойки.
*мантисса* - дробная часть числа.
как это выглядит на практике? допустим, у меня есть число $12.375$ и я хочу посмотреть на его битовое представление.
сначала мне нужно отдельно перевести целую и дробную части:
$$
12 = 00001100
$$
если с целой все понятно, то как я могу перевести дробную?
$$
0.375 \times 2 = 0.75 \implies 0
$$
$$
0.75 \times 2 = 1.5 \implies 1
$$
$$
0.5 \times 2 = 1 \implies 1
$$
каждый раз я умножаю дробную часть на два. в результате получаю значение одного бита, а остаток от умножения я снова умножаю на два и так, пока остатком не будет ноль.
в итоге, если сложить целую и дробную части, то получится число $1100.011$. на этом рассчеты не заканчиваются, т.к далее следует провести *нормализацию*, т.е. найти *мантиссу* и *экспоненту*. на практике это означает, что мы приводим число в двоичной системе к виду:
$$
1.\\x\\x\\x\\x\dots \times 2^n
$$
я не буду вдаваться в супер-математические подробности, мне сейчас нужно подвинуть в числе $1100.011$ запятую так, чтобы получилось число $1.100011$, т.е. на три знака вперед:
$$
1100.011 = 1.100011 \times 2^3
$$
мантиссой здесь является значащая часть числа (часть числа без единицы) - $100011$, а экспонентой степень двойки - $3$.
схема `float` числа в памяти выглядит так:
**1 бит знак | 8 бит экспонента (со смещением) | 23 бита мантисса (дробная часть)**
поэтому у меня получится в памяти примерно такое число:
- знак - 0 (число положительное)
- экспонента = $127 (смещение) + 3 = 130 \implies 10000010$
- мантисса = $100011$
итог:
$$
010000010100011
$$
так как ячейка памяти под `float` выделяется с размером в 32 бита, то оставшиеся свободными биты будут заполнены нулями с конца.
я отвлекся от C#.
короче, тут, так же как и с целыми числами тоже есть CLR имя и тип в C# - можно использовать оба.

# числовые преобразования
каждый из встроенных числовых типов для хранения чисел в памяти использует собственное представление. преобразование числа и одной формы в другую требует специальных инструкций (потому что это отдельный набор инструкций).
как обычно, существует явное преобразование и преобразование неявное:
```C#
int a = 1;
double b = 2;
float c = a;
Console.WriteLine(a / b);
Console.WriteLine(c * b);
```
как видно, я не делаю никаких специальных преобразований для выполнения операций с числами, которые принадлежат к разным типам.
однако такое преобразование так же протекает по определенным правилам, т.к. C# при преобразовании выбирает тип с наибольшим диапазоном (бит в памяти) и переводит в него числа с меньшим диапазоном. например, `double` занимает в памяти 64 бита, поэтому все остальные участники арифметической операции будут сведены к нему.
так как у нас есть правила, значит есть преобразования, которые C# не может выполнить неявно. это преобразования из типов с большим диапазоном в меньший:
```C#
int a = 42;
int WillFail = 42.0;
int AlsoWillFail = a / 1.0;
```
такой код вернет ошибки компиляции:
![[floating_point_type2.PNG]]
однако такой алгоритм будет валидным, если я попытаюсь выполнить преобразование явно:
```C#
int a = 42;
int WillFail = (int) 42.0;
int WillAlsoFail = (int) (a / 1.0);
```
теперь следует задаться вопросом: *если C# приводит числа к типу с наибольшим диапазоном, то к какому типу будут приведены числа с одинаковым диапазоном?*
это нужно проверить:
```C#
int a = 42;
float b = 42;
Console.WriteLine(a / b);
```
ничего плохого не случается.
вероятно вступают какие-то тайные техники преобразования.
>[!important] есть очень большие числа типа `float`, которые находятся вне диапазона типов `int` и `uint`.
# проверенные контексты
в C# есть ключевое слово `checked`, которое можно разместить перед оператором блока или перед выражением, сделав его проверенным контекстом.
это означает, что арифметические операции будут провереяться на арифметическое переполнение. арифметическое переполнение - это когда величина значения больше или меньше, чем его тип может вместить.
```C#
uint a = uint.MaxValue;

checked {
	Console.WriteLine(a + 3);
}
```
вывод будет таким:
![[overflow1.PNG]]
кстати, здесь я применяю `checked` к блоку кода, но следует знать, что действие `checked` не распространяется на код, который реализуют посторонние функции. если в них произойдет переполнение, то `checked` никак на это среагировать не сможет.
короче, оператор `checked` помогает предотвращать (или позволять) переполнение при приведениях и арифметических операциях.
что будет, если я не проверю с помощью `checked` сумму максимального значения типа и тройки?
```C#
uint a = uint.MaxValue;
unchecked
{
	Console.WriteLine(a + 3);
}
checked 
{
	Console.WriteLine(a + 3);
}
```
для целых чисел это приводит к тому, что число "переворачивается" - если прибавить единицу к максимальному значению типа, то получится минимальное значение (единица), а если добавить тройку, то получится:
![[overflow2.PNG]]
двойка.
так же `checked` можно применять к арифметическим операциям, где может возникнуть переполнение:
```C#
uint a = uint.MaxValue;
uint b = 1;
uint c = 2;
uint result = checked(a + b) + c;
Console.WriteLine(result);
```
здесь код опять покажет ошибку `OverflowException`.
>[!important] можно настроить применение `checked` по умолчанию, если в файле `.csproj` добавить строку `<CheckForOverflowUnderflow>true</CheckForOverflowUnderflow>` внутрь тега `<PropertyGroup>`. за это нужно будет заплатить заметную цену - проверка замедлит выполнение операций.
# BigInteger
это потенциально ничем не ограниченный тип данных. это означает, что в него можно записать вообще любое число, однако следует помнить, что у этого действия есть накладные расходы. для того, чтобы позволить запись в себя неограниченного числа `BigInteger` выполняет какие-то секретные махинации при компилировании - если число маленькое, то задержка не будет заметной, но она будет расти каждый раз, когда значение увеличивается. так происходит потому, что `BigInterger` увеличивается только с ростом своего значения.
вот бесконечный цикл чисел фибоначчи:
```C#
static void Main(string[] args)
{
	BigInteger i1 = 1;
	BigInteger i2 = 2;
	Console.WriteLine(i1);
	int count = 0;
	while(true)
	{
		if (count++ % 100000 == 0)
		{
			Console.WriteLine(i2);
		}
		BigInteger next = i1 + i2;
		i1 = i2;
		i2 = next;
	}
}
```
сначала я объявляю два числа типа `BigInteger` - `i1` и `i2`. затем делаю счетчик. бесконечный цикл будет складывать оба числа, записывать в первое число значение второго числа, а во второе их сумму.
каждый раз когда счетчик будет делиться на сто тысяч, я буду выводить его значение в консоль.
фишка программы в том, что числа довольно быстро станут очень большими и это даже загрузит процессор.
