в C# представлен тип `string`. в CLR он называется `System.String`. этот тип описывает "текст". строка же с C# состоит из символов типа `char`. каждый символ `char` - это 16-битовое значение.
ошибка думать, что каждый `char` представляет собой символ. хотя чаще всего это так, бывают и исключения. нужно помнить, что существует огромная таблица кодов символов Unicode и символ может состоять из нескольких позций в ней.
например:
```C#
char[] chars = {'c', 'a', 'f', 'e', (char) 0x301, 's'};
string text = new string(chars);
Console.WriteLine(text);
```
в массиве символов типа `char` я написал шесть объектов, но на самом деле их пять, просто я использовал апостроф над буквой *e*:
![[string1.PNG]]
с такими штуками нужно быть осторожнее, т.к. если я сейчас напечатаю такую строку в обратном порядке, то получится так, что апостроф будет стоять над буквой `s`.
вообще, с Unicode есть масса проблем, которые так или иначе все уже были решены. например, символов в Unicode так много, что некоторые перестали помещаться в стандартные 16 бит размера `char`, поэтому существуют кодировки типа UTF-16.

строки в C# неизменяемы. механика примерно такая же, как в Python - если происходит попытка изменения строки, то в памяти просто создается новая строка. поэтому операции над строками могут быть затратными. сборщик мусора тратит много ресурсов, чтобы собирать устаревшие строки.
если нужно создать изменяемую строку, то лучше использовать `StringBuilder`. он концептуально похож на строку - он является последовательностью значений типа `char`, но при этом значения `StringBuilder` можно изменять.

форматирование строк можно выполнить в C# путем позиционных замен и путем прямой вставки значения в строку.
самый простой способ:
```C#
string name = "Петр";
string text = $"Привет, {name}";
Console.WriteLine(text);
```
это практически то же самое, что и f-строка.
для особых случаев (позиционные замены) существует `string.Format()`:
```C#
string name = "Петр";
string text = string.Format("Привет, {0}", name);
Console.WriteLine(text);
```
но в данном случае нужно быть осторожнее, потому что можно ошибиться с подстановкой, особенно если аргументов много. ну и еще это сложно читать.
при использовании `Format` и прямой вставки значений можно использовать спецификторы типа - это штуки, которые позволяют взять, например, `float` число, перевести его в строковое представление и обрезать пару знаков после запятой:
```C#
string name = "Петр";
float age = 121.4534534985f;
string text = string.Format("Привет, {0}! Мне {1:f1}", name, age);
Console.WriteLine(text);
```

существует термин - *буквальный строковый литерал*. это строки вида `@"Hello"`. их фишка в том, что они позволяют улучшить читаемость строк, где используется символ `\` и так же они позволяют делать многострочную строку.
как именно они улучшают читаемость строк `\`? в обычной строке символ `\` используется, как символ экранирования, например, для создания табуляции `\t` или для экранирования кавычек; в том числе поэтому описание пути в Windows выглядит примерно так: `C:\\Temp\\Something` - здесь символ `\` экранирует другой символ `\`.
иногда мне не нужно поведение экранирования и именно в таких случаях я могу использовать буквальный строковый литерал:
```C#
string text = @"C:\Users\Peter\My";
Console.WriteLine(text);
```
и еще я могу делать многострочные строки:
```C#
string text = @"
    C:\Users\Peter\My
";
Console.WriteLine(text);
```

тут, кстати, есть интересный момент, о котором я раньше не знал.
в Windows для переноса строки используется два символа - `CL` (13) и `LF` (10). они обозначают возврат каретки и перенос строки. это наследие старых систем, где для начала новой строки нужно было перемещать печатающую штуку в начало строки, а затем переместить лист для печати вверх.
в UNIX такого нет - там используется только один символ для переноса.
поэтому здесь появляется проблема - разные производные от UNIX систем сервисы, например, Git, могут вступать в конфикт с соглашениями, которые приняты в Windows. поэтому лучше всегда держать это в голове, а для гита использовать `.gitattributes`, где можно прописать поведение перевода символа переноса из одной системы в другую.
