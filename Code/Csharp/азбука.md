# директива using
В `C#` существует термин - *пространство имен*. Локально он описывает набор имен, которые задаются нами самими в проекте. Это означает, что в пространство имен данного проекта будут входить все поименованные сущности, например, переменные, списки, функции и т.д.
В [[си шарп и .NET]] я писал об использовании одним проектом метода `Main`, который находится в другом проекте. Там я применил `using`, чтобы написать:
```C#
using app;
...
Program.Main(new string[0]);
```
Как раз на этом примере видно применение директивы `using` - она открывает доступ к именам иного пространства имен.
Глобально пространства имен решают две проблемы - проблема уникальных имен в огромном количестве библиотек `C#` и проблема поиска необходимого интерфейса среди гигантского количества интерфейсов.

В книжке написано: 
> Большинство типов .NET определены в пространстве имен. 

И таких пространств имен существует невероятно много - они написаны другими разработчиками для расширения инструментария `C#` и иногда упорядочены странно. Например, существуют пространства имен внутри пространств имен - уровни вложенности могут быть любыми.
Основные пространства:
- `System` - пространство имен, которое содержит типы, являющиеся частью `.NET`.
- `Microsoft` - пространство имен, которое содержит типы, являющиеся частью какой-нибудь технологии Microsoft.
Так же существуют библиотеки других компаний, которые называются по имени компании.
Крайне рекомендуется помещать собственные типы в пространства имен. `System` не является специальным закрытым пространством имен, поэтому собственные типы можно присваивать и ему.
Так же как и в Python в `C#` можно выдавать псевдонимы для директив `using` - такое делается, чтобы не возникало совпадения между одинаковыми именами из двух пространств имен + не нужно было писать полный путь до нужного имени:
```C#
using System.IO;
using System.Windows.Shapes;
using IoPath = System.IO.Path;
using WpfPath = System.Windows.Shapes.Path;
```
# `namespace`
Директива `using` показывает какие имена наша программа будет использовать. А команда `namespace` указывает какое пространство имен мы хотим создать:
```C#
namespace TestSpace 
{
	<код в пространстве имен>
}
```
По умолчанию пространство имен называется по имени проекта, но это можно изменить.
Пространства имен можно создавать вложенными. Это делается двумя способами.
Первый способ заключается в буквальном вложении нового пространства имен в существующее пространство имен:
```C#
namespace TestParent
{
	namespace TestChild
	{
		<код в пространстве имен>
	}
}
```
Второй способ заключается в создании нового пространства имен с дополнением в названии:
```C#
namespace TestParent
{
	<код в пространстве имен>
}
namespace TestParent.TestChild
{
	<код в пространстве имен>
}
```
В обоих случаях вызов имен из `TestChild` будет происходить через конструкцию `TestParent.TestChild.<имя>`.
Если я делаю свой собственный namespace, то документация рекомендует описывать в его названии полный путь до директории, в которой лежит файл. Например:
```C#
namespace Commission.Domain.ValueObjects
```
В данном случае у меня в корне проекта лежит директория `Commission.Domain`, а внутри нее лежит директория `ValueObjects`, где размещен файл, в котором я описываю namespace.
# класс
Внутри программы:
```C#
using System;
namespace app
{
	public class Program
	{
		public static void Main(string[] args)
		{
			Console.WriteLine("Hello, world!");
		}
	}
}
```
Объявлен класс `Program`. Так как он объявлен внутри пространства имен `app`, то полный путь к классу будет `app.Program`.
Так как `C#` не поддерживает глобальные методы, весь код должен помещаться в классы, даже если классы (как в примере выше) содержат только один единственный метод.
>[!important] Кстати, класс `Program` в книжке называется "тип".
## атрибуты
Когда я писал из книжки [[си шарп и .NET#первая программа]], то там были свои особенности:
```C#
[TestClass]
public class WhenProgramRuns
{
	[TestInitialize]
	public void Initialize()
	{
	}
	[TestMethod]
	public void SaysHelloWorld()
	{
	}
}
```
В этом примере `[TestClass]`, `[TestInitialize]`, `[TestMethod]` - это атрибуты. В обычных программах `C#` они никак не влияют на выполнение кода, однако здесь у меня написан тестирующий модуль + этот тестирующий модуль написан с помощью фреймворка. Это означает, что атрибуты все же могут быть как-то использованы, т.к. фреймворк может задействовать их для выполнения какого-то функционала.
В моем конкретном случае атрибуты использованы так:
- `TestClass` - это метка для среды юнит-теста Microsoft. Так модуль юнит-теста определяет тестовые классы, которые ему нужно запускать.
- `TestInitialize` - это метка, которая отмечает, что в методе под ней лежат все методы, которые необходимо выполнить по разу для каждого теста.
- `TestMethod` - это метка, которая сообщает системе выполнения тестов, что метод является тестом.
# точка входа
Компилятор `C#` автоматически ищет в проекте точку входа и автоматически считает, что это должен быть метод `Main`. Строго говоря таким методом может быть любой метод, но в таком случае придется сообщить об этом компилятору отдельно.
Точка выхода должна быть статическим методом. Это метод, который существует как бы автономно от класса, в котором он прописан, т.е. может быть использован без создания экземпляра этого класса.
Как правило точка входа ничего не возвращает, либо возвращает код ошибки, т.е. число типа `int`.
При этом метод не должен принимать никаких аргументов, либо принимать пустую строку. Пустая строка в данном случае будет ожидать аргументы командной строки.

## public и private
Когда метод может быть использован вне своего класса его помечают словом `public`. Когда метод используется и доступен только в пределах своего класса его помечают словом `private`.

```C#
namespace app

        public string? Name { get; set; }

        {
            Console.WriteLine(WriteMyName());
        }
        public string WriteMyName()
        {
            this.Name = "Peter";
            return Name;
        }
    }
}
```
И она вылетела на стадии компиляции:
![[ошибка_c.png]]
И я долго не мог понять почему ТАК, ведь я все делаю правильно - сделал метод `WriteMyName()`, который возвращает строку, а потом передал метод в `Console.WriteLine()`.
Однако проблема в том, что я пытаюсь из статического метода вызвать нестатический. И это логично, потому что статический метод потому и статический - он не зависит от класса, в котором находится, поэтому и представлять его нужно абсолютно отдельно от всех остальных элементов класса. При этом метод `WriteMyName()` является именно методом класса `Programm`, что означает, что для использования `WriteMyName()` мне нужно сначала сделать экземпляр класса `Programm`:
```C#
namespace app
{
	public class Programm
	{
		public string? Name { get; set; }
		static void Main(string[] args)
		{
			Programm programm = new Programm(); // создаю экземпляр класса.
			Console.WriteLine(programm.WriteMyName()); // из экземпляра беру метод.
		}
		public string WriteMyName()
		{
			this.Name = "Peter";
			return Name;
		}
	}
}
```
Статические методы - это буквально имена, которые доступны во всем текущем пространстве имен, как самостоятельные сущности.
Кстати, эту же программу можно упростить - если я хочу внутри класса создать экземпляр этого же класса мне можно сделать так:
```C#
Programm programm = new();
```
# переменные
Термин "переменная" в C# может относится с сразу к нескольким сущностям - локальная переменная, поля объектов, элементы массива.
Здесь я пока буду писать о локальных переменных.
C# - это язык со статической типизацией. Это означает, что для каждой переменной, прежде чем в нее будет записано значение, будет объявлен тип данных, которые она может использовать.
Вот так:
```C#
string part1 = "Hello!";
string part2 = "World!";
```
Переменные можно сначала объявлять, а потом закладывать в них значения:
```C#
int myNumber;
myNumber = 42;
```
После того как переменной было задано значение, значение можно заменить другим:
```C#
int myNumber = 42;
myNumber = 228;
```
Вместо четкой типизации можно использовать ключевое слово `var`:
```C#
var test1 = "meme!";
var test2 = "nene!";
```
Отличие такого подхода в том, что таким образом мы позволяем решать компилятору за нас к какому типу будет принадлежать переменная. Важно помнить, что переменная все равно будет принадлежать к статическому типу данных.
>[!important] Компилятор использует пессимистичную систему оценивания того есть ли у переменной значение. Он всегда "подстраховывается" - выдает всем переменным значение (например, $0$ для числовых переменных), но при компиляции все равно жалуется на отсутствующее значение.
## область видимости
**Область видимости** - это части кода, где я могу ссылаться на объект по его имени без дополнительной квалификации.
**Дополнительная квалификация** - это когда я не могу вызвать метод (или переменную) без указания какого-то модуля, из которого я его буду вызывать. Например:
```C#
Console.WriteLine();
```
Метод `WriteLine()` использует дополнительную квалификацию - класс `Console`. Но:
```C#
string my_string = "привет!";
Console.WriteLine(my_string);
```
Переменная `my_string` использует абсолютную квалификацию (т.е. для нее не нужна никакая дополнительная квалификация), т.к. она локальная - либо она доступна с абсолютной квалификацией, либо недоступна вообще.
Область действия переменной начинается с ее объявления и заканчивается в конце содержащего ее блока.
**Блок** - это область кода, которая ограничена парой скобок `{}`. Переменная, созданная в одном блоке, недоступна в другом.
Методы классов часто содержат вложенные блоки. Все, что находится в области видимости внешнего блока, доступно и во вложенном. Например:
```C#
public class Programm
{
	static void Main(string[] args)
	{
		Programm programm = new Programm();
		if (programm.GetNumber() > 0) 
		{
			Console.WriteLine(programm);
		}
	}
}
```
В обратную сторону это не работает - переменные, которые объявлены внутри подблока недоступны внешнему блоку и действуют только в рамках подблока.

---
Существуют так же свои странности.
Например, если создать локальную переменную в подблоке, а затем попытаться создать переменную с таким же именем во внешнем блоке, то компилятор вернет ошибку:
```C#
if (programm.GetNumber() > 0)
{
	int anotherValue = 42;
	Console.WriteLine(anotherValue);
}
int anotherValue = 43;
```
Ошибка выглядит так:
![[anotherValue.png]]
Это тн "ошибка неопределенности имени переменной". Компилятор запрещает практику объявления переменных с одними и теми же именами, потому что хочет предотвратить ситуацию, когда кодер забудет, что у него уже где-то в коде была такая переменная и создаст еще одну подобную, а это уже приведет к другим забавным конфликтам.
однако не совсем понятно как это работает. если я попытаюсь изменить в подблоке переменную из внешнего блока, я еще пойму жалобу об ошибке от компилятора:
```C#
int anotherValue = 0;
if (programm.GetNumber() > 0)
{
	if (programm.GetNumber() > 0)
	{
		anotherValue += 1;
	}
	
	int anotherValue = 0; // вот поэтому компилятор сообщит об ошибке.
}
```
но если я пытаюсь сделать новую переменную с таким же именем после подблока, то по факту области видимости переменных не пересекаются.
этот феномен объясняется понятием *пространство объявления*. 
**пространство объявления** - это область кода, в которой одно имя не должно ссылаться на два разных объекта. каждый метод устанавливает собственное *пространство объявления*. каждый вложенный блок так же устанавливает собственное *пространство объявления* и во вложенном *пространстве объявлений* нельзя использовать те же имена для новых объектов, что и в родительских *пространствах объявлений*.
цель запрета на объявления переменных с одинаковыми именами в том, чтобы нельзя было нарушить логику кода при перемещении объявления переменных по пространству объявлений.
## память и переменные
каждая переменная объявляется области объявления единожды и единожды покидает область объявления. однако это не означает, что под переменную место в памяти выделяется лишь единожды.
на самом деле каждый раз, когда вызывается конкретная область видимости, под переменные в ней выделяется новое место в памяти. это означает, что каждый раз при запуске метода он получает отдельный набор хранилищ в памяти для локальных переменных.
>[!important] Согласно книжке, которую я читаю, C# не гарантирует местонахождение переменных никак. Это означает, что переменная может находиться в стэке, а может не находиться в стэке.
# инструкции и выражения
код на C# - это последовательность инструкций (или операторов).
**операторы (= инструкции)** - это описание действий, которые должны выполняться.
**инструкция (= оператор)** - это все то, что написано до точки с запятой. строго говоря, в C# не обязательно даже делать переносы строк, т.к. он не учитывается компилятором при чтении кода.
в этой простой логике есть одно важное исключение - блоки кода.
для меня было открытием, что блок кода можно написать просто так на ровном месте:
```C#
static void Main(string[] args)
{
	int a = 1;
	Console.WriteLine(a);

	{
		int b = 2;
		Console.WriteLine(b);
	}
}
```
так вот, из-за того, что блок кода является оператором, но при этом не завершается точкой с запятой + содержит несколько операторов с (тчк-запятая в конце).

---
вообще-то я не очень люблю копаться в теоретических терминах по типу *оператор*, *выражение* и прч., потому что сами разработчики в спецификации языка часто относятся к этому чисто формально.
например, определение термина *выражение* в спецификации C# такое: *последовательность операторов и операндов*. здорово. сразу все понятно.
касательно выражений самое удобное определение: *выражение - это тот участок кода, который возвращает некое значение*.
простейший вид выражения - *литерал*. грубо говоря, *литерал - это самый самый самый атомарный атом любого выражения*. например, в выражении `2 + 2` цифры `2` являются литералами-выражениями, а вместе они составляют выражение побольше. то есть большие выражения могут состоять из маленьких выражений.
а еще есть *операнды* - это участники *операций*.
короче, лучше просто забить на все это и кодить.
однако следует знать, что выражения - это почти всегда *действия*, которые *возвращают значение*. поэтому выражение `2 + 2` хотя и корректно само по себе, но если поставить в его конце точку с запятой и попытаться запустить код, то компилятор вернет ошибку. так происходит потому, что мы буквально попытались представить как оператор выражения неподходящий код.
Я ЗНАЮ ЭТО ЗВУЧИТ ОЧЕНЬ ТУПО ПОТОМУ ЧТО `2 + 2` ВОЗВРАЩАЕТ ЗНАЧЕНИЕ.
но тут следует сконцентрироваться на том, что некоторые выражения не могут быть операторами, они могут только быть операндами или литералами, т.е. могут только являться частью оператора.
функции типа `Math.Sqrt(4)` избегают такой участи только потому, что некоторые другие функции иногда могут быть операторами и такое поведение допустимо. поэтому в отличие от операции сложения `Math.Sqrt(4)` можно написать так:
```C#
Math.Sqrt(4);
```
при том, что этот код ничего не будет делать.
важно понимать, что в C# операнды оцениваются в порядке их появления в коде. это означает, что такой код:
```C#
public class Programm
{
	public int X(string label, int i)
	{
		Console.WriteLine(label);
		return i;
	}
	static void Main(string[] args)
	{
		Programm programm = new Programm();
		Console.WriteLine(X("a", 1) + X("b", 2) + X("c", 3));
	}
}
```
будет выполнен корректно и по порядку.
а такой код:
```C#
public class Programm
{
	public int X(string label, int i)
	{
		Console.WriteLine(label);
		return i;
	}
	static void Main(string[] args)
	{
		Programm programm = new Programm();
		Console.WriteLine(
			programm.X("a", 1)
			+ programm.X(
				"b", 
				programm.X("c", 2)
				+ programm.X("d", 3)
				+ programm.X("e", 4)
			)
			+ programm.X("f", 5)
		);
	}
}
```
вернет результат не в алфавитном порядке - `acdebf15`. так произошло, потому что на вход функции с `label = "b"` я передаю сумму результатов нескольких методов, поэтому сначала будет вычислена именно она. это означает, что выражение `label = "b"` начинает вычисляться раньше всех внутренних, однако заканчивает только после них.
# комментарии
комментарий - это то, что будет проигнорировано компилятором при компиляции кода. то есть это то, что нужно только для разработчика, который читает код.
в C# есть два вида комментариев - однострочные и многострочные.
однострочные:
```C#
int a = 1;  // это переменная бла-бла-бла.
```
многострочные:
```C#
/*
тут можно писать
комментарий в несколько
строк.
*/
```
причем многострочные комментарии можно использовать даже внутри выражений:
```C#
Console.WriteLine(/* это комментарий внутри функции */"Привет!");
```

# методы `object`
В C# существует базовый класс для всех классов - `object`. Абсолютно все в C# наследуется от `object`.
У класса `object` есть ряд методов, которыми он обеспечивает каждый класс в C#. Как раз об этих методах я и хотел поговорить.
## `ToString()`
Метод, который определяет как строково будет представлен класс. По сути через него определена репрезентация класса, например, в консольном выводе:
```C#
namespace app
{
	public class Programm
	{
		static void Main(string[] args)
		{
			Programm programm = new Programm();
			Console.WriteLine(programm);
		}
	}
}
```
Если я запущу такой код:
![[programm.png]]
То получу имя класса, к которому принадлежит объект `programm`.
Но что если я хочу сделать так, чтобы вместо имени класса печаталось, например, слово "Привет!"? Тогда мне можно просто переопределить метод `ToString()` для этого класса, потому что именно он производит строковую репрезентацию объектов класса:
```C#
namespace app
{
	public class Programm
	{
		static void Main(string[] args)
		{
			Programm programm = new Programm();
			Console.WriteLine(programm);
		}
		public override string ToString()
		{
			return "Привет!";
		}
	}
}
```
Если мне необходимо переоформить метод, то я его вызову по имени с атрибутом `override`, который как раз и будет указывать на мое намерение переписать метод.
## Equals()
Этот метод по умолчанию сравнивает ссылки на объекты в памяти. Например:
```C#
namespace app
{
	public class App
	{
		static void Main(string[] args)
		{
			Number number_1 = new Number { A = 1, B = 2 };
			Number number_2 = new Number { A = 1, B = 2 };

			Console.WriteLine(number_1.Equals(number_2))
		}
	}
	public class Number
	{
		public int A { get; set; }
		public int B { get; set; }
	}
}
```
Такая программа вернет `False` в консоль, потому что ссылка на первый объект в памяти отличается от ссылки на второй объект в памяти.
Здесь я встретился с двумя важными понятиями в C#:
### struct и class
Началось все с того, что я увидел, что в одном из участков кода на StackOverflow, сразу после того как прочитал про неравенство двух экземпляров одного и того же класса (с одинаковыми значениями), такой код:
```C#
static void Main(string[] args)
{
	Number number_1 = new Number { A = 1, B = 2 };
	Number number_2 = new Number { A = 1, B = 2 };

	Console.WriteLine(number_1.Equals(number_2));
}
```
Возвращает `True`.
Оказывается, что помимо класса, существует еще одна сущность - структура. Отличие между ними в том, что экземпляры класса работают через ссылки на объекты в памяти, а экземпляры структуры работают через сами значения.
То, что структуры являются работают через значения означает, что при одинаковых значениях два разных экземпляра структуры будут идентичны:
```C#
public class Programm
{
	static void Main(string[] args)
	{
		Number number_1 = new Number { A = 1, B = 2 };
		Number number_2 = new Number { A = 1, B = 2 };

		Console.WriteLine(number_1.Equals(number_2));
	}
	public struct Number
	{
		public int A { get; set; }
		public int B { get; set; }
	}
}
```
Если атрибуты двух экземпляров структуры будут равны, то и `Equals()` вернет `True`.

---
Чтобы понимать каким образом экземпляры класса отличаются от экземпляров структуры, нужно понимать как объекты хранятся в памяти.
Под программу выделяется место в оперативной памяти компьютера. Это место в свою очередь делится на *STACK memory* и *HEAP memory*.
В *STACK memory* хранятся ссылки на объекты (например, переменная `number_2`, в которой хранится ссылка на объект класса `Number`), а в *HEAP memory* хранятся сами объекты (например, экземпляр класса `Number` с значением атрибутов `A = 1, B = 2`).
Когда мы говорим о классах, речь идет именно о таком разделении - ссылка в *STACK* и объект в *HEAP*. Но структура уже хранит весь объект в *STACK*, поэтому ей не нужно создавать ссылку для своих экземпляров.
Из-за этого экземпляры структуры обладают другими свойствами, например я написал такой код:
```C#
namespace app
{
    public class App
    {
        public string? Name { get; private set; }
        static void Main(string[] args)
        {

            Numbers_1 number_1 = new Numbers_1 { A = 1, B = 2 };
            Numbers_1 number_2 = new Numbers_1 { A = 1, B = 2 };
            Numbers_1 number_3 = number_1;

            Numbers_2 number_4 = new Numbers_2 { A = 1, B = 2 };
            Numbers_2 number_5 = new Numbers_2 { A = 1, B = 2 };
            Numbers_2 number_6 = number_4;

            Console.WriteLine(number_3.A);
            number_3.A += 1;
            Console.WriteLine(number_1.A);

            Console.WriteLine(number_6.A);
            number_6.A += 1;
            Console.WriteLine(number_4.A);
        }
    }
    public struct Numbers_1
    {
        public int A { get; set; }
        public int B { get; set; }
    }
    public class Numbers_2
    {
        public int A { get; set; }
        public int B { get; set; }
    }
}
```
Здесь:
- `Number_1` - это структура.
- `Number_2` - это класс.
При запуске программы создается сначала два объекта структуры `Number_1` с одинаковыми атрибутами, а затем создается третий объект на основе копирования первого.
Затем создается два экземпляра класса `Number_2` и еще один экземпляр  на основании первого.
Здесь:
```C#
Console.WriteLine(number_3.A);
number_3.A += 1;
Console.WriteLine(number_1.A);
```
Я пытаюсь поменять значение атрибута `A` для объекта структуры `number_3`. А затем делаю то же самое для объекта класса.
Вывод программы:
![[c_sharp_5.png]]
В последнем случае видно, что, хотя я меняю атрибут у объекта `number_6`, атрибут становится другим в том числе и для объекта `number_4`. Так происходит потому, что `number_6` ссылается на объект `number_4`, который ссылается на экземпляр класса в памяти. Так как мы говорим об экземпляре класса, то в данном случае обе ссылки указывают на один объект в памяти, что позволяет через одну ссылку менять значения всех других ссылок.
Со структурой такого не происходит. В идентичной операции присваивания на самом деле создается копия структуры, т.е. новый объект, поэтому значение атрибута старого объекта при изменении значения этого же атрибута в новом остается прежним.
## GetHasCode()
У объектов на C# существует числовое представление, которое используется некоторыми структурами данных. Это числовое представление задается методом `GetHashCode()`.
Напишу программу:
```C#
namespace app 
{
	public class Programm
	{
		static void Main(string[] args)
		{
			Programm programm1 = new();
			Programm programm2 = new();
			Console.WriteLine(programm1.GetHashCode());
			Console.WriteLine(programm2.GetHashCode());
		}
	}
}
```
Она вернет:
![[get_hash_code.png]]
Это два цифровых представления двух разных объектов одного и того же класса.
Допустим, что у меня появился класс, который на вход принимает валюту и создает на основе валюты объекты. Допустим еще, что мне нужно, чтобы при сравнении хэшей двух объектов с одинаковой валютой метод `Equals()` возвращал `true`. 
Я напишу такой класс:
```C#
namespace app
{
	public class Programm
	{
		static void Main(string[] args)
		{
			Currency currency1 = new("USD");
			Currency currency2 = new("USD");
			Console.WriteLine(currency1.CurrencyCode);
			Console.WriteLine(currency2.CurrencyCode);
			Console.WriteLine(currency1.GetHashCode());
			Console.WriteLine(currency2.GetHashCode());
		}
	}
	public class Currency
	{
		public string CurrencyCode { get; set; }
		public Currency(string code)
		{
			CurrencyCode = code;
		}
	}
}
```
# функция
Здесь я пока отмечу только то, что функция в C# может быть реализована в виде нормальной функции:
```C#
public override string ToString()
{
	return "Привет!";
}
```
А может быть реализована в сокращенном виде (*стрелочная функция*):
```C#
public override string ToString() => "Привет!";
```
Второй вариант просто является сокращенным вариантом первого.
