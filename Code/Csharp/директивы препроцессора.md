директивы препроцессора это штуки, которые в языке C запускаются до начала компиляции. в C# таких директив нет, но от них осталось немного "наследия".
вообще препроцессирование - это этап обработки исходного кода перед компиляцией. компилятор читает код как текст и выполняет набор операций, который характерен только для этапа препроцессирования.
например, в C и С++ есть такая фишка *макроподстановки* - это буквально объявление переменных, которые будут работать глобально во всем коде. макроподстановки - это одна из операций препроцессирования.
в C# нет предварительной обработки и нет макросов, но все же есть набор команд, которые визуально напоминают директивы препроцессора и их называют *директивы препроцессора*.
## символы компиляции
символы компиляции - это флаги, которые сообщают компилятору "если такой флаг есть, то выполни такой код". флаги можно устанавливать свои собственные, а можно использовать те, которые устанавливает C# по умолчанию. 
например, .NET SDK устанавливает две конфигурации - `Debug` и `Release`. для конфигурации `Debug` определяется символ компиляции `DEBUG`, а для конфигурации `Release` определяется символ компиляции `RELEASE`. для обоих конфигураций доступен символ компиляции `TRACE` - это означает, что все методы, которые запускаются с его использованием будут выполнены и в конфигурации `Debug`, и в конфигурации `Release`.
символы компиляции - это флаги. флаги должны что-то делать. в данном случае, флаги можно использовать с конструкциями `#if, #else, #elif, #endif`. например:
```C#
public class Programm()
{
	static void Main(string[] args)
	{
		#if DEBUG
			Console.WriteLine("Привет, здесь я использую DEBUG!");
		#endif
		Console.WriteLine("Hello world!");
	}
}
```
в данном случае мы пользуемся символом компиляции `DEBUG`, который заранее объявлен самим .NET SDK.
кроме использования символов `#if, #else, #elif, #endif` можно использовать аннотации для разметки методов, которые следует включать для каких-то символов компиляции:
```C#
public class Programm()
{
	static void Main(string[] args)
	{
		#if DEBUG
			Console.WriteLine("Привет, здесь я использую DEBUG!");
		#endif
		MyMethod("Привет, это метод говорит!");
		Console.WriteLine("Hello world!");
	}
	
	[System.Diagnostics.Conditional("DEBUG")]
	static void MyMethod(object o)
	{
		Console.WriteLine(o);
	}
}
```
строка `[System.Diagnostics.Conditional("DEBUG")]` примерно соответствует записи `#if DEBUG` и означает, что метод может быть использован, только если программа запущена в режиме `Debug`.
а что произойдет с вызовом метода, если я запущу программу в другом режиме? компилятор по идее должен показать ошибку, т.к. вызов метода будет пытаться вызвать несуществующий метод. однако C# умнее - при вызове в другом режиме он просто удалит и метод и вызов метода при компиляции.

здесь я расскажу об интересной штуке - о `Debug.Assert`.
`Debug.Assert()` - это метод, который проверяет условие на верность. в случае правдивости условия он не делает ничего, а в случае ложности условия он выбрасывает сообщение об ошибке, которое передается вторым аргументом. в данном случае я говорю о методе `Assert()` класса `Debug`, потому что его неосторожное использование может привести к багу. 
допустим мы написали код, в котором несколько `Debug.Assert()` должны отлавливать ошибки. все работает, все круто. а теперь мы запускаем релизную сборку и ожидаем, что все `Debug.Assert()` отработают и там. однако этого не происходит, т.к. нужно понимать, что `Debug.Assert()` так же зависит от конфигурации `DEBUG`, т.е. этот метод не будет работать на релизной сборке и, более того, релизная сборка будет работать так, как будто никаких выражений с этим методом не существует в коде.
я написал такой код:
```C#
using System.Diagnostics;

namespace app2
{
    public class Programm
    {
        static void Main(string[] args)
        {
            Debug.Assert(1 != 1, "Ошибка!");
            Console.WriteLine("Hello world!");
        }
    }
}
```
и я запустил его в `Debug` режиме:
![[preproc4.png]]
и теперь я запущу программу в режиме `Release`:
![[preproc5.png]]
ошибка пропала, потому что `Debug.Assert()` привязан к символу компиляции `DEBUG`, следовательно, при смене конфигурации он просто исчезает из кода.
теперь попробую запустить в режиме `Trace`:
![[preproc6.png]]
в режиме `Trace` ошибки тоже нет.
>[!important] По умолчанию команда `dotnet run` запускает проект с конфигурацией `Debug`.
# \#error и \#warning
две директивы, которые позволяют генерировать ошибки или предупреждения компилятора. как правило они используются внутри условных конструкций:
```C#
namespace app3
{
	public class Programm
	{
		static void Main(string[] args)
		{
			#if DEBUG
				#warning что-то предупредительное
			#endif
			Console.WriteLine("Hello world!"); 
		}
	}
}
```

# \#line
директива `#line` позволяет указать строку и файл после себя. зачем? иногда код использует более одного файла и ошибки могут появляться именно в дочерних, а не в основных файлах. в этом случае полезно знать где именно в дочернем файле возникла ошибка. если использовать `#line` корректно, то можно правильно сообщить на этапе компиляции адрес ошибки.
я написал такой код:
```C#
namespace app3
{
	public class Programm
	{
		static void Main(string[] args)
		{
			#line 7 "Foo.cs"
				iint x;
			Console.WriteLine("Hello world!");
		}
	}
}
```
здесь ошибка в том, что я задаю переменной тип `iint` и компилятор покажет мне это:
![[preproc9.png]]
хотя я писал код в `app3/Programm.cs` файле ошибка демонстрируется в файле `app3/Foo.cs` на строке $7$. это именно то, что я установил в директиве `#line`.
