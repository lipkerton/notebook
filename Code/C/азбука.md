# директива using
В `C#` существует термин - *пространство имен*. Локально он описывает набор имен, которые задаются нами самими в проекте. Это означает, что в пространство имен данного проекта будут входить все поименованные сущности, например, переменные, списки, функции и т.д.
В [[си шарп и .NET]] я писал об использовании одним проектом метода `Main`, который находится в другом проекте. Там я применил `using`, чтобы написать:
```C#
using app;
...
Program.Main(new string[0]);
```
Как раз на этом примере видно применение директивы `using` - она открывает доступ к именам иного пространства имен.
Глобально пространства имен решают две проблемы - проблема уникальных имен в огромном количестве библиотек `C#` и проблема поиска необходимого интерфейса среди гигантского количества интерфейсов.

В книжке написано: 
> Большинство типов .NET определены в пространстве имен. 

И таких пространств имен существует невероятно много - они написаны другими разработчиками для расширения инструментария `C#` и иногда упорядочены странно. Например, существуют пространства имен внутри пространств имен - уровни вложенности могут быть любыми.
Основные пространства:
- `System` - пространство имен, которое содержит типы, являющиеся частью `.NET`.
- `Microsoft` - пространство имен, которое содержит типы, являющиеся частью какой-нибудь технологии Microsoft.
Так же существуют библиотеки других компаний, которые называются по имени компании.
Крайне рекомендуется помещать собственные типы в пространства имен. `System` не является специальным закрытым пространством имен, поэтому собственные типы можно присваивать и ему.
Так же как и в Python в `C#` можно выдавать псевдонимы для директив `using` - такое делается, чтобы не возникало совпадения между одинаковыми именами из двух пространств имен + не нужно было писать полный путь до нужного имени:
```C#
using System.IO;
using System.Windows.Shapes;
using IoPath = System.IO.Path;
using WpfPath = System.Windows.Shapes.Path;
```
# `namespace`
Директива `using` показывает какие имена наша программа будет использовать. А команда `namespace` указывает какое пространство имен мы хотим создать:
```C#
namespace TestSpace 
{
	<код в пространстве имен>
}
```
По умолчанию пространство имен называется по имени проекта, но это можно изменить.
Пространства имен можно создавать вложенными. Это делается двумя способами.
Первый способ заключается в буквальном вложении нового пространства имен в существующее пространство имен:
```C#
namespace TestParent
{
	namespace TestChild
	{
		<код в пространстве имен>
	}
}
```
Второй способ заключается в создании нового пространства имен с дополнением в названии:
```C#
namespace TestParent
{
	<код в пространстве имен>
}
namespace TestParent.TestChild
{
	<код в пространстве имен>
}
```
В обоих случаях вызов имен из `TestChild` будет происходить через конструкцию `TestParent.TestChild.<имя>`.
Если я делаю свой собственный namespace, то документация рекомендует описывать в его названии полный путь до директории, в которой лежит файл. Например:
```C#
namespace Commission.Domain.ValueObjects
```
В данном случае у меня в корне проекта лежит директория `Commission.Domain`, а внутри нее лежит директория `ValueObjects`, где размещен файл, в котором я описываю namespace.
# класс
Внутри программы:
```C#
using System;
namespace app
{
	public class Program
	{
		public static void Main(string[] args)
		{
			Console.WriteLine("Hello, world!");
		}
	}
}
```
Объявлен класс `Program`. Так как он объявлен внутри пространства имен `app`, то полный путь к классу будет `app.Program`.
Так как `C#` не поддерживает глобальные методы, весь код должен помещаться в классы, даже если классы (как в примере выше) содержат только один единственный метод.
>[!important] Кстати, класс `Program` в книжке называется "тип".
# точка входа
Компилятор `C#` автоматически ищет в проекте точку входа и автоматически считает, что это должен быть метод `Main`. Строго говоря таким методом может быть любой метод, но в таком случае придется сообщить об этом компилятору отдельно.
Точка выхода должна быть статическим методом. Это метод, который существует как бы автономно от класса, в котором он прописан, т.е. может быть использован без создания экземпляра этого класса.
Как правило точка входа ничего не возвращает, либо возвращает код ошибки, т.е. число типа `int`.
При этом метод не должен принимать никаких аргументов, либо принимать пустую строку. Пустая строка в данном случае будет ожидать аргументы командной строки.
# атрибуты
Когда я писал из книжки [[си шарп и .NET#первая программа]], то там были свои особенности:
```C#
[TestClass]
public class WhenProgramRuns
{
	[TestInitialize]
	public void Initialize()
	{
	}
	[TestMethod]
	public void SaysHelloWorld()
	{
	}
}
```
В этом примере `[TestClass]`, `[TestInitialize]`, `[TestMethod]` - это атрибуты. В обычных программах `C#` они никак не влияют на выполнение кода, однако здесь у меня написан тестирующий модуль + этот тестирующий модуль написан с помощью фреймворка. Это означает, что атрибуты все же могут быть как-то использованы, т.к. фреймворк может задействовать их для выполнения какого-то функционала.
В моем конкретном случае атрибуты использованы так:
- `TestClass` - это метка для среды юнит-теста Microsoft. Так модуль юнит-теста определяет тестовые классы, которые ему нужно запускать.
- `TestInitialize` - это метка, которая отмечает, что в методе под ней лежат все методы, которые необходимо выполнить по разу для каждого теста.
- `TestMethod` - это метка, которая сообщает системе выполнения тестов, что метод является тестом.
# public и private
Когда метод может быть использован вне своего класса его помечают словом `public`. Когда метод используется и доступен только в пределах своего класса его помечают словом `private`.
# static
Я написал программу:
```C#
namespace app
{
    public class Programm
    {
        public string? Name { get; set; }
        static void Main(string[] args)
        {
            Console.WriteLine(WriteMyName());
        }
        public string WriteMyName()
        {
            this.Name = "Peter";
            return Name;
        }
    }
}
```
И она вылетела на стадии компиляции:
![[ошибка_c.png]]
И я долго не мог понять почему ТАК, ведь я все делаю правильно - сделал метод `WriteMyName()`, который возвращает строку, а потом передал метод в `Console.WriteLine()`.
Однако проблема в том, что я пытаюсь из статического метода вызвать нестатический. И это логично, потому что статический метод потому и статический - он не зависит от класса, в котором находится, поэтому и представлять его нужно абсолютно отдельно от всех остальных элементов класса. При этом метод `WriteMyName()` является именно методом класса `Programm`, что означает, что для использования `WriteMyName()` мне нужно сначала сделать экземпляр класса `Programm`:
```C#
namespace app
{
	public class Programm
	{
		public string? Name { get; set; }
		static void Main(string[] args)
		{
			Programm programm = new Programm(); // создаю экземпляр класса.
			Console.WriteLine(programm.WriteMyName()); // из экземпляра беру метод.
		}
		public string WriteMyName()
		{
			this.Name = "Peter";
			return Name;
		}
	}
}
```
Статические методы - это буквально имена, которые доступны во всем текущем пространстве имен, как самостоятельные сущности.
Кстати, эту же программу можно упростить - если я хочу внутри класса создать экземпляр этого же класса мне можно сделать так:
```C#
Programm programm = new();
```
# переменные
Термин "переменная" в C# может относится с сразу к нескольким сущностям - локальная переменная, поля объектов, элементы массива.
Здесь я пока буду писать о локальных переменных.
C# - это язык со статической типизацией. Это означает, что для каждой переменной, прежде чем в нее будет записано значение, будет объявлен тип данных, которые она может использовать.
# методы `object`
В C# существует базовый класс для всех классов - `object`. Абсолютно все в C# наследуется от `object`.
У класса `object` есть ряд методов, которыми он обеспечивает каждый класс в C#. Как раз об этих методах я и хотел поговорить.
## `ToString()`
Метод, который определяет как строково будет представлен класс. По сути через него определена репрезентация класса, например, в консольном выводе:
```C#
namespace app
{
	public class Programm
	{
		static void Main(string[] args)
		{
			Programm programm = new Programm();
			Console.WriteLine(programm);
		}
	}
}
```
Если я запущу такой код:
![[programm.png]]
То получу имя класса, к которому принадлежит объект `programm`.
Но что если я хочу сделать так, чтобы вместо имени класса печаталось, например, слово "Привет!"? Тогда мне можно просто переопределить метод `ToString()` для этого класса, потому что именно он производит строковую репрезентацию объектов класса:
```C#
namespace app
{
	public class Programm
	{
		static void Main(string[] args)
		{
			Programm programm = new Programm();
			Console.WriteLine(programm);
		}
		public override string ToString()
		{
			return "Привет!";
		}
	}
}
```
Если мне необходимо переоформить метод, то я его вызову по имени с атрибутом `override`, который как раз и будет указывать на мое намерение переписать метод.
