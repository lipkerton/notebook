# предварительно
`.NET` и `C#` - одно целое и одновременно нет. Я обнаружил их вместе уже на первой странице книжки `Программируем на C# 8.0`. 
`.NET` включает в себя "среду выполнения" и библиотеку основных классов, которые используют программы `C#`. 
Частью среды выполнения является **CLR (Common Language Runtime)** - *общеязыковая среда выполнения*. Это среда, которая поддерживает не только `C#`, но и любой другой язык `.NET` (например, существует `.NET` для `Visual Basic`, `F#` и `C++`). CLR так же имеет общую систему типов (**CTS - Common Type System**), что позволяет всем языкам из `.NET` использовать библиотеки, заимствованные друг у друга.
**Управляемый код** - это способ генерации машинного кода на основе кода `C#`. Он заключается в том, что перед созданием машинного кода создается форма двоичного кода **промежуточный язык (intermediate language, IL)**. Такой подход обеспечивает переносимость выходных данных компилятора - они перестают быть привязаны к конкретной архитектуре процессора. Это означает, что я могу написать `.NET`-компонент, который будет работать схожим образом на большинстве процессорных архитектур. Время, когда CRM генерирует исполняемый машинный код, может варьировать, потому что внутри (в зависимости от используемых инструментов и прч.) применяется несколько подходов к компиляции. 
Наиболее частый - **JIT-компиляция (just-in-time)**. JIT-компиляция - это тип компиляции, при котором каждая отдельная функция компилируется при первом ее выполнении. Еще есть, например, **AoT-компиляция (ahead-of-time)**. *Многоуровневая функция компиляции* во время выполнения так же может решить, что эффективнее было бы перекомпилировать код.
Про версии `C#` и `.NET` следует знать, что они плотно связаны. Существуют разные реализации `C#` и разные `.NET`. На текущий момент стоит знать о двух основных сборках `.NET`: 
- `.NET Framework` - версия для Windows с закрытым исходным кодом.
- `.NET Core` - версия для всех остальных (и для Windows в том числе) с открытым исходным кодом.
# инфраструктура
Файлы проекта на C-подобных языках обычно имеют расширение, название которого оканчивается на `proj`. Например, для `C#` - это будет `.csproj`, а для `C++` - `.vcxproj`.

>[!important] Если открыть эти файлы в текстовом редакторе, то можно увидеть XML.

После генерации проекта на `.NET Core` складывает логику в файлы с расширением `.dll` (**dll - dynamyc link library**). Хотя все привыкли, что в винде исполняемые файлы называются `.exe`, на самом деле файлы типа `.dll` от `.exe` отличаются только тем, что для `.exe` указана "точка входа приложения". Мне кажется, что это означает, что `.dll` нельзя запустить так же просто, как `.exe`.
При создании проекта для `C#` следует думать о том, что, как правило, в `C#` большие программы строятся на основании группы проектов. Например, у меня может быть проект приложения, которое выполняет какую-то логику, проект тестов для этого приложения и т.д.
Чтобы поддерживать логику группы проектов в Visual Studio (и, мне кажется, не только в ней) есть фишка, которая называется **решение**. Решение - это что-то вроде схемы для множества таблиц в SQL. Решение является файлом с расширением `.sln` и этот файл будет описывать комбинацию из проектов. Так как решение - это файл, то оно просто будет валяться в корневой, общей для всех проектов, директории.

# первая программа
Внутри файла `app/Program.cs` я напишу программу, которая печатает в консоль "Hello World!":
```C#
Console.WriteLine("HelloWorld!")
```
А внутри файла `app_test/Test1.cs` я хочу написать тест, который будет проверять, что программа `app/Program.cs` действительно отработала как нужно:
```C#
using System;
using Microsoft.VisualStudio.TestTools.UnitTesting;

namespase app_test
{
	[TestClass]
	public class WhenProgramRuns
	{
		private string _consoleOutput;
		
		[TestInitialize]
		public void Initialize()
		{
			var w = new System.IO.StringWriter();
			Console.SetOut(w);

			Program.Main(new string[0]);

			_consoleOutput = w.GetStringBuilder().ToString().Trim();
		}
		[TestMethod]
		public void SaysHelloWorld()
		{
			Assert.AreEqual("HelloWorld!", _consoleOutput)
		}
	}
}
```
В строке `Program.Main(new string[0]);` вызывается точка входа в программу. Точкой входа в программу для большинства приложений на `C#` является метод `Main`, поэтому здесь именно он и вызывается. Запись `new string[0]`, которая передается на вход, означает, что мы в качестве аргумента отправляем на точку входа пустой массив для имитации отсутствия аргументов командной строки.
Важный момент - точка входа в программу обычно доступна только для среды выполнения. Под "средой выполнения" я имею в виду как раз тот самый CLR - часть среды `.NET`. Почему она доступна только для среды выполнения? Потому что `Main` является, буквально, точкой входа, т.е. он сам по себе не содержит жесткого исполняемого кода - он только запускает основной алгоритм программы. Это означает, что у него нет никакой другой функции.
Из-за того, что только среда выполнения обычно имеет доступ к `Main`, его никогда не делают публичным - среде выполнения пофиг публичный он или приватный, т.к. она может его выполнить в любом случае. Однако в моей тестирующей программе я пытаюсь получить к нему доступ, а доступ я могу получать только к публичным методам, следовательно, мне нужно сделать приватный метод `Main` публичным.
Я не затронул самый важный вопрос: *где блять вообще здесь метод Main*? А метод `Main` я еще не написал. Чтобы его сделать я использую первую версию программы консольного вывода "Hello, world!":
```C#
using System;

namespace app
{
	public class Program
	{
		public static void Main(string[] args)
		{
			Console.WriteLine("Hello, world!");
		}
	}
}
```
Когда `Main` объявлен, я могу двигаться обратно в тест. В нем мне нужно прописать, что я хочу использовать `Main` именно из проекта с названием `app`. Делается это такой строкой:
```C#
using app;
```
После этого публичные методы станут доступны для вызова. Кстати, что будет если таким способом попытаться вызвать приватный метод `Main`? Получится ошибка:
