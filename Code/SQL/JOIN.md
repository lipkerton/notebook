JOIN - это команда, которая позволяет творить немыслимое: объединять любое количество таблиц в одну по какому-нибудь признаку.

> [!important] Про JOIN крайне важно помнить, что все его виды являются декартовым произведением, то есть мы по определенному условию для каждого сегмента данных из таблицы слева подбираем сегмент данных из таблицы справа.

Многие (только я) задаются вопросом: «Нахуя вообще нужен REFERENCES, если делать JOIN-ы можно вообще между любыми сущностями по любому признаку?» Забудьте. REFERENCES это не обязательно приложение к JOIN - они просто хороши в связке друг с другом, но на самом деле обе команды автономны друг от друга; просто JOIN удобно и полезно использовать вместе с REFERENCES.

## INNER JOIN
> [!important] Это основной вид JOIN, настолько основной, что его можно даже просто писать, как JOIN, а не как INNER JOIN.

В результате работы JOIN в результирующей таблице будут только те значения, которые полностью удовлетворяют условию. Взяты значения будут из правой и левой таблиц по отношению к JOIN.
```SQL
SELECT author.name AS author_name, book.name AS book_name
FROM author
INNER JOIN book
ON author.author_id = book.author_id;
```
Знаете чем полезны вот эти совпадающие имена индексов в изначальной таблице `author` и в таблице `book`? Я сейчас говорю о полях `author_id`, которое есть в табличке `author`, потому что это первичный ключ (PRIMARY KEY), и в табличке `book`, потому что это поле для ссылки на таблицу `author` (FOREIGN KEY). Они, в частности, нужны, чтобы сделать вот так:
```SQL
SELECT author.name AS author_name, book.name AS book_name
FROM author
INNER JOIN book
USING(author_id);
```
И получить идентичный с первым запросом запрос. Команда `USING` пишется вместо условия `ON` и связывает две таблицы по указанному полю. Такое сработает только если в обоих таблицах поле называется одинаковым образом.
## LEFT OUTER JOIN
> [!important] Зачем это нужно? Это нужно, чтобы объединить две таблички так, что из таблицы слева будут взяты Все значения, а из таблицы справа будут взяты только подходящие по условию значения.

Там, где подходящих значений для таблички слева не будет в таблице справа будут проставлены `NULL`. То есть базово - это `INNER JOIN`, только условию будут соответствовать только значения, которые отбираются из таблички справа от `JOIN`.
```SQL
SELECT author.name AS author_name, book.name AS book_name
FROM author
LEFT OUTER JOIN book
ON author.author_id = book.author_id;
```
В результате этого запроса будут взяты все авторы из таблицы `author` и для них будут написаны их книги из таблицы `book`. Если какой-то автор есть, но его книг нет в таблице `book` в правой колонке будет стоять значение NULL.
## RIGHT OUTER JOIN
> [!important] Зачем это нужно? Это то же самое, что и LEFT OUTER JOIN, только наоборот.

Все значения берутся из таблицы, которая указана справа от `JOIN`, а из левой таблички берутся только те, что удовлетворяют условию; во всех остальных случаях будет проставлен `NULL`. Все точно так же. Но не так.
```SQL
SELECT author.name AS author_name, book.name AS book_name
FROM book
RIGHT OUTER JOIN author
ON author.author_id = book.author_id;
```
Этот запрос вернет такой же результат, как запрос в секции LEFT OUTER JOIN ().
## FULL OUTER JOIN
> [!important] Зачем это нужно? Это LEFT JOIN + RIGHT JOIN.

Значения берутся из таблицы слева и таблицы справа, проставляются те пары значений, которые удовлетворяют условию, а те значения для которых не нашлось пары в объединяемых табличках объединяются с значением `NULL`.
```SQL
SELECT author.name AS author_name, book.name AS book_name
FROM author
FULL OUTER JOIN book
ON author.author_id = book.author_id;
```
В результате получится табличка, где будут указаны все авторы и все книги для авторов. Помимо этих пар еще будут указаны авторы без книг и книги без авторов; вместо отсутствующих значений в такой таблице будут проставлены `NULL`.
## CROSS JOIN
> [!important] Это просто никому не нужно.

Эта штука просто выполняет декартово произведение таблицы слева на таблицу справа от `JOIN`. Все. То есть для каждого значения таблицы слева используется каждое значение из таблицы справа.
```SQL
SELECT table1.name AS table1_name, table2.name AS table2_name
FROM table1
CROSS JOIN table2;
```
Как посчитать сколько записей будет в итоге? По всем правилам декартового произведения нужно просто умножить количество записей в первом множестве на количество записей во втором = количество записей в итоговом множестве.
## NATURAL JOIN
> [!important] Это тоже никому не нужно.

`NATURAL JOIN` объединяет таблицы по всем одноименным колонкам в них. То есть ему не нужно условия - он просто найдет все одинаковые названия колонок в двух таблицах и соединит в одну по этим одинаковым колонкам. Звучит удобно, но на практике не очень надежно: во-первых, явное лучше, чем неявное, во-вторых, что делать если в двух таблицах будет несколько одинаковых колонок?
```SQL
SELECT author.name AS author_name, book.book_name AS book_name
FROM author
NATURAL JOIN book;
```
Этот запрос вернет по сути ничего, потому что в обеих таблицах существует несколько колонок с одинаковыми именами, поэтому соединение вернет некорректный результат.
Нормальный запрос выглядел бы так:
```SQL
SELECT author.name AS author_name, book.book_name AS book_name
FROM author
INNER JOIN book
USING(author_id);
```
## SELF JOIN
> [!important] Зачем это нужно? Это нужно когда в таблице существует колонка, значения которой ссылаются на PRIMARY KEY строк в этой же самой таблице.

Например, нам необходимо составить табличку книг и у нас внутри одной таблички Вдруг появилась книга “Война и мир”, которая одновременно представляет из себя 2 тома, т.е. распадается на две книжки. При создании таблицы мы это предусмотрели, поэтому создали поле которое `REFERENCES` на `PRIMARY KEY` книг в этой же самой таблице. Зачем мы так сделали? Потому что мы хотим сохранить в табличку все три варианта “Война и мир”, “Война и мир. Том 1”, “Война и мир. Том 2”, но при этом последние два варианта будут ссылаться на первый, как на “родительскую книжку”.
`SELF JOIN` это по сути `INNER JOIN`, только с обращением таблицы к самой себе.
```SQL
SELECT right_column.name AS parent_book_name, left_column.name AS child_book_name
FROM book AS right_column  -- берем из таблички book.
JOIN book AS left_column  -- объединяем с табличкой book.
ON right_column.book_id = left_column.parent_book_id;
```
Как во всяком `JOIN` здесь две таблички, только на самом деле это одна табличка, которая была взята дважды.

## JOIN LATERAL
> [!important] Для чего нужно это использовать?
> 
> Эта штука нужна, чтобы сделать подзапрос после использования любого из JOIN (то есть справа от JOIN). Но подзапрос не простой, а с использованием данных из той таблицы, которая была указана слева от JOIN.

В обычном JOIN у нас получится ошибка, если попытаться выполнить такой запрос с подзапросом после JOIN:
```SQL
SELECT author.name, b.name
FROM author
LEFT OUTER JOIN (
	SELECT name
	FROM book
	--здесь мы пытаемся получить доступ к левой от JOIN таблице (к колонке a.author_id).
	WHERE book.book_id = author.author_id
) AS b --здесь мы закончили писать подзапрос и дали ему наименование b.
ON true; --условие для объединения таблиц.
```
Когда СУБД дойдет до выполнения подзапроса, она будет выполнять его отдельно, как будто это автономный запрос к базе данных. Поэтому сущность `author.author_id` вызовет ошибку, т.к. подзапрос тупо не в курсе о том, что происходит снаружи его самого - он не знает о существовании таблицы `author`.
Чтобы это исправить нужно просто добавить слово LATERAL после слова JOIN:
```SQL
SELECT author.name, b.name
FROM author
LEFT OUTER JOIN LATERAL (
	SELECT name
	FROM book
	WHERE book.author_id = author.author_id
) AS b
ON true;
```
`LATERAL` можно использовать, когда нам нужно написать не только одно единственное условие объединения двух таблиц, но еще и вытащить из второй таблицы только конкретные данные. Например: я хочу достать все имена авторов и самую продаваемую их книгу (если у их книг вообще есть продажи).
```SQL
SELECT author.name, b.name
FROM author
INNER JOIN LATERAL (
	SELECT name
	FROM book
	WHERE price IN (
		SELECT MAX(price) FROM book GROUP BY author_id
	) AND author.author_id = book.author_id
) AS b
ON true; 
```
Теперь я напишу такой же запрос, но по-человечески.
```SQL
SELECT author.name, b.name
FROM author
INNER JOIN LATERAL (
	SELECT name
	FROM book
	WHERE author.author_id = book.author_id
	ORDER BY price DESC
	LIMIT 1
) AS b
ON true;
```
### `ON true` выполняет декартово произведение?
> [!important] Декартово произведение
> 
> Это когда множество умножается на множество. Если мы говорим о таблицах - то представьте две колонки: для каждого элемента первой колонки используется каждый элемент второй колонки.

В стандартном случае, например:
```SQL
SELECT * FROM table1
INNER JOIN table2
ON true;
```
Такое условие ON вернет именно декартово произведение. Но в случае использования LATERAL такого не случится, т.к. внутри подзапроса, который идет после LATERAL, выборка уже ограничена с помощью WHERE. Отсюда вывод - в этом запросе `ON true` чисто косметический.
## SEMI JOIN
> [!important] Зачем это нужно? Это нужно для того, чтобы достать все данные из левой таблицы, для которых есть соответствие в правой.

SEMI JOIN вообще не особо выглядит как JOIN. На самом деле это слово EXISTS и оно нужно не только для реализации логики JOIN.
```SQL
SELECT * FROM book WHERE EXISTS(SELECT 1);
```
EXISTS проверяет пришло ли ему хотя бы одно значение. Если оно пришло, то он возвращает true. Как это связано с JOIN?
```SQL
SELECT name
FROM book
WHERE EXISTS(
	SELECT 1
	FROM author
	WHERE author.author_id = book.author_id
);
```
После `EXISTS` в данном примере выполняется подзапрос, который будет возвращать цифру 1, если условие `author.author_id = book.author_id` будет выполнено. Цифра здесь не играет никакой роли - нам главное возвращать хоть что-то, а цифра - это константа, поэтому ее вернуть проще всего. `EXISTS` будет ловить цифры из подзапроса и выдавать всегда значение `true`. То есть мы в результате всего запроса будем получать только те книги для которых существует автор.
Условие `author.author_id = book.author_id` ищет совпадение индексов в таблице книг и в таблице авторов через `FOREIGN KEY`, поэтому эта вся история скорее всего и называется `JOIN`.
## ANTI JOIN
> [!important] Зачем это нужно? Это нужно, потому что бывает необходимость проверить, что какие-то данные не существуют в выборке. А ЕЩЕ ЭТО НУЖНО, чтобы заменить синтаксис `not in` в некоторых случаях.

ANTI JOIN это противоположность SEMI JOIN, т.е. это просто слова NOT EXISTS. Если хотя бы одна сущность будет присутствовать в выборке, то выражение не будет удовлетворять условию.
```SQL
SELECT name
FROM author
WHERE NOT EXISTS(
	SELECT 1
	FROM book
	WHERE author.author_id = book.author_id
);
```
В этом запросе мы ищем авторов, для которых у нас нет книг в таблице `book` . Чтобы это реализовать мы используем ANTI JOIN `NOT EXISTS` и в подзапросе прописываем, что для каждой книги, к которой будет найдена пара `author.author_id = book.author_id` мы будем возвращать единичку. Мы идем по списку авторов и когда найдется такой автор для которого не будет пары `NOT EXISTS` вернет true.
### Зачем нужно менять синтаксис `not in`?
Иногда в выборку `SELECT` может  попасть `null` - это значит, что синтаксис `not in`  не сможет на него нормально отреагировать. Почему?
```SQL
SELECT apple
FROM apple
WHERE cost NOT IN (100, 200, 500);
```
Такой код вернет все яблоки, цена которых не равна 100, 200 или 500. Синтаксис `NOT IN` сравнивает `cost` последовательно с каждым элементом в скобках. Условие будет выполнено только если `cost` оказался не равен ничему. Проблемы с `NOT IN` начинаются, когда в скобки попадает `NULL`.
```SQL
SELECT apple
FROM apple
WHERE cost NOT IN (100, 200, 300, NULL);
```
Такой запрос не вернет ничего, даже если в таблице были яблоки, которые стоили не 100, не 200 и не 300. Дело в том, что операция сравнения чего угодно с `NULL` все равно вернет `NULL`. А в этом случае получилось так, что мы все нужные данные последовательно сравнили с каждым элементом в скобках, в т.ч. с `NULL`.
