Если разместить на одном сервере Nginx прокси, Gunicorn application-сервер, application-код и БД на PostgreSQL, то в продакшене все это почти сразу упадет.
Вопрос: что упадет первым? Ответ: БД.
Как решить проблему? Для начала стоит перенести БД на отдельный сервер. Так как, когда application-код дерется за ресурсы системы с СУБД, это приводит к печальным результатам.
Но что если БД на отдельном сервере точно так же начала лагать и выдавать плохие результаты? Теперь уже нужно идти в саму БД и искать в ней проблемы. Чтобы это сделать нужно знать как устроена СУБД, которая постоянно достает из БД данные и кладет их обратно.
В СУБД PostgreSQL (как и в любой другой БД) существует волшебная сущность - оптимизатор. В сущности это штука, которая составляет оптимальный путь для извлечения данных после того как пользователь с помощью языка SQL обозначает какие данные для него нужно достать. 
Основное действие оптимизатора - чтение данных. Как он их читает? По блокам. Стандартный размер блока - 8 КБ; этот размер можно изменить. Блоки называют **страницами (или pages)**. 
PostgreSQL не может считать меньше одной страницы. Это означает, что если нам нужна всего одна строчка из БД и она, например, весит 1 КБ, то СУБД, чтобы ее достать, прочитает одну страницу, т.е. 8 таких строчек.
После того как строчка прочитана она попадает в **буфферный кэш**. Буфферный кэш - это специальная область в оперативной памяти. Там PostgreSQL будет хранить все данные, которые он извлек, пока новые данные их не перезатрут. Фишка кэша в том, что из него данные можно извлечь гораздо быстрее, чем из БД. Поэтому если кто-то будет обращаться несколько раз за однотипными данными PostgreSQL будет отдавать их из буфферного кэша.
Но как вообще PostgreSQL составляет запрос? Это удивительная тема, т.к. в сущности сама СУБД устроена так, чтобы оперативно строить пути для извлечения данных. Допустим у нас есть такой запрос:
```SQL
select * from species;
```
Как СУБД будет строить путь для извлечения этих данных?
Прежде чем ответить на этот вопрос нужно сказать, что PostgreSQL - это СУБД с открытым исходным кодом. Это означает, что всю ее систему можно посмотреть на открытых ресурсах. Другие СУБД, например, Oracle являются системами с закрытым исходным кодом. В этих других системах оптимизатор может быть устроен иначе. Например, в Oracle можно посмотреть на работу оптимизатора и использовать специальные команды (хинты), чтобы влиять на работу оптимизатора, если он строит какие-то не оптимальные пути для извлечения данных. С первого взгляда может показаться, что такой прямой доступ к оптимизатору является серьезным преимуществом, потому что оптимизатор может неправильно или неэффективно выполнять запросы. Но на самом деле это фича PostgreSQL - сделать оптимизатор максимально эффективным.
Посмотреть как оптимизатор выполняет запросы можно с помощью команды `explain`:
```SQL
explain analyze select * from species;
```
Вывод будет примерно следующим:
```SQL
Seq Scan on species (cost=0.00..11.50 rows=150 width=488) (actual time=0.415..0.417 rows=39 loops=1)
Planning Time: 3.056 ms
Execution Time: 0.432 ms
```
Что это все значит?
`Seq Scan` - это способ, которым SQL будет доставать данные из таблицы. В данном случае он просто просканирует всю таблицу.
`cost` - содержит два значения через двоеточие. Первое значение `0.00` - это ожидаемая стоимость прочтения первой строки в табличке. Второе значение `11.50` - это ожидаемая стоимость прочтения всех строк в таблице.
`rows` - примерное количество строк в таблице (по статистике PostgreSQL).
`actual time` - тоже содержит два значения через двоеточие, по смыслу эти значения соответствуют значениям в `cost`, только это уже время, которое на самом деле понадобилось.
В целом первая скобка содержит ожидаемые значения, а вторая значения, которые получились на самом деле. 
Как оптимизатор рассчитывает ожидаемые значения? Время рассчитывается по формулам, а количество строчек считается из статистики PostgreSQL. Статистика не всегда бывает точной, например, в примере выше статистика насчитала 150 строк, когда на самом деле строк было 39. Но посчитать строки по такой статистике - это всегда более оптимальный выбор чем запускать функцию `count()` по табличке, где 1кк значений.
Что за формула, по которой PostgreSQL выстраивает ожидания? Это формула `затраты по времени на чтение страниц + затраты по времени на выполнение всех операций процессором`. Сначала нужно посчитать сколько времени тратится на чтение:
```SQL
SELECT
	relpages, --количество страниц в таблице
	current_setting('seq_page_cost'), --стоимость чтения одной страницы
	relpages * current_setting('seq_page_cost')::int as total --итог
FROM pg_class where relname='species';
```
Теперь сколько времени тратиться на выполнение операций процессором:
```SQL
SELECT
	reltuples, --количество строк
	current_setting('cpu_tuple_cost'), --стоимость чтения одной строки
	reltuples * current_setting('cpu_tuple_cost')::float as total
FROM pg_class where relname='species';
```
Потом сложить получившиеся результаты. Все. Это значение для параметра `cost`. Значение может немного отличаться, потому что мы проводили расчеты вручную, а оптимизатор делает это на ходу.
Какие еще способы сканирования таблицы есть, кроме `Seq Scan`?
1) `Seq Scan` - полное сканирование таблицы.
2) `Index Scan` - индексное сканирование таблицы. Сначала ищем данные в индексе, а потом идем в таблицу вытаскивать данные.
3) `Bitmap Heap Scan` - сканирование по битовой карте. Перед сканированием PostgreSQL анализирует таблицу - он составляет карту, какие страницы в этой таблице ему нужно просканировать для поиска данных, а какие нет.
4) `Index Only Scan` - это когда мы вообще не ходим в таблицу, потому что все нужные данные есть уже в индексе.
По какому принципу PostgreSQL выбирает метод для сканирования? По селективности. **Селективность** - это отношение нужного куска данных к количеству всех данных в таблице, измеряется числом от 0 до 1. Например, в таблице 100 строк, а нам нужна 1 строка, следовательно селективность будет равна `1 / 100 = 0.01`. Селективность называют "высокой", когда ее значение стремиться к нулю и "низкой", когда ее значение стремиться к единице. `Seq Scan` выбирается, когда данных нужно прочитать много, т.е. когда селективность низкая, т.е. когда запрос "длинный". `Index Scan` выбирается, когда данных нужно прочитать очень мало, т.е. когда селективность высокая, т.е. когда запрос "короткий". `Bitmap Heap Scan` выбирается, когда селективность средняя, т.е. запрос не "длинный" и не "короткий".
Когда у нас несколько таблиц, которые мы хотим соединить через `JOIN`, то возникает вопрос: как оптимизатор будет это обрабатывать?
Такой запрос:
```SQL
explain analyze
select 
	s.species_name,
	o.observer_name
from species as s
join observations as o using(species_id);
```
Вернет такие данные:
```SQL
Hash Join  (cost=13.38..24.77 rows=110 width=636) (actual time=0.033..0.036 rows=11 loops=1)
   Hash Cond: (o.species_id = s.species_id)
   ->  Seq Scan on observations o  (cost=0.00..11.10 rows=110 width=326) (actual time=0.007..0.008 rows=11 loops=1)
   ->  Hash  (cost=11.50..11.50 rows=150 width=326) (actual time=0.020..0.020 rows=39 loops=1)
         Buckets: 1024  Batches: 1  Memory Usage: 11kB
         ->  Seq Scan on species s  (cost=0.00..11.50 rows=150 width=326) (actual time=0.006..0.009 rows=39 loops=1)
Planning Time: 0.097 ms
Execution Time: 0.053 ms
```
Тут происходит много всего, но в первую очередь нужно обратить внимание на cлово `Hash`, т.к. это один из способов объединения таблиц - `Hash Join`. Он превращает данные одной таблицы в хэш и сканирует данные другой таблицы на предмет совпадения с данным хэшем. В этом случае сканируется сначала таблица `species` - это сканирование находится в самом низу цепочки. Сканируется она с помощью `Seq Scan`. Далее, на уровень выше составляется хэш из данных по таблице `species`, а затем сканируется полностью таблица `observations`. На следующем уровне происходит `Hash Join` по условию `Hash Cond`.
Всего способов соединять таблицы три:
1) `nested loop` - вложенные циклы. Очень похоже на работу вложенных циклов `for` в Python.
2) `hash join` - объединение по хэшу. Анализируем данные большой таблицы, делаем из нее хэш, а потом сканируем данные малой и с помощью хэш-функции ищем совпадение.
3) `merge join` - соединение двух отсортированных табличек. Используется редко, потому что не всегда таблицы бывают отсортированы. Смысл в том что мы берем две таблицы, сравниваем их по размеру и если размер совпадает, значит делаем соединение на основании признака, по которому они были отсортированы. А если размер не совпадает, то мы начинаем сравнивать таблицы по строкам.
