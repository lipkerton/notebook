Триггеры - это специальные конструкции, которыми можно реагировать да действия, которые делает пользователь с базой данных.
Если под стандартным триггером иметь в виду триггер, который логгирует действия пользователей в системе (например, логгирует все DDL команды), то у него будет три компонента:
1) таблица, которая хранит лог.
2) функция, которая просматривает события и грузит информацию в таблицу.
3) сам триггер.
Триггер описывает то какой функцией мы будем реагировать на то или иное действие и в общем случае его создание выглядит так:
```SQL
create event trigger <имя триггера> on <на что будет триггерить>
execute function <имя функции>;
```
## типы триггеров
Однако триггеры еще бывают двух типов:
1) просто триггер
2) триггер события
Различие в том, что "просто триггер" реагирует на действия с конкретной таблицей, т.е. он реагирует на DML (Data Manipulation Language) команды, которые связаны с одной сущностью в БД.
А "триггер события" не закреплен ни за какой конкретной сущностью - он реагирует на действия, которые влияют на всю базу данных, т.е. на DDL (Data Definition Language) команды.
## как создать ddl триггер?
Во-первых нужно создать функцию. Как ее сделать?
```SQL
create function audit_ddl_command() returns event_trigger
$$  --обозначаем тело функции.
declare obj record;  -- объявляем переменную obj.
begin
end;
$$
```
Здесь уже много всего произошло.
`create function audit_ddl_command() returns event_trigger` - строка создания функции. Все стандартно - ключевое слово `create`, после него идет то слово, которое подсказывает что мы будем создавать `function`, затем название функции и в конце то, что она будет возвращать. Функция, которая делается для триггера событий обязательно должна возвращать объект `event_trigger`.
`$$` - это знаки, которые обозначают начало и конец тела функции. Они выступают как замена стандартным кавычкам. По умолчанию тело функции всегда пишется в кавычках, но это плохо сочетается с тем, что внутри функции тоже могут быть кавычки, поэтому начальные кавычки заменили на знаки доллара.
`declare` - специальная команда, которая используется всегда, когда нам нужно объявить переменную.
`obj` - имя переменной.
`record` - тип данных переменной. В данном случае `record` означает, что в переменной будет храниться особая строка, из которой можно будет получить значения атрибутов по их именам.
`begin/end` - это ключевые слова, которые начинают блок кода функции и заканчивают его.
```SQL
create function audit_ddl_command() returns event_trigger as
$$  --обозначаем тело функции.
declare obj record;  -- объявляем переменную obj.
begin
  for obj in select * from pg_event_trigger_ddl_commands() loop
    insert into ddl_log(
	  user_name,
	  operation,
	  object_type,
	  object_name
	) values (
	  user,
	  obj.command_tag,
	  obj.object_type,
	  obj.object_identity
	)
  end loop;
end;
$$ language plpgsql;
```
Внутри цикла мы просим функцию `pg_event_trigger_ddl_commands()` отдать нам команды, которые только что произошли в БД. Одна пользовательская команда может выполнить сразу целый набор команд поменьше, поэтому нам нужен цикл, чтобы перебрать каждую и каждую отправить на хранение в нашу таблицу `ddl_log`.
Каждая команда будет попадать в переменную `obj`, которую мы объявили раньше и потом в конструкции `insert` из нее будут вытаскиваться данные - `obj.command_tag`, `obj.object_type`, `obj.object_identity`.
Переменная `user` всегда показывает имя текущего пользователя БД.
В конце после двух долларов мы пишем название языка, на котором была выполнена команда.
Теперь нужно создать таблицу `ddl_log` и, наконец, сам триггер:
```SQL
create table ddl_log (
	ddl_log_id bigint generated always as identity primary key,
	time_stamp timestamp default now(),
	user_name text,
	operation text,
	object_type text,
	object_name text
)
create event trigger ddl_command on ddl_command_end
execute function ddl_log(); -- сам триггер.
```
Здесь важна конструкция `ddl_command_end` - она означает, что триггер будет реагировать на команды, которые уже отработали и свое дело сделали.
## как создать dml триггер?
Для того, чтобы написать DML триггер потребуется изменить функцию. Для начала она теперь должна возвращать не `event_trigger`, а просто `trigger`.
Далее, чисто теоретически мы могли бы так же в цикле вытаскивать какие-то события. Но зачем, если можно определить событие в конструкции `if-elsif-else` и для каждой DML команды настроить свое поведение.
Например, мы можем настроить функцию на обработку `UPDATE` запроса к таблице и сохранять не только команду, пользователя, но и те данные, которые в таблице хотят обновить.
```SQL
create function audit_dml_command() returns trigger as
$$
begin
  if (tg_op = 'UPDATE') then
    insert into dml_log(
	  user_name,
	  operation,
	  data_id,
	  data_name
	) select user, 'U', new.*;
	return new;
  end if;
end;
$$ language plpgsql;
```
Здесь я применяю конструкцию `if` в теле функции, чтобы узнать какое действие было использовано пользователем для работы с таблицей.
Далее я использую специальную переменную `tg_op = 'UPDATE'`. Эта переменная хранит DML команду пользователя и из нее эту команду можно достать. Но нужно быть осторожным - нельзя переменную вставить в саму таблицу (я не знаю почему - это вызывает ошибку).
Далее идет стандартная команда `insert`. В ней понятно все, кроме `new` - это переменная типа `record`, которая содержит строку с новыми данными, которые мы будем вставлять в таблицу. В данном случае `new.*` распакует эту строку и достанет все данные. Единственная необходимость в этом случае - заранее в `insert` подготовить все колонки целевой таблицы, чтобы данные туда поместились.
Теперь создаем таблицу. Для примера я еще создам целевую таблицу:
```SQL
-- целевая таблица.
create target(
	data_id bigint generated always as identity primary key,
	data text
)
-- лог таблица
create dml_log(
	ddl_log_id bigint generated always as identity primary key,
	time_stamp timestamp default now(),
	user_name text,
	operation text,
	data_id bigint,  -- такое же поле из целевой.
	data_name text   -- такое же поле из целевой.
)
-- триггер.
create trigger dml_command after update on target
for each row execute function audit_dml_command();
```
Здесь появилось новое в триггере - `after update` и `for each row`. Первая команда обозначает после какой операции триггер должен действовать. Вторая команда означает для каких сущностей нужно выполнить функцию. В данном случае функция будет выполняться для каждой строки, которую пытаются обновить.
# еще примеры
## пример с логами
Я создам две таблицы:
- таблица клиентов.
- таблица, которая фиксирует изменения в таблице клиентов.
```SQL
create table client(
	client_id bigint generated always as identity primary key,
	first_name varchar(200) not null,
	last_name varchar(200),
	middle_name varchar(200)
)
create table client_log(
	client_log_id bigint generated always as identity primary key,
	client_id bigint,
	action text,
	change_time timestamp default now(),
	new_first_name varchar(200),
	new_last_name varchar(200),
	new_middle_name varchar(200),
	old_first_name varchar(200),
	old_last_name varchar(200),
	old_middle_name varchar(200)
)
```
Теперь я напишу функцию, которую будет использовать триггер для реакции на события в таблице `client`:
```SQL
create or replace function client_log_event()
returns trigger
as
$$
begin
	insert into client_log(
		client_id, action, change_time,
		new_first_name, new_last_name, new_middle_name,
		old_first_name, old_last_name, old_middle_name
	) values (
		coalesce(new.client_id, old.client_id), tg_op, now()
		new.first_name, new.last_name, new.middle_name,
		old.first_name, old.last_name, old.middle_name	
	);
	return new;
end;
$$
language plgsql;
```
Поскольку функция будет участвовать в триггере, внутри нее доступны специальные объекты - `new` и `old`. Из них можно получить состояние старых данных и состояние тех данных, которые пытаются изменить состояние таблицы.
Точно так же доступны некоторые особенные константы. Например, константа `tg_op` содержит в себе тип выполняемого действия. То есть триггер запишет в таблицу какое конкретно действие было проведено с состоянием БД.
Теперь осталось написать триггер:
```SQL
create trigger client_changes_trigger
	after insert or update or delete on client
	for each row
	execute function client_log_event();
```

## пример с валидацией
Для работы триггера не обязательно делать доп таблицу. С помощью триггера можно валидировать входящие данные:
```SQL
-- таблица
create table client(
	client_id bigint generated always as identity primary key,
	name varchar(200) not null,
	email varchar(200) not null
);
-- функция
create or replace function client_email_change()
returns trigger
as 
$$
begin
	if new.email distinct from old.email then
		raise exception 'нельзя менять поле `email`';
	end if;
	return new;
end;
$$
language plpgsql;
-- триггер
create trigger prevent_email_change
before update on client
for each row
execute function client_email_change();
```

# правила
Правила - это что-то вроде функций, которые меняют стандартное поведение обычных SQL команд. Например, с помощью правил в БД можно реализовать "мягкое удаление" данных - вместо выполнения команды `DELETE` на самом деле будет выполняться команда `UPDATE`.
Чтобы показать эту штуку в действии я создам таблицу клиентов (снова):
```SQL
create table client(
	client_id bigint generated always as identity primary key,
	name varchar(200) not null,
	is_deleted boolean default false not null
);
```
И теперь напишу правило, которое должно менять команду `DELETE` для этой таблицы на команду `UPDATE`:
```SQL
create rule soft_delete as
on delete to client
do instead
	update client set is_delete=true where client_id = old.client_id;
```
Все!
Если попытаться удалить данные из таблицы с помощью `DELETE` то запрос будет выполнен, однако вместо стандартной команды будет выполнена логика правила.
С помощью правил можно вообще запретить делать что либо с данными в таблице:
```SQL
create rule for_fun as
on delete to client
do instead nothing;
```
Теперь при выполнении `DELETE` с данными не произойдет ничего.

create or replace function body_hash()

    if (tg_op = 'INSERT') then

            where id = old.id;
    end if;
    return new;
end;
$$
