## база
CTE - common table expressions. Общие табличные выражения. Это штука, которая выполняется внутри запроса и во многом ее работа похожа на подзапрос, но не совсем.
CTE -  это вычисление запроса в рамках одного большого запроса. CTE создает виртуальную таблицу на время выполнения запроса. Зачем это нужно если можно просто написать подзапрос? 
Иногда бывают ситуации, когда нам нужно обращаться на протяжении одного запроса к одним и тем же данным несколько раз. Например, я хочу вытащить все уникальные цвета птиц из таблицы и посчитать сколько птиц каждого цвета у нас сохранено в таблице. Для решения этой задачи я могу использовать подзапрос:
```SQL
select
	distinct s1.primary_color,
	(
		select count(s2.species_id)
		from species as s2
		where s1.primary_color = s2.primary_color
		group by s2.primary_color
	)
from species;
```
Да, я знаю такие же данные можно было бы получить другими более эффективными способами, но для демонстрации сойдет и этот.
Вроде как задача решена - данные получены, но что если я вдруг захочу добавить какое-нибудь условие в `where` к главному запросу и при этом это условие будет связано именно с результатом подзапроса? 
Например, я хочу получить только те данные о цветах, в которых можно насчитать больше пяти птиц. Если я не знаю о том в каком порядке выполняются команды SQL я представлю, что эту новую задачу можно решить с помощью псевдонима:
```SQL
select
	distinct s1.primary_color,
	(
		select count(s2.species_id)
		from species as s2
		where s1.primary_color = s2.primary_color
		group by s2.primary_color
	) as counter  -- добавили псевдоним.
from species
where counter >= 5;
```
Такой запрос покажет ошибку - операция `where` выполняется до `select`, поэтому на своем этапе `where` еще знать не знает ни о каких псевдонимах.
Ошибка:
```SQL
select 
	distinct s1.primary_color,
	(
		select count(s2.species_id)
		from species as s2
		where s1.primary_color = s2.primary_color
		group by s2.primary_color
	) as counter
from species as s1
where counter >= 5;

СТРОКА 1: ...rimary_color) as counter from species as s1 where counter >=...
```
Здесь мы сталкиваемся с самой неприятной неприятностью - нужно переписывать весь подзапрос в `where`, чтобы сравнить его с пятеркой и получить верный результат. Получится что-то такое:
```SQL
select 
	distinct s1.primary_color,
	(
		select count(s2.species_id)
		from species as s2
		where s1.primary_color = s2.primary_color
		group by s2.primary_color
	) as counter
from species as s1
where (
	select count(s2.species_id)
	from species as s2
	where s1.primary_color = s2.primary_color
	group by s2.primary_color
) >= 5;
```
Это выглядит очень неэффективно, потому что запросы выполняются дважды.
Что можно с этим сделать? Например, расчитать подзапрос один раз и применять его результат в тех местах запроса, где он нам нужен, через имя "переменной". При этом я не хочу создавать вью, потому что она сохраниться на уровне всей БД, - мне не нужно такое поведение.
В этой ситуации можно воспользоваться CTE:
```SQL
with test_cte as (
	select 
		primary_color, 
		count(species_id) as counter
	from species
	group by primary_color
)
select
	primary_color,
	counter
from test_cte
where counter >= 5;
```
Пока я писал это, то понял, что это не самый красноречивый пример. Смысл, короче, в том, что мы можем убрать какой-то запрос на выполнение в конструкцию `with`. Чем это поможет? Во-первых, если данные из подзапроса нужны несколько раз на протяжении всего запроса. Во-вторых, если нам нужно выполнить атомарную операцию с БД и при этом включить в нее выполнение еще одной операции. В-третьих, это нужно если нам нужны какие-то данные только на период выполнения запроса.
## материализованные CTE
Выше я написал небольшую строчку о том, что CTE создает виртуальную таблицу на время выполнения запроса. Это может создать впечатление, что PostgreSQL всегда выполняет конструкцию `with` до основного тела запроса, а после начинает уже работать с основной командой. В реальности это не всегда так - в реальности может сложиться обратная картина: PostgreSQL увидит CTE и умным образом включит команду, которую мы положили в CTE, в основной запрос, то есть он будет выполнять наше CTE, как часть основного запроса.
Однако что если нам нужно, чтобы расчет какой-то команды был произведен именно один раз до начала выполнения запроса? Такое поведение может понадобится, когда нам необходимо использовать какую-то тяжелую функцию, чтобы сделать по ней виртуальную таблицу и использовать эту виртуальную таблицу при работе основного запроса.
Эту механику можно реализовать с помощью материализованного CTE:
```SQL
with materialized <имя> as (
	<запрос>
) <основное тело запроса>;
```
Добавляется всего одно слово - `materialized`.
