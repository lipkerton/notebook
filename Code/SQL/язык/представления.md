Представления иногда называют "вьюшка", потому что на английском эта сущность называется "view".
Представление - это как переменная, которую можно использовать, чтобы достать какие-то данные прямо как из обычной таблицы. Но на самом деле представление является запросом к настоящим таблицам, который возвращает какие-то данные.
Существуют представления:
1) материализованные
2) нематериализованные
## нематериализованные представления
Чтобы создать нематериализованное представление нужно иметь какой-то запрос, который мы хотим запихнуть в такое представление. Например, у меня есть сложный запрос с несколькими `JOIN`:
```SQL
select f.family_name, g.genus_name, s.species_name
from family f
left join genus g using(family_id)
left join species s using(genus_id);
```
Если я захочу запихнуть такой запрос в view, то получится как-то так:
```SQL
create view family_genus_species_name as
	select f.family_name, g.genus_name, s.species_name
	from family f
	left join genus g using(family_id)
	left join species s using(genus_id);
```
Тогда все данные из получившейся таблице можно будет вытащить, если мы обратимся к имени созданной view:
```SQL
select * from family_genus_species_name;
```
Вроде как очень удобно. Однако существуют некоторые проблемы.
Во-первых, эта вьюшка нематериальная. Это означает, что при каждом ее вызове по имени мы по сути будем выполнять запрос, который в ней содержится. То есть такая вьюшка не является таблицей - это просто запрос, который спрятан за удобным наименованием.
Во-вторых, проблема, которая вытекает из предыдущего пункта связана с тем, что попытка вытащить несколько раз данные из такой вьюшки обернется многократным вызовом таблицы:
```SQL
explain (analyze, buffers)
select n1.family_name, n2.genus_name
from family_genus_species_name n1
join family_genus_species_name n2
on n1.family_name = n2.family_name;
```
Каждое обращение к view  в такой ситуации будет полным повторением заложенного в нее запроса. Это может сказаться на производительности в будущем.
## материализованные представления
Это тоже представления, но есть нюанс - они действительно являются таблицами. 
Обычно такие таблицы стоит использовать как кэш. Например, у нас есть какой-нибудь супер-тяжелый запрос, который мы хотим выполнять, чтобы использовать где-то данные из него. В таком случае можно создать материализованное представление.
При создании материализованное представление выполняет запрос, сохраняет запрос, а еще сохраняет полностью все данные, которые получил запрос, в виде таблицы.
От нематериализованной view это отличается тем, что в следующий раз при попытке вытащить данные из материализованной view мы не будем выполнять запрос - мы обратимся за данными в таблицу, которая была уже создана этим запросом раньше.
Чтобы создать материализованную view к запросу, который я написал ранее надо добавить слово `materialized`:
```SQL
create materialized view family_genus_species_name as
	select f.family_name, g.genus_name, s.species_name
	from family as f
	left join genus as g using(family_id)
	left join species as s using(genus_id);
```
Теперь при попытке обратиться с запросом:
```SQL
explain (analyze, buffers)
select n1.family_name, n2.genus_name
from family_genus_species_name n1
join family_genus_species_name n2
on n1.family_name = n2.family_name;
```
Мы увидим, запрос, который мы прописали при создании view не выполняется дважды - таблица читается единожды, потому что view материализована и хранит в себе все нужные данные.

>[!important] Обновляет ли такая view актуальность данных внутри себя, если они обновляются в исходных таблицах? Нет! Обновлять материализованную view нужно вручную.

Обновление материализованной view может быть выполнено двумя способами. Они принципиально различаются только тем, что в первом способе таблица будет заблокирована - это значит, что пока она обновляется к ней нельзя будет обращаться и брать из нее данные. А во втором способе обращаться к таблице и брать из нее данные будет уже можно.
Преимущество первого способа в том, что он выполняется быстрее:
```SQL
refresh materialized view family_genus_species_name;
```
Когда мы попробуем выполнить обновление вторым способом (конкурентно):
```SQL
refresh materialized view concurrently family_genus_species_name;
```
Скорее всего PostgreSQL потребует создать для таблицы уникальный индекс. Для моей таблицы я создал его так:
```SQL
create unique index fgs_pk
on family_genus_species_name(
	family_name, genus_name, species_name
);
```
