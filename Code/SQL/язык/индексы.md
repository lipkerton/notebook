# тестовая таблица и данные
Чтобы тестировать я создам таблицу `entity`:
```SQL
create table entity (
	entity_id biging generated always as identity primary key,
	name varchar(50)
);
```
Срыв покровов: на самом деле тот самый `id`, который мы создаем для каждой таблицы, сам является индексом. Это можно заметить если попросить у интерфейса `psql` список сущностей в текущей БД с помощью `\d entity`:
```SQL
lessons=> \d entity
...
Indexes:
    "entity_pkey" PRIMARY KEY, btree (entity_id)
```
Помимо прочей информации там будет рубрика `Indexes`, где сразу указан первичный ключ таблицы `entity`, который принадлежит к типу `btree`.
Теперь я хочу вставить в таблицу кучу данных. Данных много, потому что иначе не получится нормально протестировать возможности индекса:
```SQL
lessons=> insert into entity(name)
select substring(
	md5(random()::text) || md5(random()::text)
	from 1 for (floor(random() * 46)::int + 5)
)
from generate_series(1, 10000000);
```
Вставка будет происходить некоторое время. Что за функции тут использованы?
1) `md5()` преобразует строку в хэш с помощью алгоритма `md5`.
2) `random()` возвращает случайное число в диапазоне от 0 до 1.
3) `floor()` делает округление в меньшую сторону.
4) `substring()` извлекает подстроку из строки.
5) `generate_series()` генерирует число повторений. 
В данном случае происходит следующее: 1) делаем два хэша из двух рандомных чисел, складываем получившиеся строки, передаем это в качестве первого аргумента для функции `substring`; 2) в качестве второго аргумента мы должны передать "с какого по какой" символ мы заберем из строки в первом аргументе, поэтому здесь написано `from 1` (с первого символа) и `for` (далее нужно написать до какого символа); 3) вместо конечного символа у нас функция:
- `random() * 46` - получаем число от 0 до 1 и умножаем его на 46. Получившееся число будет в диапазоне от 0 до 46.
- `floor(random() * 46)` - округляем любой результат в меньшую сторону. Получившееся число будет в диапазоне целых чисел от 0 до 45.
- `floor(random() * 46)::int + 5` явно преобразуем получившееся число в `int` и добавляем пять. Получившееся число будет в диапазоне от 5 до 50.
То есть из хэшированной строки будут выбраны символы от 1 до 5-50. Даже не спрашивайте почему для этого понадобилась такая сложная математика.
В итоговой таблице получатся примерно такие сущности:
```SQL
lessons=> table entity limit 2;
 entity_id |               name
-----------+-----------------------------------
         1 | 287d221015b64ddd96df1864feb1
         2 | 08ce4d01004ede8da46d17c05989156b4
(2 rows)
```
Так же можно посмотреть размер таблицы в `psql` с помощью команды `\dt+ <имя таблицы>`:
```SQL
lessons=> \dt+ entity
                                   List of relations
 Schema |  Name  | Type  |  Owner  | Persistence | Access method |  Size  | Description
--------+--------+-------+---------+-------------+---------------+--------+-------------
 public | entity | table | student | permanent   | heap          | 653 MB |
(1 row)
```
Хотя обсидиан в таком режиме отображения показывает таблицу криво, здесь можно разглядеть, что ее размер теперь равен 653 МБ.
Так же это можно посмотреть с помощью команды:
```SQL
select pg_relation_size('entity');
-- или
select pg_size_pretty(pg_relation_size('entity'))
```
В первом случае будет показан вывод в килобайтах:
```SQL
lessons=> select pg_relation_size('entity');
 pg_relation_size
------------------
        684376064
(1 row)
```
А во втором в мегабайтах:
```SQL
lessons=> select pg_size_pretty(pg_relation_size('entity'));
 pg_size_pretty
----------------
 653 MB
(1 row)
```
# начинаем запрашивать
Для начала я поставлю таймер, который будет подсказывать сколько времени выполнялся запрос:
```SQL
\timing
```
И вот теперь напишу запрос:
```SQL
lessons=> select entity_id, name from entity where name='aaa';
 entity_id | name
-----------+------
(0 rows)

Time: 1524.726 ms (00:01.525)
```
Поиск сущности с именем `aaa` по таблице занял 1524 мс. Это конкретно не быстрый результат. Можно получить результат получше, если сделать индекс:
```SQL
lessons=> create index idx_identity_name on entity(name);
```
В данном случае индекс строится по колонке `name` и на время его постройки таблица блокируется для запросов. Проверить, что он создан можно по прежнему:
```SQL
lessons=> \d entity
...
Indexes:
    "entity_pkey" PRIMARY KEY, btree (entity_id)
    "idx_identity_name" btree (name)  -- новый индекс.
```
При этом на весе самой таблицы индекс не отразиться, т.к. индекс - это избыточная структура, которая хранится отдельно.
Посмотреть размер индекса можно с помощью:
```SQL
\di+ <имя_индекс>
```
Результат:
```SQL
lessons=> \di+ idx_identity_name
                                             List of relations
 Schema |       Name        | Type  |  Owner  | Table  | Persistence | Access method |  Size  | Description
--------+-------------------+-------+---------+--------+-------------+---------------+--------+-------------
 public | idx_identity_name | index | student | entity | permanent   | btree         | 475 MB |
(1 row)
```
Опять все коряво, но видно, что индекс весит 475 МБ.
Теперь я попробую поискать `bbb` (потому что запрос с `aaa` мог закэшироваться):
```SQL
lessons=> select entity_id, name from entity where name='bbb';
 entity_id | name
-----------+------
(0 rows)

Time: 3.118 ms
```
Вот это уже время для крутых. Именно за такое время должен осуществляться поиск сущностей в БД в идеальной ситуации.
# по нескольким колонкам
Бывает так, что индексы создаются по нескольким колонкам. Здесь нет ничего необычного. В данном случае индекс ничем не отличается от того индекса, который я создавал ранее.
Однако важно помнить одну особенность - порядок признаков, на основании которых создается составной индекс, крайне важен. 
Допустим у нас есть три признака A, B, C. Если мы будем совершать поиск по таблице по признаку A, то наш индекс по этим признакам будет использован. Точно так же он будет использован при поиске по A + B и при поиске по A + B + C. Но! Наш индекс не будет использован при поиске по B или по B + С, или только по C.
Я создал новую таблицу и заполнил данные:
```SQL
create table entity(
	entity_id bigint generated always as identity primary key,
	name text,
	address text
)
insert into entity(name, address)
select 
  substring(md5(random()::text) || md5(random()::text)
            from 1 for (floor(random() * 46)::int + 5)),
  substring(md5(random()::text) || md5(random()::text)
            from 1 for (floor(random() * 41)::int + 10))
from generate_series(1, 10000000);
```
Создаю индекс:
```SQL
create index entity_name_address_idx on entity(name, address);
```
И анализирую поиск:
```SQL
explain analyze select * from entity where name == 'aaa';
explain analyze select * from entity where name == 'aaa' and address == 'bbb';
explain analyze select * from entity where address == 'bbb';
```
Из всех вариантов только в последнем использовался тип сканирования `Parallel Seq Scan`, т.е. полное сканирование таблицы.
# уникальные индексы
В PostgreSQL так же существуют индексы, которые требуют от признаков, по которым они создаются, уникальности значений. Такой индекс можно сделать, если при создании БД указать слова `primary key` или `unique` для колонок.
К слову `primary key` требует не только уникальности значений, но и включает в себя параметр `not null`. Если мы используем `unique`, то `not null` нужно упоминать отдельно.
Допустим у меня есть таблица:
```SQL
lessons=> create temp table some_unique_values (
lessons(> some_unique_value_id bigint generated always as identity primary key,
lessons(> value varchar(30) unique);
```
И я закидываю в нее значения:
```SQL
lessons=> insert into some_unique_values (value) values
lessons-> ('hello'), (''), ('bye'), (null);
```
Очевидно, что из-за параметра `unique` я не смогу больше вставить в колонку еще одно слово `hello`. Однако тут стоит заметить, что я смогу вставить еще один `null`, потому что любое сравнение с `null` даст `null`, соответственно определить, что такие значения уже есть в колонке невозможно.
Кстати, уникальный индекс называется так:
```SQL
"some_unique_values_value_key" UNIQUE CONSTRAINT, btree (value)
```
Вручную уникальный индекс можно создать так:
```SQL
create unique index some_index on table_1(value_1, value_2);
```
