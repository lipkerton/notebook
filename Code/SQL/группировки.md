## DESTINCT
Эта штука, которая выбирает уникальные значения в колонке. Пишется перед названием колонки после слова SELECT.
```SQL
SELECT DISTINCT author FROM book;
```
## GROUP BY
Группирует значения в колонках. Результатом будет новая таблица, где в целевой колонке будут уникальные значения.
```SQL
SELECT author FROM book GROUP BY author;
```
## SUM, COUNT
`SUM` - суммирует числа столбца.
`COUNT` - считает количество строк.
```SQL
SELECT SUM(price), COUNT(amount) FROM book;
```
Это две функции группировки. Они обе вернут именно табличку при использовании.
С их помощью мы сможем сгруппировать таблицу по авторам, найти для каждого автора сумму цен его книг и их количество.
```SQL
SELECT author, SUM(price), COUNT(amount) FROM book GROUP BY author;
```
Группируем мы по автору - оставляем только уникальные имена. Затем к делу подключаются функции SUM и COUNT. Они видят цель группировки и начинают работу отдельно по каждой группе, т.е. отдельно по каждому автору. Это означает, что суммироваться для каждого автора будут те числа, которые были подписаны его именем в изначальной таблице book.
## MAX, MIN, AVG
Это тоже функции группировки. Они тоже возвращают таблички при использовании.
`MAX` - возвращает максимальное значение по столбцу.
`MIN`- возвращает минимальное значение по столбцу.
`AVG` - возвращает среднее значение по столбцу.
```SQL
SELECT MIN(price), MAX(price), AVG(price) FROM book;
```
Касательно группировок мы можем проворачивать с ними то же самое, что проворачивали с COUNT и SUM. Например, найти максимальную цену книги для каждого автора.
## Что делать если…?
У нас есть автор, а у него куча книг. Для каждой книги в табличке прописано имя одного и того же автора. Можно ли все это сгруппировать в одну строчку? Нет. Такой запрос выдаст ошибку, потому что SQL попытается запихнуть в одну строчку для одного автора сразу несколько значений - это **ошибка группировки**.
```SQL
SELECT author, title FROM book GROUP BY author;
```
В результате группировки в каждой строке должно быть одно сгруппированное значение.
## Задача 1
Я хочу найти только те названия книг, которые имеют максимальную цену по автору. Тогда мне нужно сгруппировать все по автору и в итоге получить табличку, в которой будут только названия книг.
```SQL
SELECT title
FROM book
WHERE price IN (
	SELECT MAX(price)
	FROM book
	GROUP BY author
);
```
Здесь я получаю только те заголовки книг, цены которых находятся в табличке максимальных цен книг по каждому автору.
>[!important] **я блядский дегенерат:** короче, когда я в первый раз писал этот конспект Я НА АБСОЛЮТНО СЕРЬЕЗНОМ ЕБЛЕ размышлял так: “Если я обращаюсь к ПОДЗАПРОСУ, который выполняет SELECT по той же самой таблице, то это означает, что я выполняю SELF JOIN”. Так вот: ЭТО ЕБАНЫЙ БРЕД. Подзапрос возвращает отдельную таблицу, которая никак не связана с изначальной таблицей.

Можно попробовать сделать решение через JOIN. Левая таблица - book, а правая - подзапрос, который будет отбирать максимальную цену по автору.
```SQL
SELECT title
FROM book
JOIN (
	SELECT MAX(price)
	FROM book
	GROUP BY author	
) AS p
ON book.price = p.price;
```
Здесь я обращаюсь к самой табличке с помощью JOIN. Внутри подзапроса к табличке мы находим максимальные цены по каждому автору, а в условии объединения табличек используем колонку price. Таким образом в ответе у нас будут только совпадения между ценами из первой и второй табличек.
Можно попробовать решить через JOIN LATERAL (про LATERAL тут: )
```SQL
SELECT DISTINCT book1.author, book2.title
FROM book AS book1
JOIN LATERAL (
	SELECT title
	FROM book
	WHERE book.author = book1.author --для каждого автора таблицы book1 ищем его книги.
	ORDER BY price DESC
	LIMIT 1
)
ON true;
```
Здесь первым шагом мы делаем колонку уникальных авторов. Далее мы обращаемся к самой табличке через JOIN LATERAL. Внутри подзапроса мы по условию ищем наборы книг для каждого автора отдельно, далее сортируем наборы по цене в порядке убывания и берем первую книгу. Объединяем получившиеся таблицы. LATERAL здесь нужен, чтобы внутри подзапроса обратиться к левой части JOIN.

## HAVING
HAVING - это буквально WHERE, только для групповых функций.
>[!important] `HAVING` используется после того как данные были сгруппированы, с помощью `GROUP BY`. В `WHERE` нельзя использовать групповые функции, потому что `WHERE` - это условие, которое проверяется **ДО** группировки. 
```SQL
SELECT author.name AS author_name, COUNT(book.name) AS books_counter
FROM author
INNER JOIN book
USING(author_id)
GROUP BY author.name
HAVING COUNT(book.name) > 1;
```
Можно ли сделать такое с помощью WHERE? Да, но запрос будет другим:
```SQL
SELECT author_name, books_counter
FROM (
	SELECT author.name AS author_name, COUNT(book.name) AS books_counter
	FROM author
	INNER JOIN book
	USING(author_id)
	GROUP BY author.name
)
WHERE books_counter > 1;
```
Почему нужно делать именно так?
Потому что в первом случае HAVING считает книги именно по каждой группе, которую объединил GROUP BY, то есть он делает для каждого автора отдельный счетчик.
И потому что групповые функции нельзя применять в конструкции WHERE.
Такой запрос вернет ошибку:
```SQL
SELECT author.name AS author_name, COUNT(book.name) AS books_counter
FROM author
INNER JOIN book
USING(author_id)
WHERE COUNT(book.name) > 1
GROUP BY author.name;
```
## JSON_AGG и JSONB_AGG (и другие)
### JSON_AGG и JSONB_AGG
Есть такая тема, когда хочется сложить сразу несколько значений в одну ячейку. Это можно провернуть с помощью множества функций, но две самые крутые - `json_agg()` и `jsonb_agg()`. По сути они просто берут несколько значений и запихивают их в json-формате в одну ячейку. Это функции агрегирования и они полезны, когда нужно, например, сгруппировать данные.
Представим у нас есть таблица книг и таблица авторов - они связаны связью многие-ко-многим. Я хочу объединить эти таблицы так, чтобы в одной колонке у меня были все авторы, а в другой для каждого автора был указан список его книг.
```SQL
SELECT
	author.name as author_name,
	json_agg(book.name) as book_name
FROM author
LEFT OUTER JOIN book_to_author USING(author_id)
LEFT OUTER JOIN book USING(book_id)
GROUP BY author.name;
```
### другие
Помимо этих двух функций существуют еще функции, которые проворачивают примерно тот же фокус, только превращают данные не в json, а во что-то другое, например, в array.
- `array_agg()` - массив.
- `string_agg()` - эта принимает два аргумента. Второй - сепаратор между фрагментами данных.
```SQL
SELECT
	author.name AS author_name,
	string_agg(book.name, ', ')
FROM author
LEFT OUTER JOIN book_to_author USING(author_id)
LEFT OUTER JOIN book USING(book_id)
GROUP BY author.name
ORDER BY author.name;
```
Это все выглядит круто, но что если нам вдруг понадобится отсортировать книги по группам до того как они попадут в функцию `string_agg()`? На этот случай у функции `string_agg()` и у любой другой подобной функции группировки есть дополнительный аргумент `ORDER BY`.
```SQL
SELECT
	author.name AS author_name,
	string_agg(book.name, ', ' ORDER BY book.name)
FROM author
LEFT OUTER JOIN book_to_author USING(author_id)
LEFT OUTER JOIN book USING(book_id)
GROUP BY author.name
ORDER BY author.name;
```
## FILTER
Выражение `FILTER` ставиться после агрегатной функции со звездочкой в выборке. После фильтра пишется какое-нибудь условие с `WHERE`.
```SQL
SELECT
	count(*) FILTER (WHERE publication_year BETWEEN 1900 AND 1999) AS year_90s_lit
FROM book_detail;
```
