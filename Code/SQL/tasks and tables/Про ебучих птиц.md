## Создаем таблицы
```SQL
create table family(
	family_id bigint generated always as identity primary key,
	family_name varchar(150) not null check (length(family_name) >= 2),
	description text check (length(description) >= 20)
);  --здесь будут храниться семейства птиц.

create table genus(
	genus_id bigint generated always as identity primary key,
	genus_name varchar(150) not null check (length(genus_name) >= 2),
	description text check (length(description) >= 20),
	family_id bigint not null references family(family_id)
);  --здесь будут храниться роды птиц.

create table species(
	species_id bigint generated always as identity primary key,
	species_name varchar(150) not null check (length(species_name) >= 2),
	description text check (length(description) >= 20),
	average_length smallint not null check (average_length > 0),
	average_weight smallint not null check (average_weight > 0),
	primary_color varchar(50) not null check (length(primary_color) >= 2),
	genus_id bigint not null references genus(genus_id)
);  --здесь будут храниться виды птиц.

create table observations (
    observation_id bigint generated always as identity primary key,
    species_id bigint not null references species(species_id),
    observer_name varchar(150),
    observation_date date,
    location varchar(150),
    notes text
);  --здесь будут храниться наблюдения за птицами.
```

## Задача 1
Достать всех наблюдателей, которые хотя бы раз наблюдали представителей семейства Синицевые. Вывести на экран табличку, где будут имена этих счастливчиков и даты, когда они наблюдали Синицевых.
Все.
Это вся задача.
И это блять **худший** кошмар в моей жизни.
Базовый алгоритм такой - у нас четыре таблицы.
```Mermaid
graph TD
  family --> genus --> species --> observations
```
Нюанс в том, что в табличке `observations` дана конкретная птица, которую наблюдали наблюдатели. Дана она через связь с таблицей `species` через `species_id` . То есть, чтобы понять к какому семейству принадлежит эта птица, сначала нужно постучаться в `species` , затем в связанную таблицу `genus` , а уже оттуда в табличку `family` .
Первый вариант самый простой и я до него даже не додумался, потому что не практиковался в связях сразу нескольких таблиц:
```SQL
SELECT observer_name, observation_date
FROM observations
JOIN species USING(species_id)
JOIN genus USING(genus_id)
JOIN family USING(family_id)
WHERE family_name = 'Синицевые'
ORDER BY observer_name;
```
Здесь из всех таблиц собирается одна большая; представьте, что к таблице `observations` прикручивается имя вида птиц из таблицы `species` , а потом к этому комбо добавляется имя рода, к которому принадлежат птички, из таблицы `genus` , а потом имя семейства из таблицы `family` . Для того, чтобы искомый алгоритм был найден нам нужно просто добавить условие; это условие будет отсеивать из получившейся таблицы все строки, где в имени семейства `family_name` указано что-то кроме ‘Синицевые’. Вся задача решается с помощью INNER JOIN и объединения через индексы.

## Задача 2
Давай достанем семейства птиц и виды в этих семействах. Виды разделяем запятой и пробелом. Например, что-то такое может быть на выходе (данные могут быть другим

| family_name  | species_names                                        |
| ------------ | ---------------------------------------------------- |
| Дроздовые    | Дрозд белобровик, Дрозд болотный, Дрозд красногрудый |
| Ещё какое-то |                                                      |
| Простые      |                                                      |
| Синицевые    | Большая синица, Гаичка болотная, Синица зелёная      |
Обрати внимание на то, что если в семействе нет видов, то в колонке `species_names` выводится пустая строка. Не запятая и пробел и не `null`.
Сортировка строк по названию семейства. Внутри списка видов (то есть внутри `species_name`) сортировка по названию вида.
Самая большая проблема этой задачи - сортировка значений внутри списков species_names. То есть без этого нюанса можно было бы решить эту задачу крайне просто:
```SQL
SELECT
	family.family_name,
	CASE
		WHEN count(species.species_name) > 0 THEN string_agg(species.species_name, ', ')
	ELSE ''
	END species_names
FROM family
LEFT OUTER JOIN genus UNSING(genus_id)
LEFT OUTER JOIN species USING(species_id)
GROUP BY family.family_name
ORDER BY family.family_name;
```
Но так ничего не выйдет: значения `species.species_name` не будут отсортированы, а если добавить это наименование в `GROUP BY`, то вылетит ошибка - так делать нельзя.
Во-первых, `CASE` можно сократить при помощи функции `coalesce()`.
```SQL
SELECT 
	family.family_name,
	coalesce(
		string_agg(species.species_name, ', '),
		''
	) AS species_name
...
```
`coalesce()` будет менять все `NULL` на пустую строку.
Во-вторых, у функции `string_agg()` есть опция - указать сортировку.
```SQL
SELECT
	family.family_name,
	coalesce(
		string_agg(species.species_name, ', ' ORDER BY species_name),
		''
	) AS species_name
FROM family
LEFT OUTER JOIN genus USING(genus_id)
LEFT OUTER JOIN species USING(species_id)
GROUP BY family.family_name
ORDER BY family.family_name;
```
