# обновить статистику планировщика
Статистика планировщика влияет на то как планировщик делает план для запросов. Поэтому иногда ее полезно обновлять. Как это можно сделать?
```SQL
analyze
```
Если ввести такую команду просто так, то она не выполнится "идеально", потому что для обновления некоторых встроенных в PostgreSQL отношений не хватит прав, но для таблиц в моей собственной БД хватит.
# оптимизировать пространство БД
Оптимизировать пространство БД означает, что весь шлак, который PostgreSQL собирает в процессе работы, будет помечен как "шлак" и убран из БД. Такую работу делает команда:
```SQL
vacuum
```
У нее есть несколько вариаций:
```SQL
vacuum --очищает пространство БД от устаревших данных.
vacuum full --очищает пространство + оптимизирует место на диске + блочит таблицу
vacuum analyze --очищает пространство БД + обновляет статистику планировщика.
```
# bloat
Термин **bloat** в контексте postgresql описывает ситуацию, когда физический размер БД больше, чем размер данных, которые в ней находятся.
Такое происходит из-за механизма postgresql, который при удалении данных из БД с помощью `DELETE` или `UPDATE` на самом деле сохраняет их в качестве "мертвого слоя". Мертвые слои занимают все то же место на диске. Чтобы их убрать используют команду `vacuum` - она помечает мертвые слои метками, чтобы последующие команды типа `INSERT` и `UPDATE` могли записать на место мертвого слоя новые данные. Но если команда `vacuum` не будет использована, то мертвые слои так и останутся висеть в памяти, что приведет к замедлению БД.
Подробнее: https://pganalyze.com/docs/vacuum-advisor/what-is-bloat
# pg_repack
Штука, которая позволяет реорганизовать пространство, которое занимает БД. Она выполняет функционал `vacuum full`, но без блокировки БД.
# кастомная статистика
Иногда бывает так, что планировщик может **сильно** недооценить количество строк, которые вернет запрос из БД. В таких случаях лучше всего использовать кастомную статистику: https://postgrespro.ru/docs/enterprise/17/sql-createstatistics. Это статистика, которую можно создать вручную, например:
```SQL
create statistics <название_статистики> on column_1, column_2 from test_table;
analyze test_table;  -- обязательно обновляем по таблице статистику, чтобы кастомная статистика "зарегистрировалась".
```
Такая штука позволяет гибко настраивать те данные, исходя из которых оптимизатор делает план запроса.
