Во всех СУБД существует механизм автоматических блокировок. Например, такой встроен в операцию `vacuum full`. Суть блокировки в том, что запросы ко всей БД или к ее части становятся невыполнимыми, пока происходит какая-то другая операция. Эта операция "блокирует" таблицу.
В PostgreSQL помимо автоматических блокировко есть еще возможность выставлять пользовательские блокировки. Зачастую они нужны, чтобы запретить другим операциям работать с конкретной сущностью на время нашей работы с ней, например, через транзакцию.
Самих принципов пользовательских блокировок огромное количество, поэтому я знаю только одну основную. 
# `select for update`
Блокировка позволяет получить блокировку на изменение в одной строке или в ряде строчек. Блокировка автоматически снимается, когда транзакция делает коммит или откатывается.
Самый классический пример - бронирование комнат. Сначала я создам таблицу бронирований и залью в нее данные:
```SQL
lessons=> create table booking(
lessons(> booking_date date not null,
lessons(> room_id bigint not null,
lessons(> booked_by bigint,
lessons(> primary key (booking_date, room_id));
CREATE TABLE
lessons=> insert into booking(booking_date, room_id, booked_by) values
lessons-> (current_date, 1, 777), (current_date, 2, null);
INSERT 0 2
```
В таблице `booking` есть три поля `booking_date, room_id, booked_by` и составной первичный ключ на первые два поля. `booked_by` - это по идее должен быть внешний ключ на таблицу пользователей. 
Чтобы показать проблему я начну две транзакции. Обе сначала поищут в таблице те комнаты, которые никем не забронированы:
```SQL
lessons=*> select * from booking where booking_date = current_date and booked_by is null;
 booking_date | room_id | booked_by
--------------+---------+-----------
 2025-07-06   |       2 |
(1 row)
```
И обе увидят, что одна комната свободна. Транзакция_1 начнет обновлять таблицу - попытается забить комнату для пользователя с id 888:
```SQL
lessons=*> update booking set booked_by=888 where room_id = 2 and booking_date = current_date;
UPDATE 1
```
Теперь транзакция_2 хочет обновить таблицу, но уже забить комнату 2 для пользователя 999:
```SQL
lessons=*> update booking set booked_by=999 where room_id = 2 and booking_date = current_date;
```
Здесь обновление не случиться сразу - транзакция_2 будет ждать пока завершит работу транзакция_1.
Проблема в том, что когда транзакция_1 сделает коммит, то транзакция_2 сможет без ошибок внести изменения в ту же строку и сделать свой коммит. Следовательно, в таблице останутся данные от транзакции_2.
## решение через `repeatable read`
Эту задачу можно решить с помощью  уровня изоляции `repeatable read`. Суть решения будет в том, что этот уровень изоляции уронит ошибку, когда мы попытаемся совершить конкурентное обновление одних и тех же данных в таблице:
```SQL
--транзакция_1
lessons=*> update booking set booked_by=888 where room_id = 2 and booking_date = current_date;
UPDATE 1
--транзакция_2
lessons=*>  update booking set booked_by=999 where room_id = 2 and booking_date = current_date;
```
Транзакция_2 все так же ждет завершения первой транзакции. Если транзакция_1 закоммитит свои изменения, то транзакция_2 сразу вернет ошибку:
```SQL
ERROR:  could not serialize access due to concurrent update
```
И перезаписать данные уже не получится.
## решение через блокировку
Чтобы выполнить блокировку не нужно делать сальто через голову. Достаточно внутри запроса `select` прописать дополнительную строку - `for update`:
```SQL
lessons=*> select * from booking where booking_date = current_date and booked_by is null for update;
 booking_date | room_id | booked_by
--------------+---------+-----------
 2025-07-06   |       2 |
(1 row)
```
Этот запрос я написал в транзакции_1 с уровнем изоляции `read commited`. `for update` блокирует выбранные данные. Это означает, что все остальные транзакции, прежде чем сделать обновления будут ждать, пока эта блокировка кончится. 
Здесь важно помнить, что после того как заблокированные данные были закоммичены их все равно можно поменять. Например:
```SQL
-- транзакция_1
lessons=*> select * from booking where booking_date = current_date and booked_by is null for update;
 booking_date | room_id | booked_by
--------------+---------+-----------
 2025-07-06   |       2 |
(1 row)
lessons=*> update booking set booked_by=888 where room_id = 2 and booking_date = current_date;
UPDATE 1
-- транзакция_2
lessons=*>  update booking set booked_by=999 where room_id = 2 and booking_date = current_date;
```
Транзакция_2 точно так же как и раньше будет ждать пока данные обновятся, но после их обновления все равно даст возможность данные обновить повторно.
У `select for update` не точно такой же способ решения проблемы, как у `repeatable read`. `select for update` запрещает читать данные таким же `select for update`, пока они не были изменены в той транзакции, которая их блокирует.
## преимущества `select for update`
Исходя тупо из того, что мы запускаем блокировку именно в режиме `read committed`, можно сделать вывод, что наши операции над строками будут проходить быстрее, чем они проходят в `repeatable read` и `serializable`.
#  `skip locked`
Это инструкция, которая позволяет пропустить строки таблицы, которые были заблокированы другими блокировками. Например, я хочу заблокировать для изменений только один номер в моем отеле и дать возможность изменять все остальные:
```SQL
lessons=*> select * from booking where booking_date = current_date and booked_by is null limit 1 for update skip locked;
 booking_date | room_id | booked_by
--------------+---------+-----------
 2025-07-06   |       2 |
(1 row)
```
Здесь я добавил:
1) `limit 1` - ограничиваем выборку до одной строки.
2) `skip locked` - пропуск всех уже заблокированных для изменений строк.
Теперь я могу пойти во вторую транзакцию и ввести ту же строку, чтобы она мне показала поле, которое я могу поменять и которое точно не заблокировано.
