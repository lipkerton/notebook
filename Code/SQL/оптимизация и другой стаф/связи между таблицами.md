С одной стороны у нас есть JOIN, который позволяет смешивать таблички, но с другой нужно понимать, что само это смешивание может происходить по разному. 
## Один ко многим
Например, если у нас есть таблица категорий фильмов и таблица самих фильмов, то понятно, что одна категория будет проставлена для многих фильмов и, наоборот, если мы посмотрим на эту ситуацию с точки зрения таблицы фильмов - многие фильмы относятся к одной категории.
Это базовый Паттерн связи между таблицами - он называется **один ко многим**.
Как это делается?
```SQL
CREATE TABLE author (
	author_id bigint generated always as identity primary key,
	name varchar(150) not null check(length(name) >= 2),
);
CREATE TABLE book (
	book_id bigint generated always as identity primary key,
	name varchar(150) not null check(length(name) >= 2),
	author_id bigint not null references author(author_id)
)
```
Таблица `author` и таблица `book` относятся друг к другу по типу связи "один ко многим". Это означает, что несколько записей в таблице `book` ссылаются на одного автора в таблице `author`. Например, Пушкин написал сразу несколько произведений.
## Один к одному
Один к одному - это когда у нас одна запись из одной таблицы ссылается на конкретную запись из другой таблицы. Кто-то спросит: а зачем это нужно, если можно просто сшить две эти таблицы в одну таблицу? Ответ: это, нужно, когда у нас есть основные сущности в основной таблице и дополнительные сущности в другой. Одна таблица делится на несколько по принципу частоты использования данных. Зачем нагружать одну табличку, если можно положить наиболее часто используемые данные в одну таблицу, а дополнительные сведения в другую?
Принцип создания - первичный ключ одной таблички является одновременно первичным ключом в другой табличке.
```SQL
CREATE TABLE book (
	book_id bigint generated always as identity primary key,
	name varchar(150) not null check(length(name) >= 2),
)
CREATE TABLE book_detail (
	book_detail_id bigint primary key references book(book_id),
	description text
)
```
## Многие ко многим
У нас может быть несколько авторов для одной книги и несколько книг для одного автора, а еще может быть так, что несколько авторов написали несколько книг. Такая связь называется "многие ко многим" и делается через отдельную табличку. В этой отдельной табличке есть всего две колонки - одна ссылается на одну табличку, а другая на другую. Отдельная табличка хранит пары значений первичных ключей из двух табличек. Так создается связь многие ко многим.
```SQL
CREATE TABLE author (
	author_id bigint generated always as identity primary key,
	name varchar(150) not null check(length(name) >= 2),
);
CREATE TABLE book (
	book_id bigint generated always as identity primary key,
	name varchar(150) not null check(length(name) >= 2),
);
CREATE TABLE book_to_author (
	author_id bigint references author(author_id),
	book_id bigint references book(book_id),
	primary key(author_id, book_id)
);
```
Здесь парный первичный ключ - да, так делать можно. Первичным ключом этой таблички будет пара из двух значений.
