## база
MVCC (Multiversion Concurrency Control) - это механизм, который позволяет транзакциям видеть разные состояния одних и тех же строк.
Я хочу в одной транзакции поменять значения в каких-то строках таблицы. Допустим, что я это сделал: будет ли видеть эти изменения до коммита другая транзакция, которая пытается получить те же данные с помощью `select`?
Чтобы это проверить я создам таблицу `account`:
```SQL
create table account (
	account_id bigint generated always as identity primary key,
	balance int check(belance between 0 and 15000000)
);
```
И вставлю в нее значения:
```SQL
insert into account(balance) values (1000), (2000);
```
Теперь я начну одну транзакцию и в ней поменяю значение для одного из полей в таблице `account` + покажу содержимое таблицы с помощью `select`:
```SQL
begin;
update account set balance=balance+1 where account_id = 1;
select * from account;
```
Вывод будет таким:
```SQL
 account_id | balance
------------+---------
          2 |    2000
          1 |    1001
(2 rows)
```
Теперь я одновременно открою другую транзакцию в соседнем окне и попрошу через `select` показать содержимое `account`:
```SQL
begin;
select * from account;
```
Вывод будет таким:
```SQL
 account_id | balance
------------+---------
          1 |    1000
          2 |    2000
(2 rows)
```
Возникает вопрос: **каким образом две разные транзакции могут видеть одни и те же данные в двух разных состояниях?**
Весь секрет в дополнительных полях, которые существуют в таблицах PostgreSQL, - `xmin` и `xmax`. Первое поле содержит номер транзакции, которая создала текущую версию данных в строке, а второе поле содержит номер той транзакции, которая в данный момент вносит изменения в состояние данных строки. Посмотреть содержимое этих полей можно, если мы достанем их из таблицы стандартным `select` запросом.
Я это сделаю в первой начатой транзакции:
```SQL
select account_id, balance, xmin, xmax from account;
```
Вывод:
```SQL
 account_id | balance | xmin | xmax
------------+---------+------+------
          2 |    2000 | 1285 |    0
          1 |    1001 | 1289 |    0
(2 rows)
```
Из примера видно, что в данные с `account_id = 1` внесла изменения транзакция под номером 1289. Теперь я хочу посмотреть номер текущей транзакции - это можно сделать с помощью функции `txid_current()`:
```SQL
select txid_current();
```
Вывод:
```SQL
 txid_current
--------------
         1289
(1 row)
```
В рамках этой транзакции я выполнял изменения в таблице `account`, поэтому ее номер сохранился в качестве номера транзакции, которая создала текущую версию данных в таблице.
Теперь я хочу посмотреть как это выглядит с точки зрения другой транзакции - я введу команду, чтобы глянуть `xmax` и `xmin` той же таблицы `account`. Вывод будет таким:
```SQL
 account_id | balance | xmin | xmax
------------+---------+------+------
          2 |    2000 | 1285 |    0
          1 |    1000 | 1288 | 1289
(2 rows)
```
Номер 1289 в другой транзакции перекочевал в колонку `xmax` - это означает, что эта другая транзакция будет видеть прежнее состояние данных, но так же она видит и то, что данные в этот момент меняются в транзакции под номером 1289.
Если я закоммичу изменения в первой транзакции, то номер 1289 перейдет в колонку `xmin` - теперь это будет основная версия данных для всех транзакций:
```SQL
commit;
select account_id, balance, xmin, xmax from account;
```
Вывод:
```SQL
 account_id | balance | xmin | xmax
------------+---------+------+------
          2 |    2000 | 1285 |    0
          1 |    1001 | 1289 |    0
(2 rows)
```
Окей, это я понял, а что происходит с неактуальными данными? Стираются ли они тут же из памяти? Нет, PostgreSQL хранит неактуальные данные еще какое-то время.
## pg_dirtyread
Чтобы их глянуть нужно установить расширение для PostgreSQL, которое называется `pg_dirtyread`. Оно выложено на гитхаб, поэтому сначала я его скачаю:
```Bash
git clone https://github.com/df7cb/pg_dirtyread.git
```
Затем использую make-файл (внутри склонированной папки), чтобы расширение раскидало все нужные файлы по моему экземпляру PostgreSQL:
```Bash
make PG_CONFIG=/var/lib/postgresql/16/bin/pg_config
make install PG_CONFIG=/var/lib/postgresql/16/bin/pg_config
```
Параметр `PG_CONFIG` указывает на конфигурационный файл моей версии PostgreSQL. Лучше его всегда указывать, потому что иначе расширение может установится для другой версии кластера. Мой кластер собран из исходников, поэтому без указания кофигурационного файла команда `make` для `pg_dirtyread` создает инфраструктуру расширения для несуществующей на моем компе версии кластера.
Теперь можно идти в `psql`, чтобы установить от лица `postgres` новое расширение:
```SQL
create extension pg_dirtyread;
```
