## база
Каждая транзакция имеет свой уровень изоляции. В PostgreSQL существует четыре уровня изоляции транзакции (от самого мягкого до самого жесткого):
1) `read uncommitted`
2) `read committed`
3) `repeatable read`
4) `serializable`
При создании транзакции ей можно указать нужный уровень изоляции. Если уровень не указывать, то по умолчанию будет использован уровень `read committed`.
Признак разделения уровней транзакций - по количеству аномалий.
Аномалия - это неожиданный результат, который возникает при попытке выполнить несколько одновременных запросов в рамках одной БД.
Один из примеров аномалии - "грязное чтение". В этом случае транзакция_1 видит изменения в таблице, которые внесла транзакция_2, при том, что транзакция_2 еще не применила операцию `commit`, чтобы зафиксировать это изменение. Эта аномалия проявляется на уровне `read uncommitted`.
## read (un)committed
В PostgreSQL нет уровня изоляции `read uncommitted`. Этот уровень формально означает, что мы можем видеть еще незакоммиченные изменения внутри других транзакций.
На практике - это означало бы, что внесение изменений в БД транзакцией_1 может быть увидено транзакцией_2, которая так же может внести изменения, на основании еще незакоммиченных изменений транзакции_1. Если в транзакции_1 мы удалили с баланса 900 рублей (1000 - 900 = 100), а затем в транзакции_2 удалили еще 100 рублей, то, очевидно по итогу второй транзакции баланс будет нулевым в этом поле; но что если первая транзакция вдруг откатилась, а вторая закоммитилась уже после совершения отката? Мы ожидаем увидеть на балансе 900, а будет 0:
```SQL
-- транзакция_1
lessons=> begin;
BEGIN
lessons=*> update account set balance=balance-900 where account
_id = 1;
UPDATE 1
-- транзакция_2
lessons=> begin;
BEGIN
lessons=*> update account set balance=balance-100 where account
_id = 1;
UPDATE 1
-- транзакция_1
lessons=> rollback;
```
Опять же в PostgreSQL такой фокус не сработает - все транзакции, если видят, что данные меняются какой-то другой транзакцией, будут терпеливо ждать пока меняющая транзакция закоммитит изменения или откатится.
При большом желании можно прописать команде `begin` указание на уровень изоляции транзакции `read committed`:
```SQL
begin transaction isolation level read committed;
```
Но смысла в этом немного, т.к. все транзакции изначально обладают этим уровнем изоляции.
Так же можно все таки начать транзакцию в режиме `read uncommitted`:
```SQL
begin transaction isolation level read uncommited;
```
Но и в этом большого смысла нет, т.к. уровни `read committed` и `read uncommitted` никак не отличаются в PostgreSQL - грязного чтения в этой СУБД не существует.
## repeatable read
Как понятно "грязное чтение" - это не единственная существующая аномалия. В процессе выполнения транзакции уровня `read committed` может возникнуть еще пара аномалий:
1) неповторяемое чтение
2) фантомное чтение
Неповторяемое чтение в контексте транзакций означает, что при выполнении внутри транзакции одного и того же запроса нам по какой-то причине могут приходить разные результаты. Например, я в транзакции_1 читаю базу, параллельно транзакция_2 вносит в БД изменения, поэтому при следующем чтении транзакцией_1 БД мы получим измененный транзакцией_2 результат. Сначала покажу состояние до изменений:
```SQL
lessons=> begin;
BEGIN
lessons=*> table account;
 account_id | balance
------------+---------
          2 |    2000
          1 |    1001
(2 rows)
```
Вношу изменения другой транзакцией:
```SQL
lessons=> update account set balance=balance+1 where account_id = 1;
UPDATE 1
```
Проверяю результат в первой:
```SQL
lessons=*> table account;
 account_id | balance
------------+---------
          2 |    2000
          1 |    1002
(2 rows)
```
Как видно из примера: транзакция_1 в процессе выполнения видит, что данные меняются, т.е. запросы транзакции_1 могут выдавать уже измененные данные в разные моменты времени.
Это и называется "неповторяемое чтение".
Чтобы избежать такой проблемы существует новый уровень изоляции транзакции. Он называется `repeatable read`. Я открою такую транзакцию и повторю предыдущие операции:
```SQL
lessons=> begin transaction isolation level repeatable read;
BEGIN
lessons=*> table account;
 account_id | balance
------------+---------
          2 |    2000
          1 |    1002
(2 rows)
```
Обновим таблицу в новой транзакции:
```SQL
lessons=> update account set balance=balance+1 where account_id = 1;
UPDATE 1
```
И посмотрим снова на ту же таблицу в первой транзакции:
```SQL
lessons=*> table account;
 account_id | balance
------------+---------
          2 |    2000
          1 |    1002
(2 rows)
```
Ничего не поменялось.
Что такое аномалия фантомного чтения? Фантомное чтение в общем случае - это ситуация, когда транзакция читает больше записей в таблице, чем она могла прочитать при том же запросе ранее, или, альтернативная ситуация, - когда транзакция читает меньше записей в таблице.
Например, я начал транзакцию_1, посчитал все записи в текущей таблице, а затем добавил в рамках транзакции_2 еще одну запись в ту же таблицу. Что будет с подсчетом всех записей в транзакции_1? Он увидит на одну запись больше. Такая же история происходит и с удалением записи.
Сначала открою первую транзакцию и проверю количество строк:
```SQL
lessons=> begin;
BEGIN
lessons=*> select count(*) from account;
 count
-------
     2
(1 row)
```
Теперь обновлю таблицу другой транзакцией:
```SQL
lessons=> insert into account(balance) values (3000);
INSERT 0 1
```
И снова посмотрю количество записей в первой таблице:
```SQL
lessons=*> select count(*) from account;
 count
-------
     3
(1 row)
```
Это и есть фантомное чтение.
`repeatable read` позволяет исключить ситуацию фантомного чтения:
```SQL
lessons=> begin transaction isolation level repeatable read;
BEGIN
lessons=*> select count(*) from account;
 count
-------
     3
(1 row)
```
Теперь снова добавим в таблицу запись:
```SQL
lessons=> insert into account(balance) values (7000);
INSERT 0 1
```
И снова посмотрим количество записей в первой транзакции:
```SQL
lessons=*> select count(*) from account;
 count
-------
     3
(1 row)
```
Количество записей не изменилось в транзакции с уровнем изоляции `repeatable read`.
### проблемы высокого уровня
На этом этапе у меня уже появился вопрос: "А почему бы не использовать более высокий уровень транзакций всегда, если он более аккуратно работает с данными?"
Во-первых, так делать не стоит, потому что более аккуратный подход к работе с данными требует больших ресурсов PostgreSQL. Ресурсы в основном тратятся на контроль за такой безопасностью. Например, нужно держать в памяти все необходимые для работы транзакции данные.
Во-вторых на высоких уровнях защиты транзакции возникают ошибки сериализации. В общем случае эти ошибки возникают, когда мы открываем транзакцию_1, например, `repeatable read`, обновляем в ней данные, а затем пытаемся обновить те же данные в параллельной транзакции_2. Вторая транзакция упадет с ошибкой, потому что `repeatable read` увидит измененные дважды данные и не даст сделать коммит.
Сначала я открою первую транзакцию в `repeatable read` и обновлю в ней данные:
```SQL
lessons=> begin transaction isolation level repeatable read;
BEGIN
lessons=*> update account set balance = balance + 1
lessons-*> where account_id = 3;
UPDATE 1
```
Теперь открою вторую транзакцию на том же уровне изоляции и попытаюсь обновить те же данные:
```SQL
lessons=> begin transaction isolation level repeatable read;
BEGIN
lessons=*> update account set balance = balance + 1
lessons-*> where account_id = 3;
```
Вторая транзакция повиснет - ожидает коммита первой транзакции. Я так и сделаю:
```SQL
lessons=*> commit;
COMMIT
```
В тот же момент вторая (повисшая) транзакция выдаст:
```SQL
lessons=*> update account set balance = balance + 1
lessons-*> where account_id = 3;
ERROR:  could not serialize access due to concurrent update
```
