# другой сервер
Здесь я решал задачу: *как я могу разделять трафик, который нужно использовать без прокси, и трафик для которого нужен прокси?*
Чтобы решить эту задачу нужно разобраться с маршрутизацией и такой гайд так же [существует](https://xtls.github.io/ru/document/level-1/routing-lv1-part1.html#%D0%BA%D1%80%D0%B0%D1%82%D0%BA%D0%B8%D0%B8-%D0%BE%D0%B1%D0%B7%D0%BE%D1%80-%D1%84%D1%83%D0%BD%D0%BA%D1%86%D0%B8%D0%B8-%D0%BC%D0%B0%D1%80%D1%88%D1%80%D1%83%D1%82%D0%B8%D0%B7%D0%B0%D1%86%D0%B8%D0%B8-routing-%D1%87%D0%B0%D1%81%D1%82%D1%8C-1), но действия в нем становятся понятнее, только когда архитектура имеющихся у меня серверов становится сложнее, что я не понял сразу.
xray предлагает сделать выделенный сервер с запущенным на нем демоном, который будет обрабатывать входящие запросы и отправлять их далее, проведя через обработку. Для этой задачи у меня уже есть сервер в каких-нибудь Нидерландах. Он может забрать трафик пользователя и выдать за трафик пользователя из Амстердама, однако остается проблема: что если мне не нужно, чтобы запросы, которые я посылаю к доступным в моем регионе доменам, обрабатывались через прокси?
Изначально я подумал, что эту проблему можно решить просто игнором. Например, получив трафик от пользователя на ресурс, для которого прокси не нужен, я могу просто проигнорировать этот трафик, т.е. заставить пользователя не использовать для него прокси. Однако такой план является слишком наивным - если трафик уже находится внутри xray, то я не могу его вернуть назад с помощью маршрутизации.
Здесь я дополнительно запутался в понятиях.

---
Первое, что следует различать - действия, которые могут быть произведены xray с трафиком. Первое о чем говорят в этом контексте разработчики - трио `inbound, routing, outbound`:
1) `inbound` - входящие запросы.
2) `routing` - маршрутизация.
3) `outbound` - исходящий трафик.
В соответствующих полях конфига можно прописать все нужные настройки.

---
Когда я настраиваю маршрутизацию, то могу указать настройки для определения куда следует отправить трафик. Это выглядит так:
```JSON
"routing": {
"domainStrategy": "IPIfNonMatch",
"rules": [
  {
	"type": "field",
	"ip": [
	  "geoip:!ru"
	],
	"outboundTag": "direct"
  }
}
```
Согласно этому правилу весь трафик, который попадет под определение, уйдет на `"outboundTag": "direct"`. Это означает, что маршрутизация спокойно и свободно выпустит его с сервера, а т.к. мой сервер в Нидерладах, то и впустится трафик как нидерладский. 
Еще есть `"outboundTag": "block"` - он просто блокирует трафик и не дает ему уйти из xray.
Нужно понимать, что все эти теги не являются описанием поведения - это просто названия для реальных описаний, которые находятся в `outbounds`, например, `"outboundTag": "direct"` в `outbounds` выглядит так:
```JSON
"outbounds": [
	{
	  "tag": "direct",
	  "protocol": "freedom"
	}
]
```
То есть, когда мы в маршрутизации какой-то трафик помечаем этим тегом, на самом этот трафик будет обработан протоколом `freedom`. 
А для `"outboundTag": "block"`:
```JSON
"outbounds": [
	{
	  "tag": "block",
	  "protocol": "blackhole"
	}
]
```
Существует протокол `blackhole`, который просто поглощает весь трафик в себя.

---
Здесь в голову должна была прийти мысль: что если я могу настроить свой собственный путь для выхода трафика, а не выбирать между `direct` и `block`?
Как раз такую задачу разработчики решают [тут](https://xtls.github.io/ru/document/level-1/routing-lv1-part1.html#_2-2-%D0%B8%D1%81%D1%85%D0%BE%D0%B4%D1%8F%D1%89%D0%B8%D0%B8-%D1%82%D1%80%D0%B0%D1%84%D0%B8%D0%BA). Смысл такой, что трафик посылается теперь не только напрямую с сервера и в блок, а еще в тег `proxy-out-vless`. Однако непонятно что именно представляет из себя этот тег - куда попадают данные, которые им обозначены?
А они попадают на другой сервер)
Если я хочу решить свою задачу (перенаправить весь российский трафик на выход не через прокси), то мне нужно сделать сервер VLESS в России и отправить трафик туда.
Я сделал такой сервер, описал там все, что нужно и обозначил, что я хочу, чтобы все, что попало сюда выходило либо через `direct`, либо через `block`. Это правильно, т.к. я хочу сделать так, чтобы на этот сервер попадал ТОЛЬКО русский трафик.
Разработчики связывают свой второй сервер с текущим таким образом:
```JSON
{
  "tag": "proxy-out-vless",
  "protocol": "vless",
  "settings": {
	"vnext": [
	  {
		"address": "a-name.yourdomain.com",
		"port": 443,
		"users": [
		  {
			"id": "uuiduuid-uuid-uuid-uuid-uuiduuiduuid",
			"flow": "xtls-rprx-vision",
			"encryption": "none",
			"level": 0
		  }
		]
	  }
	]
  },
  "streamSettings": {
	"network": "tcp",
	"security": "tls",
	"tlsSettings": {
	  "serverName": "a-name.yourdomain.com",
	  "allowInsecure": false,
	  "fingerprint": "chrome"
	}
  }
}
```
И я только позже понял, что это буквально клиентские настройки для использования прокси. Другими словами здесь мы выступаем в роли клиента для прокси, который настроен в России. Сюда мы вписываем данные того сервера, а для единственного юзера заполняем `uuid`, который на том сервере мы напишем уже в параметрах клиента.
