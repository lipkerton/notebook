# образ
Сначала следует скачать `iso` образ с одного из зеркал.
Список зеркал находится здесь: [Arch HTTP(S) mirrors](https://archlinux.org/download/#http-downloads)
## верификация образа
Далее **рекомендовано** верифицировать скачанный с зеркала образ.
Это делается потому, что в теории зеркало может быть взломано (или сделано) злоумышленником, который изменит находящиеся на нем образы.
Верификация проводится с помощью пары ключей:
1) Секретный ключ хранит разработчик Pierre.
2) Публичный ключ, который тоже хранит Pierre, но уже публичный.
В списке, который доступен по ссылке есть несколько способов проверить валидность файла `iso` - я пользуюсь именно ключами PGP.
- Качаем подпись с официального сайта: https://archlinux.org/iso/2025.09.01/archlinux-2025.09.01-x86_64.iso.sig
- Качаем файл `iso` с одного из зеркал.
- Размещаем оба файла в одной директории для удобства.
Теперь нужно попросить у Pierre публичный ключ. Его ключ находится в супер-защищенной библиотеке ключей WKD. И попросить его можно с помощью команды:
```Bash
gpg --auto-key-locate clear,wkd -v --locate-external-key pierre@archlinux.org
```
Теперь можно верифицировать `iso` файл подписью:
```Bash
gpg --verify archlinux-2025.09.01-x86_64.iso.sig archlinux-2025.09.01-x86_64.iso
```
На выходе должна получится примерно такая запись:
```Bash
gpg: Подпись сделана понедельник,  1 сентября 2025 г. 19:39:15 MSK
gpg:                ключом EDDSA с идентификатором 3E80CA1A8B89F69CBA57D98A76A5EF9054449A5C
gpg:                 издатель "pierre@archlinux.org"
gpg: Действительная подпись пользователя "Pierre Schmitz <pierre@archlinux.org>" [неизвестно]
gpg: WARNING: The key\'s User ID is not certified with a trusted signature!
gpg:           Нет указаний на то, что подпись принадлежит владельцу.
Отпечаток первичного ключа: 3E80 CA1A 8B89 F69C BA57  D98A 76A5 EF90 5444 9A5C
```
Здесь `WARNING` просто предупреждает что `Pierre Schmitz <pierre@archlinux.org>` - это пользователь, с которым вы не знакомы, поэтому лучше было бы знать чей ключ мы используем. Но сейчас я точно знаю, что это наш Pierre, так что волноваться не о чем.
## механика верификации
Пьер на своем компьютере делает из файла `.iso` хэш с помощью одного из открытых алгоритмов. Затем он шифрует хэш с помощью своего секретного ключа и получается файл `.sig`.
Файл `.sig` это и есть "подпись". Она содержит:
1) Зашифрованный с помощью секретного ключа хэш.
2) Метаданные этого хэша. Например, алгоритм хэширования и проч.
Я на своем компьютере должен захэшировать свой `.iso` с помощью метаданных из `.sig`. Затем расшифровать хэш, который был зашифрован секретным ключом Пьера в `.sig`, с помощью публичного ключа (который я тоже вежливо попрошу у Пьера). И вот теперь я могу сравнить хэш Пьера и хэш моего `.iso` файла.
Если они совпадут - операция прошла успешно.
Команда:
```Bash
gpg --auto-key-locate clear,wkd -v --locate-external-key pierre@archlinux.org
```
Просит у Pierre официальный его личный публичный ключ. Этот ключ кладется утилитой `gpg` в специальную директорию на компьютере.
Команда:
```Bash
gpg --verify archlinux-2025.09.01-x86_64.iso.sig archlinux-2025.09.01-x86_64.iso
```
Делает следующее:
1) Из файла `.sig` извлекается алгоритм хэширования, который нужно применить к файлу `.iso` перед тем как сравнить хэши `.iso` и `.sig`.
2) Файл `.iso` хэшируется по алгоритму.
3) Из файла `.sig` извлекается хэш (сделанный Пьером для оригинального `.iso`), который был зашифрован с помощью секретного ключа.
4) Хэш расшифровывается с помощью публичного ключа.
5) Сравнивается хэш Пьера и хэш моего `.iso`. При совпадении хэшей операция верификации пройдет успешно.
## немного о ключах
Связка "публичный-секретный" ключ - это чудо криптографии. 
Если описывать их принцип работы просто: автор делает связку из двух ключей, секретный ключ оставляет у себя, а публичный может отдать тому, кто хочет им верифицировать автора. Продукт, который автор шифрует секретным ключом, может быть расшифрован публичным ключом, НО при этом нет никакой возможности с помощью публичного ключа восстановить секретный ключ.
Секретный ключ запирает данные, а публичный ключ может открыть данные секретного ключа. Так верифицируется продукт.
# импорт образа в UTM
Я хочу установить Arch внутрь виртуальной машины на своем Mac. Чего я не учел - архитектура процессора. У моего Mac архитектура ARM, а все образы `.iso`, которые находятся на зеркалах рассчитаны под архитектуру x86 (x64).
Про разницу архитектур лучше всего прочитать статью: https://habr.com/ru/companies/droider/articles/519732/
Грубо говоря, ARM и x86 используют разные наборы инструкций - RISC и CISC. Поэтому невозможно сделать виртуальную машину на базе процессора ARM с помощью образа `.iso`, которому нужна архитектура x86.
Как с этим жить? Просто нужно вместо виртуализации в UTM выбрать эмуляцию. UTM будет эмулировать систему с другой архитектурой и в рамках этой эмуляции я смогу запустить Arch `.iso`.
Кстати, если архитектура для образа не подходит, но образ все равно был загружен, то UTM выкинет консоль `shell` с предложением ввести команды [подробнее](https://github.com/utmapp/UTM/discussions/3470).
Дополнительно будет не лишним уметь смотреть параметры своего мака. Это делается через команду `system-profiler`, но если ввести ее просто так, то на экран выльется бесконечный набор информации. Лучше всего сначала посмотреть список доступных параметров:
```Bash
system-profiler -l
```
Выбрать нужный и использовать в качестве аргумента:
```bash
system-profiler SPHardwareDataType
```
Внутри консоли UTM все просто - достаточно закинуть образ `.iso` и вот уже появляется приветственное окно Arch и первая виртуальная консоль от лица пользователя `root` в оболочке `zsh`.

Допустим я хочу установить систему. На моем компьютере есть уже один диск - этот диск можно увидеть в `/dev/` (от "devices" - там отображаются все подключенные к системе устройства) или с помощью утилиты `fdisk -l`. Мой стартовый вывод `fdisk -l`:
![[output_fdisk_-l.jpg]]
Здесь есть два диска:
1) `/dev/sda` - это как раз тот диск, который мне нужен.
2) `/def/loop0` - это такой хитрый механизм, который на самом деле не является диском. На самом деле - это обычный файл (а не физическое устройство, как в случае с реальным диском). Почему тогда он в списке дисков? Потому что этому файлу нужно подключиться к текущей системе в виде диска, чтобы выполнить свою работу. Другими словами: файл притворяется диском, чтобы ядро Linux могло правильным образом его обработать.
Диск  `/dev/sda` мне нужно разметить. Это означает, что его нужно поделить на разделы, каждый из которых будет использован при установке для выполнения определенных функций.
В инструкции к Arch предлагают разметить так:
- `efi` - раздел загрузчика системы. Этот раздел нужен, чтобы при запуске ОС UEFI прошивка могла в нем найти те файлы, которые ей нужны, чтобы правильно инициализировать систему.
- `swap` - раздел подкачки. Эта штука понадобится когда у нас кончится RAM для запуска операций на компьютере - ОС сможет скинуть часть операций в раздел подкачки, чтобы не закрывать их принудительно.
- `/` - корневой раздел системы. То место, с которым буду работать я, как пользователь.
Главный вопрос: **как создать эти разделы?**


## форматирование
В инструкции написано "Отформатировать в файловую систему `Ext.4`" - это означает, что корневой раздел (`/`) будет очищен + в него будет добавлена инфраструктура, которая необходима для работы корневого раздела.
```
mkfs.ext4 /dev/sda3
```
И еще нужно инициализировать `swap` раздел:
```
mkswap /dev/sda2
```
И еще нужно отформатировать `efi` раздел в формат FAT32:
```
mkfs.fat -F 32 /dev/sda1
```
## монтирование
Процесс монтирования - это процесс указания дискам какой у них будет статус после установки системы. Например, здесь я привяжу диск `sda3` к корневой директории `/mnt`, чтобы при установке системы именно в нем у меня был размещен ее корень `/`:
```
mount /dev/sda3 /mnt
```
А еще я привяжу `efi` раздел к папке `boot` внутри корневого раздела:
```
mount --mkdir /dev/sda1 /mnt/boot
```
Плюс стоит подключить `swap`:
```
swapon /dev/sda2
```
## установка пакетов
Нужно установить все нужные для работы ОС пакеты:
```
pacstrap -K /mnt base base-devel linux linux-firmware neovim networkmanager
```
По сути это сам процесс установки системы.
## внутри системы
После установки нужно первым делом зайти в систему:
```
arch-chroot /mnt
```
И в ней в первую очередь нужно поставить загрузчик GRUB. Здесь у меня появился вопрос: **а зачем вообще нужен grub, если есть UEFI, который по сути тоже является загрузчиком?** Все просто - `grub` будет "дружить" UEFI и Arch. По сути GRUB является дополнением к UEFI загрузчику: UEFI запускает GRUB -> GRUB запускает ядро Linux. Почему так? Потому что UEFI не умеет работать с ядром Linux.
```
pacman -S grub efibootmgr
```
После установки `grub` нужно его инициализировать:
```
grub-install --efi-directory=/boot --bootloader-id=GRUB
```
Насколько я понимаю эта команда пишет скрипт `.efi`, который и будет запускать UEFI, взаимодействуя с GRUB. Для работы GRUB еще нужно сделать конфигурационный файл:
```
grub-mkconfig -o /boot/grub/grub.cfg
```

## sudo
Чтобы установить `sudo`:
```bash
pacman -S sudo
```
Когда установлен `sudo` можно создать пользователя:
```bash
useradd -m -g users -G wheel,storage,power,audio peter
```
Здесь:
- `-m` - создать домашнюю директорию пользователя.
- `-g` - добавить в основную группу (`primary group`).
- `-G` - перечислить дополнительные группы пользователя.
- `peter` - имя нового пользователя.
Чтобы новый пользователь мог использовать `sudo` нужно пойти в `/etc/sudoers` и там сделать одну из двух вещей:
1) дать конкретно нашему новому пользователю права на пользование `sudo`.
2) дать целой группе права на пользование `sudo`.
Я предпочитаю второй вариант, потому что если будут появляться новые пользователи мне нужно будет просто добавить их в группу, а не корректировать файл `sudoers`.
Для корректирования `sudoers` есть специальный метод - `visudo`. Чтобы открыть его через `neovim` я сделаю так:
```Bash
EDITOR=nvim visudo
```
В файле закомменированы возможности выдачи пользователям админских прав. Я просто раскомментирую эту строку:
```Bash
%wheel ALL=(ALL:ALL) ALL
```
Здесь:
1) `%wheel` - имя группы, которой я выдаю права на использование `sudo`.
2) `<пользователи правила>=(<пользователь от имени которого может быть запущена команда>:<группа от имени которой может быть запущена команда>) <список команд, которые могут быть запущены>`
Самое непонятное - то, что находится в скобках. Понятнее становится, когда узнаешь, что команда `sudo` позволяет в том числе определять пользователя от лица которого будет выполнена административная команда.
Например, если я добавлю такую строку в файл `sudoers`:
```bash
peter ALL=(root) /bin/grep
```
То это будет означать, что я позволяю пользователю `peter` запускать только утилиту `grep` от лица пользователя `root`.
По сути, когда я пишу `root` в контексте пользователя, от лица которого буду отправлять сообщения, то получается, что я делаю тавтологию - "запускаю от имени администратора от пользователя `root` такую-то команду". Что будет если я пропишу так:
```bash
peter ALL=(peter_test) /bin/grep
peter_test ALL=(root) ALL
```
Тогда при запуске команды `grep` от лица пользователя `peter` для меня будет возвращаться ошибка: я не могу больше использовать пользователя `root` для запуска команд, т.к. мне нужно использовать пользователя `peter_test`. Теперь команда будет такой:
```bash
sudo -u peter_test grep -r "wheel" /etc/.
```
Я заимствую у другого пользователя права на исполнение команды `grep`.
Короче, после экспериментов я вернусь к старому варианту - разрешу группе `wheel` выполнять все команды от имени пользователя `root` (+ от всех других желаемых пользователей):
```bash
%wheel ALL=(ALL:ALL) ALL
```
 >[!important] Видимо, `sudo` берет те правила, которые указаны ниже по файлу. Потому что изначально я добавил все правила на группу, в которой был `peter`, а потом перед этим добавил правила на `peter`, которые накладывают на него ограничения, но при этом `peter` все равно мог выполнять команды от лица `root`.
## будущий рабочий стол
На винде и маке существуют заготовленные для пользователя директории типа `Downloads`, `Pictures` и т.д. На Arch Linux, понятно, ничего такого со старта нет. Однако существуют готовые версии утилит, которые в т.ч. Arch Wiki [рекомендует](https://wiki.archlinux.org/title/XDG_user_directories_(%D0%A0%D1%83%D1%81%D1%81%D0%BA%D0%B8%D0%B9)) использовать
```
pacman -S xdg-users-dirs
```
Далее нужно ввести команду:
```
xdg-users-dirs-update
```
После нее в корне появятся пользовательские папки типа `Downloads`, `Desktop` и проч.
## aur + yay
AUR - Arch User Repository это что-то вроде пользовательских не-официальных модов на Arch, которые можно скачивать на свой страх и риск. Официальный сайт репозитория: https://aur.archlinux.org/
Так как таких пакетов нет в `pacman` по-умолчанию, нужно сначала скачать пакет (с помощью `git-ssh`), затем скомпилировать специальные файлы внутри пакета и только потом выполнять стандартную установку пакета через `pacman`.
`yay` - это штука вроде `pacman`, но для пользовательских, а не для официальных пакетов. Она позволяет пропустить этап компиляции нового пакета (чтобы установить с помощью `pacman`), чтобы сразу начать установку.
Я решил, что не буду устанавливать `yay`, чтобы прочувствовать нефильтрованый опыт использования `pacman`.
## звук
Как я понял основа для любого звука в системе - [`pulseaudio`](https://wiki.archlinux.org/title/PulseAudio_(%D0%A0%D1%83%D1%81%D1%81%D0%BA%D0%B8%D0%B9)). 
И еще я выяснил одну любопытную вещь - изображение рабочего стола + сам звук обеспечиваются специальными серверами. В данном случае `pulseaudio` как раз является таким сервером.
Звуковой сервер - это третий слой работы со звуком в операционной системе.
Изображение всего устройства звуковой системы:
![[sound_system.png]]
Существует две базовые системы драйверов - `OSS` и `ALSA`. В данном случае "система драйверов" означает, что это ПО, которое связывает звуковые устройства с ОС, позволяя ОС правильно использовать внешние девайсы.
