# образ
Сначала следует скачать `iso` образ с одного из зеркал.
Список зеркал находится здесь: [Arch HTTP(S) mirrors](https://archlinux.org/download/#http-downloads)
## верификация образа
Далее **рекомендовано** верифицировать скачанный с зеркала образ.
Это делается потому, что в теории зеркало может быть взломано (или сделано) злоумышленником, который изменит находящиеся на нем образы.
Верификация проводится с помощью пары ключей:
1) Секретный ключ хранит разработчик Pierre.
2) Публичный ключ, который тоже хранит Pierre, но уже публичный.
В списке, который доступен по ссылке есть несколько способов проверить валидность файла `iso` - я пользуюсь именно ключами PGP.
- Качаем подпись с официального сайта: https://archlinux.org/iso/2025.09.01/archlinux-2025.09.01-x86_64.iso.sig
- Качаем файл `iso` с одного из зеркал.
- Размещаем оба файла в одной директории для удобства.
Теперь нужно попросить у Pierre публичный ключ. Его ключ находится в супер-защищенной библиотеке ключей WKD. И попросить его можно с помощью команды:
```Bash
gpg --auto-key-locate clear,wkd -v --locate-external-key pierre@archlinux.org
```
Теперь можно верифицировать `iso` файл подписью:
```Bash
gpg --verify archlinux-2025.09.01-x86_64.iso.sig archlinux-2025.09.01-x86_64.iso
```
На выходе должна получится примерно такая запись:
```Bash

gpg:                ключом EDDSA с идентификатором 3E80CA1A8B89F69CBA57D98A76A5EF9054449A5C
gpg:                 издатель "pierre@archlinux.org"
gpg: Действительная подпись пользователя "Pierre Schmitz <pierre@archlinux.org>" [неизвестно]
gpg: WARNING: The key\'s User ID is not certified with a trusted signature!
gpg:           Нет указаний на то, что подпись принадлежит владельцу.
Отпечаток первичного ключа: 3E80 CA1A 8B89 F69C BA57  D98A 76A5 EF90 5444 9A5C
```
Здесь `WARNING` просто предупреждает что `Pierre Schmitz <pierre@archlinux.org>` - это пользователь, с которым вы не знакомы, поэтому лучше было бы знать чей ключ мы используем. Но сейчас я точно знаю, что это наш Pierre, так что волноваться не о чем.
## механика верификации
Пьер на своем компьютере делает из файла `.iso` хэш с помощью одного из открытых алгоритмов. Затем он шифрует хэш с помощью своего секретного ключа и получается файл `.sig`.
Файл `.sig` это и есть "подпись". Она содержит:
1) Зашифрованный с помощью секретного ключа хэш.
2) Метаданные этого хэша. Например, алгоритм хэширования и проч.
Я на своем компьютере должен захэшировать свой `.iso` с помощью метаданных из `.sig`. Затем расшифровать хэш, который был зашифрован секретным ключом Пьера в `.sig`, с помощью публичного ключа (который я тоже вежливо попрошу у Пьера). И вот теперь я могу сравнить хэш Пьера и хэш моего `.iso` файла.
Если они совпадут - операция прошла успешно.
Команда:
```Bash
gpg --auto-key-locate clear,wkd -v --locate-external-key pierre@archlinux.org
```
Просит у Pierre официальный его личный публичный ключ. Этот ключ кладется утилитой `gpg` в специальную директорию на компьютере.
Команда:
```Bash
gpg --verify archlinux-2025.09.01-x86_64.iso.sig archlinux-2025.09.01-x86_64.iso
```
Делает следующее:
1) Из файла `.sig` извлекается алгоритм хэширования, который нужно применить к файлу `.iso` перед тем как сравнить хэши `.iso` и `.sig`.
2) Файл `.iso` хэшируется по алгоритму.
3) Из файла `.sig` извлекается хэш (сделанный Пьером для оригинального `.iso`), который был зашифрован с помощью секретного ключа.
4) Хэш расшифровывается с помощью публичного ключа.
5) Сравнивается хэш Пьера и хэш моего `.iso`. При совпадении хэшей операция верификации пройдет успешно.
## немного о ключах
Связка "публичный-секретный" ключ - это чудо криптографии. 
Если описывать их принцип работы просто: автор делает связку из двух ключей, секретный ключ оставляет у себя, а публичный может отдать тому, кто хочет им верифицировать автора. Продукт, который автор шифрует секретным ключом, может быть расшифрован публичным ключом, НО при этом нет никакой возможности с помощью публичного ключа восстановить секретный ключ.
Секретный ключ запирает данные, а публичный ключ может открыть данные секретного ключа. Так верифицируется продукт.
