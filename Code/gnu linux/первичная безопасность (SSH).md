Я в последнее время часто арендую Unix сервера и полезно было бы написать маленький гайд о первичной безопасности на таком сервере.
Главный объект безопасности, который я хочу рассмотреть - `ssh`. Это протокол, который позволяет удаленно управлять сервером. Аббревиатура SSH расшифровывается как **Secured SHell** ("защищенная оболочка").
Механика в общих словах проста - между клиентом (мной) и сервером (моим сервером) создается защищенный туннель, по которому мы можем обмениваться сообщениями. В частности именно через `ssh` я могу выполнять команды на Unix сервере.
Изначально, когда только арендуешь голый сервер, хостер предалагает стандартный вариант ssh-аутентификации:
```Bash
ssh <логин>@<ip-адрес сервера>
```
После этой команды еще нужно ввести пароль и вуаля - вы на сервере.
## вход по ssh-ключу
Обычно хостер предлагает подключить такую фичу на автомате, но представим, что я пропустил этот момент или у меня хуевый хостер.
Идея проста: "если я захожу по паролю на сервер, то этот пароль могут подобрать какие-нибудь фрики за каких-нибудь 10-15 дней и получить полный доступ к моей системе через `ssh`". Что нужно сделать? Запретить вход по паролю. Но в этом случае мне самому придется пользоваться каким-то другим способом входа.
Этим альтернативным способом является `ssh-key`. Как он работает? Мы создаем пару ключей у себя на компьютере и сообщаем один из ключей на свой сервер. При следующих попытках входа сервер будет сравнивать эти ключи и, если они совпали, пускать меня на сервер. Получается, что сервер и мой компьютер будут друг друга знать "в лицо" и сервер не пустит никого, кроме моего компа к себе.
Похожим образом система работает во многих сервисах, например, на `GitHub` и `GitLab`.
Сначала создаем пару ключей локально у себя на компе:
```Bash
ssh-keygen
# или
ssh-keygen -t ed25519
# или
ssh-keygen -t ed25519 -f ~/.ssh/<имя ключа>
```
Скрипт далее спросит где нужно сохранить готовый ключ и какой пароль для доступа к этому ключу установить. Хотя доступ к ключу по сути будет даваться только с моего компьютера я все равно поставлю пароль, а папку для установки оставлю прежней.
Получившийся ключ можно посмотреть:
```Bash
cat ~/.ssh/id_rsa  # приватный
cat ~/.ssh/id_rsa.pub  # публичный
```
Это просто последовательность из кучи символов.
Теперь публичную версию ключа нужно доставить на удаленный сервер. Это можно сделать с помощью другого скрипта:
```Bash
ssh-copy-id <логин>@<ip адрес сервера>
# или
ssh-copy-id -i ~/.ssh/<имя ключа> <логин>@<ip адрес сервера>
```
Меня попросят ввести пароль и после того как я это сделаю публичный ключ будет доставлен на сервер.
Следует так же не забыть добавить нового пользователя:
```Bash
adduser lipkerton
gpasswd -a lipkerton sudo
# или
useradd -m -g lipkerton -G sudo,users lipkerton
```
Теперь при новом логине через команду:
```Bash
ssh <логин>@<ip адрес сервера>
```
Мой сервер должен пустить меня без пароля.
Однако это еще не конец истории, т.к. формально я просто сделал более предпочтительный способ аутентификации. Если я попытаюсь зайти через `ssh` используя другой компьютер, то сервер предложит мне снова ввести пароль, но именно этого я и пытаюсь избежать, потому что не хочу, чтобы всякие лузеры стучались перебором мне на сервер.
Чтобы решить эту проблему нужно сделать второй шаг (после открытия доступа по `ssh-key`) - запретить вход по паролю.
Для этого нужно на сервере открыть файл настроек `ssh` (с правами администратора):
```Bash
nvim /etc/ssh/sshd_config
```
В нем есть настройка, которая называется: 
```Bash
PasswordAuthentication yes
```
И вот для нее мне нужно вместо `yes` указать `no` - это отключит вход по паролю. И еще нужно перезагрузить скрипт ("службу", "сервис" хз как называть) `ssh`:
```Bash
systemctl reload ssh
```
Готово. Теперь никто не сможет попасть на сервер, кроме владельца ключа. Да, конечно ключ можно так же попытаться подобрать, но это сделать уже гораздо сложнее, чем подобрать пароль.
# смена порта
По умолчанию `ssh` обращается к 22 порту на сервере, а сервер ожидает обращение по `ssh` на 22 порту. Это стандартный порт для такого типа подключения, поэтому его лучше бы сменить в настройках `ssh`:
```Bash
#Port 22
Port 22228
```
