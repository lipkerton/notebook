## база
Что такое venv? Это сокращение от virtual enviroment. 
Когда говорят 'виртуальное окружение', то обычно имеют в виду папку внутри проекта, которая специально создана для того, чтобы сохранить в себя версию Python, которая используется в этом проекте и пакеты, которые используются в этом проекте. 
Это нужно в частности для того, чтобы не захламлять систему пакетами, которые нужны только при работе с данным скриптом: зачем устанавливать пакет `googletrans` для всего компа, если он используется только в одном скрипте?
Создать стандартное виртуальное окружение везде создается так:
```Bash
python -m venv <имя папки окружения>
```
Когда окружение создано оно все еще не работает.
Чтобы запустить виртуальное окружение нужно запустить скрипт активации. На винде скрипт активируется просто указанием пути к нему в PowerShell:
```PowerShell
.\<имя папки с окружением>\Scripts\activate
```
Иногда может случиться так, что винда по умолчанию не позволяет запускать скрипты из терминала. Эту настройку можно выключить командой:
```PowerShell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned
```
На macOS история другая - macOS по умолчанию не позволяет устанавливать в систему пакеты через pip без активированного виртуального окружения, поэтому виртуальное окружение нужно в каждом проекте.
Активировать виртуальное окружение можно так:
```Bash
source <имя папки с окружением>/scripts/activate
```
Когда окружение будет активировано перед строкой ввода появится имя окружения в скобках - это означает, что теперь работа происходит через это окружение.
## как это работает?
Скрипт активации окружения по сути идет в переменные окружения на компьютере и перед путями до папок с Python вставляет путь до папки `bin` внутри папки окружения.
Если сложнее - он переписывает переменную окружения `PATH` так, чтобы в первую очередь за пакетами программы Python при активированном окружении обращались именно к этому активированному виртуальному окружению. Если в виртуальном окружении нет искомого пакета, то Python будет искать в своем общем хранилище.
## requirements.txt
Когда передаешь проект на другой комп или вообще другому человеку может возникнуть потребность передать и все нужные для него пакеты. Сразу возникает мысль: "Я же могу скинуть свою папку виртуального окружения!" Нет, не можешь. Почему? Во-первых, потому что так сказала официальная документация Python - [нельзя](https://docs.python.org/3/library/venv.html):

> Not considered as movable or copyable – you just recreate the same environment in the target location.

Во-вторых, потому что виртуальное окружение содержит скрипты, где прописаны абсолютные пути к интерпретатору Python на вашей машине. То есть если его переместить на другую машину, то это виртуальное окружение тупо не сможет найти Python на ней.
В-третьих, когда пакеты устанавливаются им иногда нужно скомпилировать файлы под конкретную систему, например, на языке C. Чтобы установить Pandas потребуется такая компиляция. Фишка компиляции в том, что она берет слепок вашей системы и отталкиваясь от него выполняет все нужные действия. При переносе система будет другой = пакеты не будут работать.
Что делать?
Мудрые белые господа создали штуку, которая называется `requirements.txt` - это просто txt-файл. Но! Использовав команду (при активированном виртуально окружении):
```Bash
python -m pip freeze
```
В выводе можно увидеть все установленные в этом виртуальном окружении пакеты.
А если перенаправить поток вывода в файл:
```Bash
python -m pip freeze > requirements.txt
```
То создастся файл `requirements.txt`, где будут содержаться имена пакетов и версии этих пакетов.
Этот файл уже смело можно отправлять. На другом компьютере установку из него можно производить командой (не забыть активировать новое окружение):
```Bash
python -m pip install -r requirements.txt
```
## pipenv
Ходят слухи, что стандартный способ установки окружения не так уж и круто оптимизирован, поэтому иногда можно встретить другие способы взаимодействия `pip` и виртуального окружения.
Например, сторонняя утилита `pipenv` работает таким образом, что объединяет в себе пакетный менеджер `pip` и стандартную функцию создания виртуального окружения.
Что это дает? Ну, например, эта штука способна автоматически заносить в список все нужные для работы программы штуки (а еще она может автоматически удалять из этого списка все ненужное). Так же она автоматически создает виртуальные окружения.
Чтобы ее установить нужно использовать `pip`:
```Bash
pip install pipenv
```
И дальше все штуки писать через нее, а не через `pip`:
```Bash
pipenv install <пакет>
```
