Я писал все свои маршруты в одном файле `main.py`, но так вышло, что для окружения `jinja2` нужно указать папку приложения, где оно и будет искать шаблоны и проч., поэтому мне понадобилась файловая структура.
Дополнительно к этому - структура все равно нужна, чтобы разделить большое приложение на несколько приложений поменьше, а не пихать код в один единственный файл.
Конечно, первая мысль, которая приходит в голову - сделать папку для всего приложения рядом с файлом `main.py`, поместить туда папки с кодом отдельных приложений + `templates` + `static`. Я так и сделал. Получилась такая структура:
```Bash
├── app
│   ├── __init__.py
│   └── routers  # мое первое приложение
│       ├── __init__.py
│       └── test_1.py
├── __init__.py
├── main.py  # основной файл
└── requirements.txt
```
Файлы `__init__.py` позволяют делать относительные импорты, потому что регистрирует директории, в которых они размещены, как пакеты приложения на Python.
Теперь появляется проблема - а как дать понять основному файлу приложения FastAPI, что у меня есть еще одно дочернее приложение со своими маршрутами?
Для этого в FastAPI существует класс `APIRouter`. Он используется точно так же, как используется класс `FastAPI` в основном файле приложения. По сути код дочернего приложения больше ничем не отличается - просто добавляем другой класс и его экземпляр используем в декораторах:
```Python
from fastapi import APIRouters


router = APIRouters()


@routers.get("/test")
def test():
	return "test"
```
Все, что нужно сделать далее - сказать основном файле, что у нас существует дополнительный файл приложения:
```Python
from fastapi import FastAPI
from app.routers import test_1


app = FastAPI()

app.include_router(test_1.router)
```
