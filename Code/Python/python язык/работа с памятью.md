модуль `memoryview` позволяет работать со структурами данных Python, напрямую влияя на них в памяти. 
прежде чем управлять объектом в памяти нужно сначала представить какой-нибудь Python-объект в виде объекта в памяти:
```Python
from array import array
a = array('B', range(6))
```
в данном случае `'B'` означает `unsigned char`.
`unsigned char` - это положительное число размером в один байт. если число может занять только один байт, то в этом случае диапазон значений в десятичной системе у него будет таким - `0-255`.
почему только положительное? потому что в названии типа указано слово `unsigned`, что переводится как *"без знака"*. в `array` так же есть тип `signed char`, который так же занимает один байт, но т.к. этот тип уже `signed`, то он должен выражать числа с знаком `+ -`. это означает, что в десятичной системе счисления у такого типа диапазон значений - `-128-127`.

теперь можно сделать объект-представление-в-памяти:
```Python
from array import array
a = array('B', range(6))
b = memoryview(a)
```
в `b` будет храниться что-то вроде такого: `<memory at 0x7b4028f61fc0>`.
важно понимать, что теперь `b` указывает на ту же самую область в памяти, на которую указывает `a`. это означает, что любые изменения, которые будут проведены над `b` тут же отразятся на объекте, на который ссылается `a`.
теперь я проверю, что в `b` действительно хранится тот же самый набор чисел:
```Python
b.tolist()  # => [0, 1, 2, 3, 4, 5]
```
теперь я сделаю что-нибудь безобидное - буду манипулировать вложенностью нашего списка:
```Python
c = b.cast('B', [2, 3])
c.tolist()  # => [[0, 1, 2], [3, 4, 5]]
print(c)  # => <memory at 0x7b4028f61fc0>
```
как видно указатель на место объекта в памяти остался тот же.
я сказал ранее, что *любые изменения отразятся на исходном объекте*, однако странно, что увеличение вложенности объекта никак не повлияло на исходный:
![[memoryview5.png]]
как же так?
на самом деле ничего не меняется, потому что мы не меняем байты, т.е. сами числа, мы меняем только способ их "просмотра". не важно в каком виде представлены байты в коде, важно что они неизменны в памяти. поэтому, хотя метод `.cast()` демонстрирует другое представление объекта, исходный объект остается все тем же. таким образом я могу получить различные представления одного и того же массива.
однако, когда я попытаюсь присвоить новое значение одному из элементов:
```Python
from array import array
a = array('B', range(6))
b = memoryview(a)
c = b.cast('B', [2, 3])
c[1, 1] = 22
print(a)  # => array('B', [0, 1, 2, 3, 22, 5])
```
теперь я не просто изменил представление байтов, но еще и буквально поменял один из них.

теперь пришло время для получения удовольствия.
как можно использовать модуль во вред?
очень просто.
но перед этим нужно научиться различать типы данных в `C`.
когда я работал выше с `unsigned` и `signed` значениями, все было просто, т.к. их запись занимает лишь один байт в памяти. а что будет если число занимает несколько байт?
т.к. в `unsigned char` помещаются только числа в диапазоне `0-255`, для чисел побольше нужно использовать другой тип данных. например, `unsigned short`.
переменная типа `unsigned short` может принимать числовые значения в диапазоне `0-65535`. а может она так, потому что для записи одного числа тип `unsigned short` использует не один байт, а сразу два байта.
как это работает?
```
2 байта = 16 бит = 1111 1111 1111 1111
```
каждая единичка тут это $2^{n}$, где $n$ - это номер позиции единицы, начиная справа.
например:
$$
0000.0000.0000.0001 = 2^{0} = 1
$$
$$
0000.0000.0000.0010 = 2^{1} = 2
$$
вопрос: *а как сделать тройку*?
просто:
$$
0000.0000.0000.0011 = 2^{1} + 2^{0} = 3
$$
так вот.
чтобы понять почему в два байта может поместиться число $65535$, а в один байт $255$, нужно просто заполнить все позиции единицами и просуммировать:
$$
1111.1111.1111.1111 = 2^0 + 2^1 + 2^2 + 2^3 + 2^4 + 2^5+ \dots + 2^{15} = 65535
$$
другой лор происходит в `signed short`. так как это число *со знаком*, это означает, что в переменную с таким типом могут быть записаны числа в диапазоне `-32768-32767`.
если просуммировать числа (не учитывая, что одно из них отрицательное), то получится $65535$. все вроде сходится, но теперь нужно понять каким образом можно получить отрицательное число в двоичной записи.
для начала я попробую вычислить $32767$:
$$
0111.1111.1111.1111 = 2^0 + 2^1 + 2^2 + 2^3 + 2^4 + 2^5 + \dots + 2^{14} = 32767
$$
почему в первом бите записи стоит $0$? потому что это как раз указатель на знак (aka `sign`) числа - плюс.
теперь я вычислю $32768$:
$$
1000.0000.0000.0000 = 2^{15} = 32768
$$
первая единичка указывает на знак - минус.

так вот.
как использовать `memoryview` во вред на примере `signed short`.
для начала нужно подготовиться:
```Python
from array import array
a = array('h', [-2, -1, 0, 1, 2])
b = memoryview(a)
```
так как каждое `signed short` число занимает два байта, я хочу посмотреть на эти байты.
чтобы это сделать я применю метод `.cast()`:
```Python
c = b.cast('B')
```
и как видно я превращаю каждое число из `signed short` в `unsigned char`.
магия!
на самом деле нужно понять, что `signed short` и `unsigned char` - это просто два разных представления байтов в памяти, поэтому трансформации между ними лишь покажут в нашем коде два разных представления одних и тех же данных.
так как, в `signed short` числа записываются двумя байтами, а в `unsigned char` одним, на выходе в переменной `c` будет лежать следующее:
![[memoryview8.png]]
что это?
числа были представлены двумя байтами, следовательно тут для каждого числа исходного массива появилось по два числа в новом:
```
-2 -> [254, 255]
-1 -> [255, 255]
0  -> [0, 0]
1  -> [1, 0]
2  -> [2, 0]
```
вопрос: *почему `-2 -> [254, 255]`*?
потому что
$$
0000.0000.0000.0010 = 2^1 = 2
$$
секретная техника, чтобы сделать отрицательное число:
$$
1111.1111.1111.1101 = -2
$$
втф...
а так происходит по правилу преобразования отрицательных чисел в двоичной системе счисления:
$$
-2 = 256 - 2 = 254 = 1111.1101
$$
вопрос: *а почему мы вычитаем из $256$, если в сумме один байт равен $255$*?
потому что в диапазоне `0-255` не $255$ возможных значений, а $256$, так как ноль тоже учитывается.
в формате `signed short` (16 бит):
$$
1111.1111.1111.1101 = -2
$$
а в формате `unsigned char` (8 бит):
$$
1111.1111 = 255
$$
$$
1111.1101 = 254
$$
по аналогии можно понять откуда функция `c.tolist()` достала  такой список в коде выше.
теперь я сделаю гадость - поменяю один байт:
```Python
c[5] = 4
c.tolist()  # => [254, 255, 255, 255, 0, 4, 1, 0, 2, 0]
```
и посмотрю что стало с исходной последовательностью:
![[memoryview9.png]]
