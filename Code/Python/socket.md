# База

## Что вообще такое socket?

Это интерфейс, встроенный в набор стандартных пакетов Python. Он, с помощью набора функций, позволяет настроить обмен пакетами в сети.

Она содержит ряд основных функций:

1. `socket()` - создает сокет.
2. `bind()` - привязывает к сокету адрес и порт.
3. `connect()` - отправляет запрос на коннект от клиента к серверу. **Блокирующая функция.**
4. `accept()` - принимает запросы на коннект от пользователей на сервере. **Блокирующая функция.**
5. `recv()` - обычно используется с TCP, потому что возвращает только данные, которые были “пойманы” на соединении. **Блокирующая функция.**
6. `recvfrom()` - обычно используется с UDP, потому что возвращает не только сами данные, но еще и адрес отправителя. Так происходит, потому что соединение может быть не установлено, а знать отправителя все равно может быть полезным. **Функция является блокирующей.**
7. `sendto()` - используется, когда нам нужно отправить сообщение на конкретный адрес. Эта функция принимает два аргумента - сообщение и адрес. Нужна при создании запросов и ответов UDP, т.к. там нет никакого соединения, поэтому для обмена сообщениями нужно знать адрес.
8. `sendall()` - функция, которая отправляет сообщения целиком через соединение. Используется с протоколом TCP. Для TCP - **это блокирующая функция**, потому что TCP гарантирует доставку пакетов получателю.
9. `send()` - функция, которая отправляет конечное количество байт. То есть в нее можно отдать только ограниченное по размеру сообщение. Используется с TCP. Для TCP - **это блокирующая функция**, потому что TCP гарантирует доставку пакетов получателю.

## Мои мысли по поводу socket

Это программа, которая позволяет составлять низкоуровневые запросы сервер-клиент. Работать с ней довольно сложно и я не уверен реализуются ли на ней другие сетевые паттерны (типа “публикатор-подписчик”). При этом на ней можно реализовать прокси-сервер, который будет обмениваться запросами по протоколу HTTP и, я уверен, на нем можно реализовать еще и нормальный прокси по протоколу HTTPs.

```Python
import socket


server_address = ('localhost', 6789)

server = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
server.bind(server_address)
```

Тут надо объяснить где на самом деле создается сокет - в переменной `server` . В ней вызывается функция `socket()` из модуля `socket` (если че в коде это выглядит вот так - `socket.socket()` ). Что за странные константы передаются в качестве аргумента внутрь функции? Это как раз указания на то, что мы создаем основу сокета на протоколах IP и UDP.

`AF_INET` - это **address family**. В модуле `socket` есть огромная куча формочек для адресов, которые мы можем использовать для привязки конкретного типа адреса к ним. Мы хотим использовать стандартный IPv4 - это значит, что мы будем привязывать к сокету стандартный IP-адрес, который находится в диапазоне от 0.0.0.0 - 255.255.255.255. Поэтому после **AF** мы через нижний слеш указываем **INET**. Если бы мы хотели использовать другой формат адреса, например, IPv6, то нам бы пришлось вызвать другую константу из модуля `socket` - `AF_INET6` .

`SOCK_DGRAM` - это **socket type**. **DGRAM** означает, что мы будем использовать протокол UDP, который отправляет информацию как раз в виде датаграм (ну и расшифровывается он как User Datagram Protocol).

```Python
import socket


server_address = ('localhost', 6789)

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.bind(server_address)
```

Здесь изменилась только одна константа - `socket.SOCK_STREAM` .

`SOCK_STREAM` - это тоже **socket type**, просто другой. **STREAM**, вероятно означает, что данные будут “передаваться в потоке”. Короче, это константа, которая готовит сокет к использованию протокола TCP (Transmission Control Protocol).

# Сервер-клиент на UDP

## Сервер-клиент

Как написать полную программу сервака, который будет слушать порт?

```Python
import socket


server_address = ('localhost', 6789)

server = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
server.bind(server_address)

max_size = 4096
data, client = server.recvfrom(max_size)

print('Данные и адрес, с которого они пришли:', data, client)
server.sendto(b'Are you talking to me?', client)
server.close()
```

Очень много произошло. Во-первых, после организации сокета в переменной `server` мы обращаемся именно к сокету, а не к модулю `socket` . Во, вторых появилось три функции `recvfrom()` , `sendto()` , `close()` .

`server.recvfrom(max_size)` - это функция, которая ожидает сообщения на сервер. Сообщения могут быть любого размера, но в итоге будут обрублены до размера, который указан в переменной `max_size` - 4096 байт. Строго говоря мы можем указать любой размер вплоть до 65535. Кажется, это обосновано лимитами протокола UDP. Максимальный размер называется **чанком (chunck)** - это размерность куска данных, которые мы будем получать за раз, то есть в теории мы можем считать что-то большое, просто по кускам. Но это не точно (особенно с UDP форматом - TCP больше подходит для передачи больших сообщений). Короче, данные, которые мы получаем через функцию `recvfrom()` приходят в формате кортежа, который можно распаковать на две переменные - `data` и `client` .

`client` - это переменная, которая хранит адрес, с которого пришли данные в точно таком же формате, как мы храним адрес в переменной `server_address`.

`data` - это переменная, которая будет в формате bytes хранить ту строчку, которая пришла от клиента.

`server.sendto(b'some_message', client)` - это функция, она отправляет сообщение, которое было передано в первом аргументе, на адрес, который передан во втором.

`server.close()` - завершает работу сервера.

Теперь к серверу нужно написать клиент. Строго говоря мы можем попытаться обратиться напрямую к серверу через браузер, но я такое пока не тестировал.

```Python
import socket


server_address = ('localhost', 6789)
max_size = 4096

client = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
client.sendto(b'Hey!', server_address)
data, server = client.recvfrom(max_size)
client.close()
```

Здесь все то же самое, только нет метода `bind()` , который по идее должен привязывать сокет к определенному адресу. Любанович это объясняет тем, что сервер сначала принимает данные, а потом отправляет; клиент же поступает наоборот, поэтому и `bind()` ему не нужен.

## Блокировки

> [!important] Главный вопрос - как наш скрипт будет ждать запросы, если он написан “сплошняком”, т.е. перед командой `close()` нет никаких препятствий?

Ответ на него в том, что часть функций, которые представлены в коде являются блокирующими, то есть функция `server.recvfrom()` будет ждать какой-нибудь пакет и не пустит код дальше.

# Сервер-клиент на TCP

Основа основ и базированная база. Сервер-клиент на TCP - это то, что ищут все, когда речь идет об отправке пакетов в интернет. Схематично работу сервер-клиента на TCP можно продемонстрировать с помощью одной картинки:

Здесь описан набор самых основных функций, которые мы сейчас будем использовать - основное различие состоит в том, что TCP-сервер принимает соединения через функцию `accept()` , а TCP-клиент коннектится к серверу через функцию `connect()` . Перед тем как сервер примет соединения ему необходимо их ожидать - это он делает через функцию `listen()` .

> [!important] **handshake** - это термин, который используют для описания первоначальной связи клиента и сервера.

## Сервер-клиент (базовый)

Итак, ничего принципиально не меняется, но для TCP используются немного иные функции.

```Python
import socket


server_address = ('localhost', 6789)
max_size = 1000

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.bind(server_address)

server.listen(5)

client, addr = server.accept()
data = client.recv(max_size)

client.sendall(b'Are you talking to me?')
client.close()
server.close()
```

Давайте проговорим еще раз - TCP сначала принимает все запросы на подключение к серверу, а потом уже начинает передавать сообщения.

`server.listen()` - это функция, которая слушает очередь из запросов на подключение к серверу. Она ожидает пять подключений, а всем остальным отказывает - это аналог SERVER BUSY IN THIS MOMENT. На самом деле она не является обязательной - если этой функции не будет в коде количество соединений будет ограничено параметрами системы.

`server.accept()` - функция одобряет запрос на подключение и возвращает две сущности. Первая сущность - это ~~объект клиента~~ не объект клиента, а вторая - адрес клиента. ~~Объект клиента нам необходим, т.к. в рамках протокола TCP мы будем обращаться к конкретному клиенту, чтобы слушать от него сообщения.~~`server.accept()` блокирует выполнение кода, потому что дожидается входящих соединений.

`client` - я был не прав - Любанович в учебнике просто хуево называет эту переменную, поэтому не совсем понятно что она делает. Итак! На самом деле переменная `client` не хранит в себе объект клиента она хранит объект соединения между сервером и клиентом. Поэтому все следующие команды будут отдаваться именно через обращение к этой переменной.

`client.recv()` - функция ловит все то, что скажет клиент в размере `max_size` и записывает в переменную `data` .

`client.sendall()` - почему здесь указана именно эта функция, хотя раньше я использовал `sendto()` ? Метод `sendto()` требует передачи ему в качестве аргумента адрес получателя сообщения. Но зачем нужен адрес получателя сообщения, если у нас уже установлена связь между клиентом и сервером? Получается, мы можем не указывать этот адрес и использовать метод `send()` , чтобы просто передавать данные по “мосту”. Фишка в том, что и метод `send()` здесь лучше не использовать, т.к. он довольно низкоуровневый и отправляет столько байт, сколько ему скажут отправить - он обрежет большое сообщение до определенного количества байт. Метод `sendall()` , напротив, будет отправлять сообщение до тех пор пока не отправит его полностью, либо пока не вернется ошибка.

Все, что нужно сделать клиенту - отправить запрос на коннект к серверу и начать отправлять сообщения.

```Python
import socket


server_address = ('localhost', 6789)
max_size = 1000

client = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client.connect(server_address)

client.sendall(b'Hey!')
data = client.recv(max_size)
print(f'Someone replied: {data}; from {server_address}')
client.close()
```

`client.connect()` - с помощью этой функции клиент отсылает запрос на коннект серверу.

`client.sendall()` - идентичная функция, как и в коде сервака. Мы просто отправляем полное сообщение по коннекту с сервером.

> [!important] НАПОМИНАНИЕ! Многих может смутить, что и в коде сервера и в коде клиента я обращаюсь именно к переменной клиента, чтобы отправлять сообщения. Как в таком случае они вообще попадают на сервер? Они попадают на сервер, потому что изначально сокет организован именно через протокол TCP. То есть этот сокет может отправлять сообщения ТОЛЬКО при соединении с сервером. А когда соединение установлено мы больше не пользуемся изначальным объектом сокета - мы пользуемся объектом соединения.

## Эхо сервер

Идея простая - у нас будет сервак, который на все запросы отправляет ответы, которые полностью совпадают с запросом. Это даже легче, чем было в предыдущем примере - там сервер обменивался строчкой с клиентом. Здесь мы просто будем принимать от клиента сообщения и тут же отдавать их обратно.

```Python
import socket


server_address = ('localhost', 6789)
max_size = 1024

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as server:
	server.bind(server_address)
	server.listen(5)
	connection, client_address = server.accept()
	
	with connection:
		print(f'Подключен к {client_address}')
		while True:
			data = connection.recv(max_size)
			# здесь проверяем не пришла ли пустая строка - это означает,
			# что сообщение, которое направлялось на сервер либо закончилось,
			# либо пришло пустым.
			if not data:
				break
			connection.sendall(data)			
```

Здесь произошло очень много нового. Во-первых, для создания сокета используется менеджер контекста ([[context manager]]). Во-вторых, используется цикл для обработки сообщений, которые приходят по соединению; то есть если сообщение будет очень большим (если не поместится в 1024 байта) - оно будет по кускам (чанками) обработано. Когда сообщения закончатся, цикл прервет свою работу.
