## база
ООП - объектно-ориентированное программирование. Этой аббревиатурой описывается способ программировать через проектирование собственных объектов.
Все объясняют зачем нужно ООП по-разному (иногда пишут, что оно вообще не нужно). Чтобы понять ответ на этот вопрос лучше всего обратиться к практике использования функций в Python.
Когда я занимаюсь динамическим программированием (т.е. разбиваю задачу на много мелких подзадач), то логика программы часто состоит только из функций, которые передают данные одна в другую, строя такую длинную-длинную цепочку. Проблема такого подхода в том, что он немного "грязный" и запутанный: что если я захочу обратиться к какой-нибудь функции, которая выполняет нужную мне логику, но немного не так как мне бы хотелось? Придется переписать функцию в новую функцию или создать внутри существующей какую-нибудь неудобную конструкцию, которая будет реализовывать решение для разных случаев. А иногда я смотрю на код и понимаю, что функции в нем просто свалены в кучу, хотя какие-то из них никак не взаимодействуют с другими, потому что применяются только в конкретной ситуации.
Обычно именно в таком контексте спасает ООП. Оно отвечает на вопрос: что если у нас бы были структуры данных, которые посвящены одной конкретной задаче и содержат полный набор функций для решения этой задачи?
Обычно под ООП имеют в виду использование классов:
```Python
class NewClass:  # класс.
	m = 10
	n = 15

a = NewClass()  # экземпляр класса.
print(a.m + a.n)
```
У классов есть методы:
```Python
class NewClass:  # класс.
	m = 10
	n = 15

	def sum_two(self):
		return self.m + self.n

a = NewClass()  # экземпляр класса.
print(a.sum_two())
```
Но если рассматривать ООП только таким образом, то получится, что мы выдумали альтернативу функциям, которая вроде как бы должна решать проблему стандартного подхода, но в итоге превращается в такого же мутанта с гигантским набором случайных инструментов.
Здесь стоит вспомнить, что ООП - это не классы, а парадигма программирования. Это значит, что не каждый класс будет являться реализацией ООП, потому что не каждый класс соответствует особым принципам ООП.
Принципы - это штуки, которые нужны, чтобы спроектировать свой класс правильно, чтобы на его основе можно было создать кучу красивых объектов и пользоваться ими эффективно. Сами принципы не являются правилами - это рекомендации к написанию классов.
Так как принципы являются рекомендациями, их набор иногда меняется - в каких-то статьях их четыре, а где-то три. Но основной набор все еще тот же: инкапсуляция, полиморфизм и наследование, а опционально наиболее часто упоминается абстракция.
## инкапсуляция
Это когда мы не даем менять на ходу атрибуты объекта или класса. То есть не даем прямой доступ к данным любому, кто захочет воспользоваться нашим классом. Это нужно, чтобы стандартное поведение класса не превратилось случайно в помойку.
```Python
class NewClass:
	__number1 = 20
	__number2 = 21

	def __init__(self, number):
		self.__number3 = number

	def get_number1(self):
		return self.__number1
	def get_number2(self):
		return self.__number2
	def get_number3(self):
		return self.__number3
```
Чтобы соблюсти принцип инкапсуляции я добавил к именам атрибутов двойное подчеркивание (это делается, чтобы было меньше шансов случайно изменить атрибут + скрыть его реальное имя) и специальные методы, которые будут выдавать значение атрибута по запросу.
```Python
p = NewClass(10)
print(p.get_number1())
```
### property
Но такой подход не очень удобный, потому что заставляет писать синтаксис функции, чтобы получить значение атрибута. Лучше, чтобы мы применяли стандартный синтаксис, чтобы получить атрибут. Как это сделать?
Можно добавить декоратор `@property` или использовать функцию `property`:
```Python
class NewClass:
	__number1 = 20
	__number2 = 21

	def __init__(self, number):
		self.__number3 = number

	@property  # для имени атрибута будет использовано имя функции.
	def number1(self):
		return self.__number1

	def get_number2(self):
		return self.__number2
	def get_number3(self):
		return self.__number3
	# для имени атрибута будет использовано имя переменной.
	number2 = property(get_number2)
	number3 = property(get_number3)
```
Теперь атрибуты можно получать с помощью этих функций стандартным методом:
```Python
p = NewClass(10)
print(p.number1, p.number2, p.number3)
```
Если кто-то попытается заменить значение атрибута:
```Python
p.number1 = 100
```
То вылетит ошибка `AttributeError`.
Единственная проблема такого подхода только в том, что в сущности таким образом мы устанавливаем неизменяемые атрибуты - после создания класса их значение уже нельзя поменять.
### setter
Чтобы все же дать доступ к изменению атрибута можно задать `setter`.
Точно так же как `property` делается двумя способами - с помощью декоратора и с помощью функции `property`.
```Python
class NewClass:
	__number1 = 20
	__number2 = 21

	def __init__(self, number):
		self.__number3 = number

	@property  # для имени атрибута будет использовано имя функции.
	def number1(self):
		return self.__number1
	@number1.setter
	def number1(self, value)
		# добавляем проверку для входных значений.
		if value > 0 and type(value) is int:
			self.__number1 = value
		else:
			raise AttributeError

	def get_number2(self):
		return self.__number2
	def get_number3(self):
		return self.__number3
	# добавляем сеттеры, чтобы вставить их в функцию property.
	def set_number2(self, value):
		if value > 0 and type(value) is int:
			self.__number2 = value
		else:
			raise AttributeError
	def set_number3(self, value):
		if value > 0 and type(value) is int:
			self.__number3 = value
		else:
			raise AttributeError
	# для имени атрибута будет использовано имя переменной.

	number3 = property(get_number3, set_numbers)
```
В примере в каждом сеттере проводится проверка входного значения - это вариант, в котором можно использовать сеттеры. Мы не даем изменять атрибут как угодно.
## полиморфизм
Я постоянно забываю какие-то имена функций и лезу проверять как они пишутся. Особенно все плохо, когда разработчики называют одно и то же действие десятью разными словами. 
Полиморфизм - это принцип, который нужно соблюдать, чтобы избежать этой проблемы. 
Класс часто называют "интерфейс". Фишка в том, чтобы мы могли вызывать один и тот же функционал из разных классов по одному имени. То есть интерфейс должен быть один.
Например, у нас есть две функции в двух разных классах с именами "спать" и "вздремнуть". Обе функции выполняют одно и то же действие - заставляют программу ждать пять секунд. Если их функционал одинаков, то и их имена (интерфейс) должен вызываться одинаково.
Общие интерфейсы должны быть реализованы в том числе для классов-наследников. Нельзя, радикально переопределять родительский метод в наследнике.
```Python
import time


class NewClass1:
	def sleep(self):
		time.sleep(5)

class NewClass2:
	def sleep(self):
		time.sleep(5)

a = NewClass1()
b = NewClass2()
a.sleep()
b.sleep()
```
## наследование
Наследование - это главный принцип проектирования в парадигме ООП. Построить один класс не является проблемой. Но ООП - это не про строительство одного класса, а про строительство системы взаимосвязанных классов.
Кажется сложным, но на самом деле это принцип, который описывает схему, в которой существуют родительские и дочерние классы.
Например, у нас есть класс "пользователь", а этому пользователю наследует дочерний класс "админ". У админа есть все характеристики пользователя + несколько своих собственных. Точно так же от класса "пользователь" могут быть унаследованы классы "аноним", "VIP-пользователь" и проч.
Смысл в том, что мы наделяем каждый класс общими характеристиками, а потом делаем спецификацию там, где это необходимо.
```Python
class User:
	def __str__(self):
		message = 'Я пользователь!'
		return message

class Admin(User):
	def __str__(self):
		# вызываем метод родительского класса.
		message = super().__str__()
		return message + ' А еще я админ!'

a = User()
b = Admin()
print(a)  # Я пользователь!
print(b)  # Я пользователь! А еще я админ!
```
В текущем примере я переопределил метод родительского класса в классе наследнике, чтобы показать как могут наследоваться методы. 
### `super()`
Функция `super()` нужна, чтобы вызвать метод, который записан в родительском классе и каким-то образом его использовать в методе дочернего класса. В данном случае я вызвал в дочернем классе функцию `__str__()` и добавил к результату еще одну фразу.
В классических примерах наследования однако есть правило - нужно, чтобы класс наследник мог заменить в коде родительский класс без поломок. Это показывает, что наследник сохраняет нетронутыми все родительские характеристики - значит наследование прошло успешно.
Еще одна важная особенность - наследование должно быть оправдано по смыслу. Можно унаследовать самолет от машины - интерпретатор не выделит это как ошибку, но такая архитектура по смыслу ошибочна.
## абстракция
В сущности - это фундамент проектирования. Прежде чем создавать какой-то класс мы должны подумать какие характеристики станут ключевыми для его будущих объектов. Если мы хотим создать класс "машина", то ее ключевые характеристики - скорость, мощность двигателя, расход топлива и проч. Далее при желании можно подумать о том какие характеристики лучше оставить для конкретной марки автомобиля и т.д.
Смысл в том, что мы делаем класс для создания множества различных объектов, поэтому мы должны найти то базовое, что присутствует в каждом из них. Это и есть абстракция - мы абстрагируемся (отдаляемся) от детального рассмотрения предмета и пытаемся сконцентрироваться на его ключевых особенностях.
