## делаем заготовку
Первое, что нужно сделать в пустой папке, где в будущем будет находится проект Django - создать виртуальное окружение. Оно будет хранить все нужные для приложения компоненты.
```Bash
python3.8 -m venv .venv
source .venv/bin/activate
```
Далее я создал `gitignore`:
```Bash
touch .gitignore
```
И прописал в него все вещи, которые я не хочу загружать в GitHub:
```
.venv
.gitignore
.env  # файл будет хранить секреты.
```
Далее сразу можно установить сам Django (буду пользоваться версией `3.2.16`) и дополнительно библиотеку `dotenv`, которая поможет сохранить секретный ключ из файла `settings.py` в секрете.
```Bash
python3.8 -m pip install Django=3.2.16
python3.8 -m pip install python-dotenv
```
## создаем проект
Далее я создаю сам проект:
```
django startproject second_one
```
Получается вот такая структура:
```Bash
second_one
├── .env  # секреты проекта
├── .gitignore  # игнорируемые гитом файлы
├── requirements.txt  # зависимости
├── second_one  # сам проект
│   ├── manage.py  # файл управления проектом
│   └── second_one
│       ├── asgi.py
│       ├── __init__.py
│       ├── settings.py  # настройки проекта
│       ├── urls.py  # пути проекта
│       └── wsgi.py
└── .venv
```
## активируем сервер разработки
Активация сервера разработки происходит через файл `manage.py` - он принимает разные аргументы из командной строки и отправляет в функцию `execute_from_command_line()`.
Чтобы активировать сервер нужно обратиться к файлу `manage.py` и передать аргумент - команду `runserver`.
```Bash
python3.8 manage.py runserver
```
Ответ будет таким:
```Bash
Watching for file changes with StatReloader
Performing system checks...

System check identified no issues (0 silenced).

You have 18 unapplied migration(s). Your project may not work properly until you apply the migrations for app(s): admin, auth, contenttypes, sessions.

Django version 3.2.16, using settings 'second_one.settings'
```

В этом ответе сказано несколько вещей:
1) `Starting development server at http://127.0.0.1:8000/` - сервер запустился и обратиться к нему можно через браузер по адресу `http://127.0.0.1:8000`.
2) `You have 18 unapplied migration(s). Your project may not work properly until you apply the migrations for app(s): admin, auth, contenttypes, sessions. Run 'python manage.py migrate' to apply them.` - есть каких-то 18 не принятых миграций.
## миграции
Django идет со встроенной поддержкой базы данных. Это означает, что он по умолчанию сразу создает файл БД в директории проекта.
Однако как мы все знаем не так просто из Python управлять SQL базой, поэтому в Django есть особая процедура - миграции. 
Миграции - это штуки, которые синхронизируют текущую структуру моделей Django с БД. Например, в будущем появится модель "Пользователь" - эта модель будет образом таблички, которую мы хотим видеть в БД. Фишка в том, что мы можем создавать таблички не вручную на языке SQL, а написать модель в Django. Вот, чтобы на основании этой модели создалась табличка нужно применить миграции.
Почему Django просит применить миграции уже сейчас?
Потому что он хочет создать базовую инфраструктуру проекта, например, таблицу для администрирования проекта. 
Чтобы применить стартовые миграции нужно обратиться к файлу `manage.py`:
```Bash
python3.8 manage.py migrate
```
Результат будет примерно таким:

```Bash
python3.8 manage.py makemigrations
python3.8 manage.py migrate
```
## создаем приложение
Приложения в Django - это отдельные папки, которые содержат внутри себя все, что нужно для работы.
В идеале - любое приложение должно обладать свойством транспортабельности. То есть нужно создать такое приложение, которое можно выдернуть из одного проекта, как кирпич и вставить в другой без масштабных ошибок.
Чтобы создать приложение нужно обратиться к файлу `manage.py` и написать команду `startapp` + добавить имя приложения.
```Bash
python3.8 manage.py startapp catalog
```
По итогу - новая папка появится в директории, откуда была вызвана команда. Структура у меня вышла такой:
```Bash
second_one
├── .env
├── .gitignore
├── requirements.txt
├── second_one
│   ├── catalog  # новое приложение
│   │   ├── admin.py
│   │   ├── apps.py
│   │   ├── __init__.py
│   │   ├── migrations
│   │   ├── models.py
│   │   ├── tests.py
│   │   └── views.py
│   ├── db.sqlite3  # файл базы данных
│   ├── manage.py
│   └── second_one
│       ├── asgi.py
│       ├── __init__.py
│       ├── __pycache__
│       ├── settings.py
│       ├── urls.py
│       └── wsgi.py
└── .venv
```
## регистрация приложения
Недостаточно просто создать приложение. Нужно дать Django понять, что появился новый компонент проекта.
Чтобы это сделать быстро - внутри папки с новым приложением лежит файл `apps.py` и в нем обозначен класс, который задает базовую конфигурацию приложению.
Мой файл выглядит так:
```Python
from django.apps import AppConfig


class CatalogConfig(AppConfig):
    default_auto_field = 'django.db.models.BigAutoField'
    name = 'catalog'  # сокращенное имя нашего приложения.
```
Чтобы "зарегистрировать" новое приложение нужно прописать путь до этого конфига в переменной `INSTALLED_APPS`. Она находится в файле `settings.py`.
```Python
INSTALLED_APPS = [
    'catalog.apps.CatalogConfig', # новое приложение.
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
]
```
