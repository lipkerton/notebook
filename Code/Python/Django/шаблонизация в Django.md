## база
Почему мы говорим вообще о каком-то шаблонизаторе, если для нас достаточно написать код в HTML файле, а потом отправлять этот HTML файл в ответ на запрос? Не все так просто, потому что шаблонизатор по сути предоставляет множество инструментов, которые помогут сделать наши шаблоны более гибкими. Например, с помощью шаблонизатора можно вставлять в шаблон нужные нам данные или делать какие-нибудь циклы по вставке данных.
Чтобы шаблонизатор выполнил свою функцию нужно перед оправкой шаблона сначала провести рендеринг шаблона - так мы "крафтим" из нашего скелета готовый вариант, который и будет отправлен пользователю.
Соответственно получаем два главных пункта:
1) создание шаблона
2) рендеринг шаблона
## настройки шаблонизатора
Настройки шаблонизатора хранятся в `settings.py` в константе `TEMPLATES`. Внутри лежит словарь с ключами, для которых прописаны какие-то значения.
1) `BACKEND` - здесь прописывается текущий шаблонизатор для проекта. Существует несколько вариантов шаблонизаторов и они различаются синтаксисом + возможностями. В Django по умолчанию используется шаблонизатор Django Template - он и прописан здесь `django.template.backends.django.DjangoTemplates`.
2) `DIRS` - список ссылок на папки, которые хранят шаблоны проекта.
3) `APP_DIRS` - насколько я понял, определяет нужно ли фреймворку искать шаблоны внутри приложений проекта.
4) `OPTIONS` - определяет настройки шаблонизатора.
А что такое `context_processors`? Это функции Python, которые хранятся на беке Django. Базово - это штуки, которые обрабатывают пользовательский `request`, извлекают оттуда какую-то информацию и добавляют в словарь контекста при рендеринге шаблона. Эта история нужна, чтобы добавить внутрь каждого рендеринга в проекте какие-то настройки по умолчанию.
## где хранят шаблоны
Перед тем как рендерить шаблоны нужно указать фреймворку место где он их будет искать. Такое место указывается в файле `settings.py` в константе `TEMPLATES`.
```Python
TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [TEMPLATES_DIR],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.debug',
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
            ],
        },
    },
]
```
В ключе `DIRS` должен храниться список директорий, где Django будет искать шаблоны.
Существует два способа хранить шаблоны в проекте. 
Первый заключается в том, что мы будем хранить шаблоны для каждого приложения автономно - это означает, что в каждом приложении будет лежать папка с шаблонами `templates`. Чаще всего этот способ используют, когда нужно создать полностью независимые блоки приложений, которые потом можно будет отнести в другие проекты.
Второй заключается в том, что мы будем использовать единую папку шаблонов для всего проекта. В этом случае мы сохраним ее в корне проекта (рядом с `manage.py`). Преимущества в данном случае понятны - все шаблоны лежат в одном месте.
Я сделал все вторым способом, поэтому у меня получилась такая структура проекта:
```Bash
second_one
├── catalog
├── db.sqlite3
├── homepage
├── manage.py
├── second_one
└── templates  # директория для шаблонов.
```
А как прописать путь к папке шаблонов? Как он должен строится? В файле `settings.py` есть константа, которая указывает на корневую директорию проекта (та директория, где лежит `manage.py`) - `BASE_DIR`. Она хранит путь относительно файла `settings.py` в формате `Path`, поэтому можно просто сделать еще одну константу, которая добавит к этой `BASE_DIR` наименование папки `templates`:
```Python
TEMPLATES_DIR = BASE_DIR / 'templates'
```
## как настроить выдачу шаблонов?
Когда я писал простые респонсы в ответ на запрос пользователя, я использовал класс `HttpResponse` из `django.http`.
### метод через HttpResponse
Теперь мне нужно использовать не только его, но еще специальный метод, который будет "подгружать" шаблоны из папки, где они хранятся в текущую функцию.
```Python
from django.template import loader


def homepage(request):
	template = loader.get_template("index.html")
```
В качестве аргумента методу `get_template` передается путь внутри папки `templates`, которая указана в `DIR` константы `TEMPLATES` в `settings.py`.
Далее, я должен зарендерить полученный шаблон с помощью метода `render` и это можно сделать, обратившись к результату, который вернулся после работы функции `get_template` - она возвращает объект класса `Template`.
```Python
from django.http import HttpResponse
from django.template import loader


def homepage(request):
	template = loader.get_template("index.html")
	return HttpResponse(template.render({}, request))
```
На вход методу `render` передается словарь контекста + сам запрос пользователя. 
### метод через django.shortcuts
Выше я использовал функцию `get_template()`, чтобы достать шаблон из папки шаблонов + использовал довольно длинную запись через передачу зарендеренного шаблона в `HttpResponse`. Все это дело можно сократить, если использовать функцию `render()` из модуля `django.shortcuts`:
```Python
from django.shortcuts import render


def catalog(request):
	template = "catalog/catalog.html"
	return render(request, template)
```
## аргументы
Одной из основных фишек шаблонов являются аргументы. Наверное, всем было бы полезно, если бы мы могли создавать страницы HTML с какими-нибудь переменными значениями. Например, я бы не хотел писать карточки товаров индивидуально для каждого из ста товаров на сайте - я бы хотел придумать один шаблон, контент на котором немного меняется в зависимости от товара.
Я сделаю тестовый HTML:
```HTML
<!DOCTYPE html>
<html lang="ru">
  <head>
    <meta charset="utf-8">
    <title>Страница товара</title>
  </head>
  <body>
    <h1 style="color: blue">Страница товара {{ index }}</h1>
  </body>
</html>
```
Место, куда я буду вставлять свой аргумент в данном случае обозначено фигурными скобками, `index` - это имя аргумента, которое должно быть передано на рендеринг внутри словаря контекста.
Теперь я хочу переписать url в `urls.py`. Страницу товара я буду отдавать по номеру товара, поэтому мне нужно добавить конвертер пути, который будет проверять полученную ссылку на наличие числа в ней.
```Python
path("<int:index>/", views.catalog_item)
```
Теперь я пойду в `views.py` и пропишу там функцию `catalog_item`:
```Python
def catalog_item(request, index):
	template = loader.get_template("catalog/catalog_item.html")
	context = {"index": index}
	return HttpResponse(template.render(context, request))
```
### обращаемся к элементам последовательностей
Аргументы могут быть не плоскими, то есть я могу предать последовательность Python в качестве аргумента. Но вопрос: как обратиться к последовательности внутри шаблона? Через точечную нотацию:
```HTML
{{ items.key }}
{{ items.0 }}
{{ items.attribute }}
{{ items.method }}
```
Если мы передадим в словаре контекста словарь, то к его элементам можно будет обращаться через вызов ключа - это первый вариант синтаксиса `{{ items.key }}`.
Если мы хотим обратиться к элементу последовательности по индексу, то достаточно указать индекс через точку после вызова последовательности: `{{ items.0 }}`.
### работа с классами
Так же можно обратиться, например, к атрибуту класса. Чтобы это проверить я создал простой класс из именованного кортежа:
```Python
from collections import namedtuple
from django.templates import loader
from django.http import HttpResponse

def test_4(request)
	MyClass = namedtuple('MyClass', ['attr_1', 'attr_2'])
	a = MyClass('Привет!', 'Пока!')
	template = loader.get_template('test/test_4.html')
	context = {
		"item": a
	}
	return HttpResponse(template.render(context, request))
```
Когда я это сделал, можно пойти и составить шаблон:
```HTML
<h1>Мой класс говорит: {{ item.attr_1 }}</h1>
<h2>Мой класс говорит: {{ item.attr_2 }}</h2>
```
В результате на странице будут отображены именно атрибуты класса.
Вопрос: а могу ли я обратиться к методу класса, чтобы получить от этого метода значение?
Да. Чтобы это проверить я создал класс:
```Python
class MyClass:
	def __init__(self, name, index):
		self.__name = name
		self.__index = index
	@property
	def name(self):
		return self.__name
	@property
	def index(self):
		return self.__index
	@name.setter
	def name(self, value):
		if value:
			self.__name = value
	@name.setter
	def index(self, value):
		if type(value) is int and value >= 0:
			self.__index = value
```
Теперь я напишу функцию в `views.py`:
```Python
from django.db import connection
from django.http import HttpResponse
from django.template import loader


def test(request):
	template = loader.get_template("test/test.html")
	with connection.cursor() as cursor:
		cursor.execute(
			"select product_id, product_name from product"
		)
		items = [
			MyClass(item[1], item[0]) for item in cursor.fetchall()
		]
	context = {"items": items}
	return HttpResponse(template.render(context, request))
```
И теперь шаблон:
```HTML
{% extends "base.html" %}
{% block content %}
{% for item in items %}
  <h1>{{ item.index }}.{{ item.name }}</h1>
{% empty %}
  <h1>Пустая последовательность.</h1>
{% endfor %}
{% endblock %}
```
В итоге атрибуты экземпляров будут взяты при рендеринге из функций-свойств.
## франкенштейн из шаблонов
Как понятно я не долбоеб, чтобы прописывать `footer`, `header` для каждого шаблона отдельно. Я хочу, чтобы контент был написан для каждой страницы индивидуально (в соответствии с задачами страницы), но все прочие элементы HTML документа сохранялись. Например, я не хочу каждый раз делать шаблон, в котором все части кроме `<body></body>` являются копиями других шаблонов в моем проекте, - навигация в подвале и голове документа, стили, кодировка и проч.
Шаблонизаторы позволяют реализовать такую историю с помощью блоков `extends`, `include` и `block`. Основная идея в том, что мы создаем шаблон, который не будет сам по себе использоваться в роли страницы проекта, но он будет являться заготовкой для всех остальных HTML документов. Другие шаблоны просто должны унаследовать от него всю структуру и определить какие индивидуальные элементы должны быть в этой структуре у этой страницы.
### extends, block
Допустим я хочу сделать шаблон для всех страниц на своем сайте - для каждой страницы все элементы, кроме основного содержания в теге `main` будут одинаковыми. Эта история называется "базовый шаблон" и я его сделаю в директории `templates` своего проекта с названием `base.html`:
```HTML
<!DOCTYPE html>
<html lang="ru">
  <head>
    <meta charset="utf-8">
    <title>Страница</title>
  </head>
  <body>
    <header></header>
    <main>
    </main>
    <footer></footer>
  </body>
</html>
```
Вопрос: а как при рендеринге Django Template поймет куда ему вставлять уникальный контент?
С помощью тега `block`. Он обозначает место куда нужно вставить уникальный контент, который прописан внутри такого же тега на дочерней странице:
```HTML
<!DOCTYPE html>
<html lang="ru">
  <head>
    <meta charset="utf-8">
    <title>Страница</title>
  </head>
  <body>
    <header></header>
    <main>
      {% block content %}{% endblock %}
    </main>
	<footer></footer>
  </body>
</html>
```
`content` в данном случае просто имя блока, которое используется, как имя переменной, чтобы понимать куда вставлять значения.
Окей, допустим, но как дочерняя страница, например, страница моего каталога `catalog/catalog.html` поймет, что нужно использовать этот шаблон в качестве основы? 
Через тег `extends` - его нужно добавить в самое начало дочернего HTML документа:
```HTML
{% extends "base.html" %}
```
Теперь нужно просто прописать сам уникальный контент страницы. Чтобы шаблонизатор понял куда этот контент из дочерней страницы вставлять нужно обрамить его тегами `block` с соответствующим именем блока:
```HTML
{% extends "base.html" %}

{% block content %}
  <h1 style="color: red;">Эта страница каталога!</h1>
{% endblock %}
```
>[!important] Ставить кавычки в `extends "base.html"` обязательно, иначе шаблонизатор выкинет ошибку!
### include
Даже `base.html` можно разобрать на блоки. Например, я хочу разделить код "подвала" и "чердака" по разным файлам - пусть в `base.html` останется только основной скелет. Для меня это означает, что я смогу редактировать блоки кода в отдельных файлах вместо того, чтобы крутить один огромный основной шаблон.
Такую штуку позволяет провернуть тег `include`. Для начала я зайду в `base.html` и пропишу те части, которые хочу импортировать из отдельных файлов.
```HTML
<!DOCTYPE html>
<html lang="ru">
  <head>
    <meta charset="utf-8">
    <title>Страница</title>
  </head>
  <body>
    <header>{% iclude "includes/header.html" %}</header>
    <main>{% block content %}{% endblock %}</main>
	<footer>{% include "includes/footer.html" %}</footer>
  </body>
<html>
```
Как видно внутри `includes` я ссылаюсь на два файла `header.html` и `footer.html` - они то как раз и будут содержать весь контент, на который я ссылаюсь.
`header.html` выглядит так:
```HTML
<!-- header.html -->
<p style="font-size:20px;font-family:sans-serif;">
  Это чердак страницы
</p>
```
`footer.html` выглядит так:
```HTML
<!-- footer.html -->
<p style="font-size:20px;font-family:sans-serif;">
  Это подвал страницы
</p>
```
Все эти строки будут импортированы в базовый шаблон. Напоминаю - на базовый шаблон ссылаются все прочие страницы, а он может ссылаться на какие-то дочерние аргументы.
## управляющие конструкции
В шаблонизаторе Django Templates существует возможность вставить в шаблон управляющую конструкцию типа `if` и `for`. Чтобы конструкция работала как требуется нужно передать какие-то значения на вход функции рендеринга с помощью словаря контекста.
### if
Например, я хочу отдавать пользователю страницу, которая будет делать текст черным или красным в зависимости от аргумента, который был передан на вход функции рендеринга.
```Python
# вариант 1
return HttpResponse(template.render({'color': 'red'}, request))
# вариант 2
return HttpResponse(template.render({'color': 'black'}, request))
```
Тогда мне проще не оформлять два разных варианта шаблонов, а сделать некое подобие условной конструкции внутри одного шаблона. Такая функция существует:
```HTML
<p>
  {% if color == "red" %}
    <h1 style="color:red;">Привет!</h1>
  {% elif color == "black" %}
    <h1>Привет!</h1>
  {% endif %}
</p>
```
В механизме условной конструкции можно прописать любое поведение итогового шаблона - после рендеринга страница HTML будет выглядеть в инструментах разработчика точно так же, как если бы она была написана вручную.
### for
Если вдруг мне нужно задать какое-то поведение для последовательности, которую я передал на рендеринг внутри словаря контекста, то я могу использовать цикл `for`. Он работает точно так же, как перебор последовательности в Python, поэтому ничего сложного в его применении нет.
```Python
context = {"items": ["Картошка", "Яблоки", "Апельсины"]}
return HttpResponse(template.render(context, request))
```
Тогда в HTML шаблоне я могу написать так:
```HTML
<ul>
  {% for item in items %}
    <li><h3>{{ item }}</h3></li>
  {% endfor %}
</ul>
```
В итоге будет организован маркированный список на HTML странице. Для каждого элемента списка `items` будут проставлены теги `li` и `h3`.
### немного допов про for
>[!important] Цикл `for` может перебирать итерируемые объекты в Python, а не только списки. Это означает, что на вход шаблону можно передать даже генератор.

Внутри цикла доступны всякие фишки, вроде подсчета текущего шага цикла. Например, я могу обратиться к механизму итерации с вопросом "Последний ли это шаг?":
```HTML
{% if forloop.last %}
<!-- какие-то действия -->
{% endfif %}
```
"...или первый?":
```HTML
{% if forloop.first %}
<!-- какие-то действия -->
{% endif %}
```
>[!important] `forloop.first` и `forloop.last` возвращают булевское значение, которое можно использовать в управляющей конструкции `if`.

Я могу вызвать счетчик шага с помощью `forloop.counter` и `forloop.conter0`:
```
{% if forloop.counter == 1 %}
<!-- какие-то действия -->
{% elif forloop.counter0 == 0 %}
<!-- какие-то действия -->
```
В данном случае они обозначают одно и то же, просто в разных системах счета:
1) `forloop.counter` - это счетчик, который начинает отсчет с единицы.
2) `forloop.counter0` - это счетчик, который начинает отсчет с нуля.
>[!important] Все эти фишки доступны через переменную `forloop`. Там есть еще пара интересных штук, которые я не особо попробовал.

Так же поведение цикла можно настроить для случаев, когда ему передается пустая последовательность:
```HTML
{% for item in items %}
  <!-- код -->
{% empty %}
  <!-- код на случай пустой последовательности -->
{% endfor %}
```
## name
Мне нужно разместить на одной странице ссылки на несколько товаров. Как я это сделал бы сейчас?
```HTML
{% for item in items %}
  <a href="catalog/{{ item.index }}/">Товар {{ item.name }}.</a>
{% empty %}
  <h1 style="color:red;">Товаров нет.</h1>
{% endfor %}
```
Вроде бы как удобно: если пользователь кликнет на ссылку, то на сервер отправится запрос к ресурсу, например, `/catalog/1/`. Запрос попадет в маршрутизатор и совпадет вот с таким маршрутом:
```Python
path('catalog/<int:index>/', views.item_page)
```
А что если мне нужно разместить не несколько единообразных ссылок, а кучу ссылок на разные ресурсы. Тогда в шаблоне я буду прописывать разные адреса для этих ресурсов:
```HTML
<a href="/">Главная страница сайта</a>
<a href="/catalog/">Главная страница каталога</a>
<a href="/catalog/details/">Справка о каталоге</a>
```
Если любой из этих ресурсов поменяет свой маршрут в файле `urls.py`, то придется так же искать все места в шаблонах, где этот маршрут был указан, чтобы изменить и его. Как понятно, при большом проекте - это реально тяжело сделать.
Поэтому в Django есть возможность разместить дополнительный параметр `name` в функцию `path`:
```Python
path('catalog/', views.catalog_main, name='catalog_main'),
path('catalog/<int:index>/', views.item_page, name='item_page')
```
Теперь внутри шаблона можно вставлять не ссылки на ресурсы, а ссылки на маршруты:
```HTML
<a href="{% url 'catalog_main' %}">Страница каталога</a>
<a href="{% url 'item_page' <аргументы> %}">Страница предмета из каталога</a>
```
### namespace
Это дополнение к фишке с `name`. В проекте может существовать много ресурсов, поэтому в какой-то момент наверняка закончатся вакантные уникальные имена для каждого маршрута в `path`. Эту проблему решает `namespace` - для каждого приложения можно сделать отдельный `namespace`, в рамках которого имена должны быть уникальными, но при этом в разных `namespace` могут быть одни и те же имена.
Чтобы указать `namespace` нужно прописать переменную `app_name` в файле `urls.py`:
```Python
app_name = 'catalog'
...
path('catalog/', views.catalog_main, name='catalog_main')
```
Тогда в HTML доке ссылку можно будет дать так:
```HTML
<a href="{% url 'catalog:catalog_main' %}">
  Страница каталога
</a>
```
## параметры переменных
Данные, которые мы передаем в словаре контекста можно дополнительно обработать в самом шаблоне. Делается это с помощью фильтров прямо в шаблоне.
### длинна строки
Например, я передаю в словаре контекста какую-то длинную строку:
```Python
context = {"long_string": 'asgsdfgsdfgdsfgsdfgsdfgsdfgsdfgsfdg'}
```
А на странице я хочу видеть цифру, которая будет отображать количество символов в этой троке. Чтобы это реализовать нужно применить такой синтаксис:
```HTML
{{ long_string|length }}
```
При рендеринге эта конструкция заменится цифрой количества символов в строке `long_string`.
В общем случае синтаксис фильтрации всегда такой:
```HTML
{{ variable|параметры }}
```
### safe
Это параметр, который используется, чтобы превратить строку, переданную на вход в рабочий кусок HTML:
```Python
html_string = '''
	<h1>Мой заголовок<br>с разрывом</h1>
'''
context = {
	"save_try":html_string
}
```
Теперь шаблон:
```HTML
{{ save_try }}
{{ save_try|safe }}
```
На страницу будет выведено следующее:
![[safe_usage.PNG]]
Лучше НИКОГДА не использовать параметр `safe`, потому что он обеспечивает уязвимость ресурса - если в данный аргумент на рендеринг попадет не кусок кода разработчика, а кусок кода злоумышленника, то на странице будет размещен и выполнен скрипт, который может быть очень опасным. Например, внутри HTML записи может быть размещен скрипт JS.
### linebreaksbr
Между языками Python и HTML временами происходят разногласия, например, в области разрыва строк. Я могу отправить в словарь контекста строку:
```Python
my_string = "Моя строка с\nразрывом"
context = {"broken_str":my_string}
```
Теперь HTML шаблон:
```HTML
<h1>{{ broken_str }}</h1>
```
На странице получится такое:
![[broken_string.png]]
Разрыва нет. Потому что `\n` не переводится при рендеринге в HTML тег `<br>`. Чтобы исправить эту ситуацию шаблонизатору нужно указать на то, что в строке есть имеющиеся разрывы строк `\n`, которые необходимо перевести в HTML. Это делает параметр `linebreaksbr`:
```HTML
<h1>{{ broken_str|linebreaksbr }}</h1>
```
Получится так:
![[right_broken_string.PNG]]
### работа с датой
Если передать в словарь контекста дату в формате модуля `datetime`:
```Python
from datetime import datetime
context = {
	"gogol_bd":datetime(1809, 4, 1)
}
```
А потом написать ее в HTML шаблоне в таком виде:
```HTML
{{ gogol_bd }}
```
То получится не самая красивая запись:
```HTML
April 1, 1809, midnight
```
Такое поведение можно изменить, если применить параметр работы с датой в Django Templates - `date`. Чтобы это сделать нужно передать на вход `date` какой-нибудь шаблон для форматирования даты:
```
{{ gogol_bd|date:"d.m.y" }}
{{ gogol_bd|date:"j.n.Y" }}
{{ gogol_bd|date:"d M Y" }}
```
В итоге получится такая картина:
![[datetime_django_templates.png]]
### допы про параметры + еще параметры
В рамках одного использования переменной в шаблоне можно применить несколько параметров. Например:
```
{{ armstrong_said|title|truncatewords:4 }}
```
Параметры:
1) `title` делает первую букву каждого слова в строке заглавной.
2) `truncatewords:4` обрезает строку до первых четырех слов (включительно).
>[!important] Важно помнить, что строка передается из параметра в параметр слева на право, т.е. сначала строка будет преобразована параметром `title`, затем результат этого преобразования попадет в параметр `truncatewords:4`.
## подключение статических файлов
Статические файлы - это файлы, которые никак не обрабатываются на сервере, а просто отправляются пользователю в ответ на его запрос.
В статические файлы включают:
1) файлы СSS - описание стилей
2) файлы JS - исполняемый код, который запускается не на сервере, а прямо в браузере пользователя.
3) файлы изображений (логотип, картинки иконок, декоративные изображения для оформления страниц).
4) файлы шрифтов: пользователю загружаются шрифты, которые захотел применить дизайнер, но они не установлены на комп пользователя.
В Django за работу со статикой отвечает собственное приложение - `django.contrib.staticfiles`. Оно при развертывании проекта записывается в `INSTALLED_APPS`:
```Python
INSTALLED_APPS = [
    'homepage.apps.HomepageConfig',
    'catalog.apps.CatalogConfig',
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',  # тут.
]
```
По умолчанию Django ожидает, что статические файлы будут лежать в папках с названием `static/` внутри приложений, т.к. для джанги автономность приложений - это все.
Чтобы переопределить это поведение нужно в настройках проекта указать значение для переменной `STATICFILES_DIRS`. В качестве значения ей передается список, где прописываются пути для папок с статическими файлами.
Я сделаю в этом случае так же, как делал с шаблонами - размещу все файлы в одной папке `static`. Она будет храниться в главной папке проекта, рядом с `manage.py`. Чтобы Django о ней знал:
```Python
STATIC_URL = '/static/'
STATIC_DIR = BASE_DIR / 'static'
STATICFILES_DIRS = [
    STATIC_DIR
]
```
Чтобы подгрузить статику в шаблон нужно добавить в шаблон конструкцию `{% load static %}`. Она не наследуется, поэтому в каждом шаблоне, где будет использована статика, нужно ее прописывать.
Чтобы подгрузить файл CSS нужно в шаблон добавить стандартную ссылку:
```HTML
<link rel="stylesheet" href="{% static 'css/my_css.css' %}"
```
И после этого можно ссылаться на классы из CSS-файла.
Чтобы добавить картинку нужно точно так же прописать путь до нее в файлах статики:
```HTML
<img src="{% static 'img/logo.png' %}">
```
