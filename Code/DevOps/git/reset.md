# soft
Допустим, что я хочу изменить последний коммит `new words`. Чтобы это сделать мне нужно откатить репозиторий к состоянию коммита `initial commit`, но сохранить все изменения, которые я внес в ходе работы над коммитом `new words`. В этом мне поможет:
```Bash
git reset --soft HEAD~1
```
Что тут:
1) `git reset` - команда откатывает состояние репозитория и индекса к какому-то коммиту.
2) `--soft` - "мягкий" откат означает, что изменения останутся в индексе (не нужно будет по-новой добавлять изменения с помощью `git add`).
3) `HEAD~1` - указание куда мне нужно откатываться. В данном случае я хочу откатиться на предыдущий коммит.
Что в сущности означает "изменения останутся в индексе"?
Допустим у меня есть два файла: 
```Bash
echo "Hello_1" > file_1.txt && \
echo "Hello_1" > file_2.txt
git add .
git commit -m "Init 2 files"
git push
```
Теперь я внесу изменения в оба:
```Bash
echo "Hello_2" >> file_1.txt && \
echo "Hello_2" >> file_2.txt
```
А в индекс добавлю только один и закоммичу изменения:
```Bash
git add file_1.txt
git commit -m "обновлен файл file_1.txt"
```
Нужно проверить статус:
```Bash
git status
```
А он покажет:
![[git_status_1.png]]
Он покажет, что моя ветка на один коммит дальше, чем ветка на удаленном репозитории + что у меня есть изменения `file_2.txt`, которые не внесены в индекс и не закоммичены.
Чтобы проверить, что параметр `--soft` при откате к предыдущему коммиту реально оставляет все изменения в индексе, мне нужно использовать его сейчас и снова глянуть в статус, но перед этим я посмотрю в лог (список коммитов):
```Bash
git log
```
Что в нем?
![[git_log_2.png]]
А в нем видно, что мой локальный репозиторий находится на коммите `обновил file_1.txt`, но так как я этот коммит еще не запушил - мой удаленный репозиторий все еще остался на `new files`. Теперь я откачусь как раз на коммит `new files`, но с сохранением всех изменений в индексе `--soft`:
```Bash
git reset --soft HEAD~1
git status
```
Вывод будет таким:
![[git_status_2.png]]
Здесь показано, что при мягком откате `file_1.txt` не вернулся в прошлое состояние - он все еще остался в индексе и его статус отличается от статуса `file_2.txt`. То есть, чтобы вернуть `file_1.txt` обратно в закоммиченое состояние, мне нужно просто сделать команду `git commit` снова. 
А вот для `file_2.txt` ситуация другая - он даже не бывал в индексе, поэтому, чтобы принять его изменения, нужно сначала внести их в индекс через `git add`.
Что такое `use "git restore --staged <file>..." to unstage`? Это Git предлагает вынести `file_1.txt` из индекса (при этом все изменения в файле сохранятся - он просто будет вынесен из индекса).
Допустим я передумал делать прошлый коммит и хочу восстановить все файлы:
```Bash
git restore --staged file_1.txt  # убираю из индекса файл
git restore file_1.txt file_2.txt
```
Вторая команда восстановит файлы в том виде, в котором они были в самом начале коммита `new files`.
# mixed
Это режим, который команда `git reset` использует по умолчанию. Он откатывает события к предыдущему коммиту + убирает все, что было добавлено в индекс из индекса.
Делаю тот же эксперимент, что и прежде - обновляю файлы:
```Bash
echo "hello_2" >> file_1.txt && \
echo "hello_2" >> file_2.txt
```
Опять внесу один файл, закоммичу и посмотрю статус:
```Bash
git add file_1.txt
git commit -m "обновил file_1.txt"
git status
```
Вывод:
![[git_status_3.png]]
Отсюда видно, что файл `file_2.txt` опять требует внесения в индекс, а `file_1.txt` не требует, т.к. он уже в нем + в коммите.
Теперь я применю обычный:
```Bash
git reset HEAD~1
# ИЛИ
git reset --mixed HEAD~1
```
Я написал `ИЛИ`, потому что это эквивалентные команды.
Что теперь выведет `git status`?
![[git_status_4.png]]
Он покажет, что у меня оба файла теперь находятся вне индекса и вне коммита. И даже не нужно лишний раз применять `git restore --staged`.
# hard
Параметр `--hard` тупо стирает все. Если его применить, то пропадут все изменения, так будто их вообще не было и состояние репозитория вернется к состоянию, которое было на момент указанного коммита:
```Bash
echo "hello_2" >> file_1.txt && \
echo "hello_2" >> file_2.txt
git add file_1.txt
git commit -m "обновил file_1.txt"
git status
```
Вывод `git status`:
![[git_status_3.png]]
Теперь я применю:
```Bash
git reset --hard HEAD~1
git status
```
И что случиться? Ничего - мы просто вернемся на коммит назад, а все изменения в любой форме будут стерты:
![[git_status_5.png]]
