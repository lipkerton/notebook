Такая ошибка возникает у меня при попытке сделать `pull` и скачать изменения с удаленного сервера. Выглядит она так:
```
hint: You have divergent branches and need to specify how to reconcile them.
hint: You can do so by running one of the following commands sometime before
hint: your next pull:
hint:
hint:   git config pull.rebase false  # merge
hint:   git config pull.rebase true   # rebase
hint:   git config pull.ff only       # fast-forward only
hint:
hint: You can replace "git config" with "git config --global" to set a default
hint: preference for all repositories. You can also pass --rebase, --no-rebase,
hint: or --ff-only on the command line to override the configured default per
hint: invocation.
fatal: Need to specify how to reconcile divergent branches.
```
У гита есть две истории коммитов - одна удаленная, а другая развернута локально. Такая ошибка возникает, когда две истории не могут быть автоматически совмещены в одну на удаленном репозитории.
Например, у меня локально существуют коммиты `A, B, C`, а в удаленный репозиторий были залиты коммиты `Z, X, C`, сразу после коммита `A`, то есть:
- удаленно: `A -> Z -> X -> C`
- у меня: `A -> B -> C`
Как совместить эти вещи?
В этом случае гит предлагает три варианта:
1) `git config pull.rebase false` - гит предлагает внести в конфигурацию поведение по умолчанию `merge`. Это означает, что две ветки будут слиты, а история того, что разработка шла параллельно будет сохранена.
2) `git config pull.rebase true` - гит предлагает внести в конфигурацию поведение по умолчанию `rebase`. Это означает, что удаленные коммиты `Z, X, C`  будут записаны поверх тех, что есть у меня так, как будто они были написаны после моих. В этом случае, в отличие от предыдущего, сохранится единая история изменений.
3) `git config pull.ff only` - гит предлагает внести в конфигурацию поведение по умолчанию `fast-forward`. Этот вариант будет работать только в том случае, когда я не делал вообще никаких коммитов, т.е. когда моя ветка просто немного отстала от удаленной. Самый безопасный вариант обновления и в этой ситуации он не прокатит.
Я не хочу применять никакие изменения к конфигурации, поэтому я просто добавлю дополнительный аргумент к `pull`, чтобы использовать стратегию №2 один раз:
```bash
git pull --rebase
```
Чтобы применить стратегии №1, №2 в таком же формате (на один раз):
```bash
git pull --no-rebase
git pull --ff-only
```
