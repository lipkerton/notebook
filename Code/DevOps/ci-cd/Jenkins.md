# установка
Центральный механизм системы - *jenkins-controller*. При первичной установке Jenkins устанавливается в первую очередь именно он. Ссылка: https://www.jenkins.io/doc/book/installing/linux/#debianubuntu.
При установке на Debian у меня возникла проблема с версией Java JDK, которая нужна для работы Jenkins. В инструкции сказано, что нужно установить:

Однако, видимо из-за того, что у меня 12-я версия Debian, пакета `openjdk-21-jre` в моем пакетном менеджере нет:

Однако существуют решения проблемы отсутствия пакетов: https://www.reddit.com/r/linux4noobs/comments/1e1x48k/how_can_i_install_java_21_on_debian_12_without/

```Bash
sudo journalctl -u jenkins.service
```

# Jenkinsfile
`Jenkinsfile` - это файл, который написан на языке `Groovy`. Он размещается в корне репозитория и именно его ищет jenkins-controller, чтобы запустить процесс сборки проекта. Внутри `Jenkinsfile` описываются шаги, которые нужно сделать с проектом в рамках пайплайна.
Самый простой `Jenkinsfile` выглядит так:
```Groovy
pipeline {
    agent any
    stages {
        stage('hello') {
            steps {
		    	echo 'hello world!'
            }
        }
    }
}
```
Вся логика разворачивается внутри тега `pipeline`.
## agent
*jenkins-agent* - это система, которая будет выполнять все шаги, которые указаны далее. В качестве агента может выступать отдельный сервер или отдельный docker-контейнер, где jenkins-controller будет собирать и тестировать проекты.
В примере выше проставлен тег `any` - в качестве агента будет выступать любая доступная в рамках jenkins-controller сущность. Посмотреть список доступных можно в настройках Jenkins в `Nodes`:
![[jenkins_pipeline_10.png]]
У меня сейчас есть два агента - один мой, а другой выдан по умолчанию при установке Jenkins. Скорее всего, когда я использую `any` действует именно агент по умолчанию.
Проще всего создать агента можно через Docker: https://www.jenkins.io/doc/book/using/using-agents/#configuring-agents-with-docker
Официальная документация советует разделять *jenkins-controller* и всех агентов, которыми он управляет. Однако я не ищу легких путей, поэтому постарался настроить агeнта в той же системе, где развернут *jenkins-controller*, + постарался развернуть его без помощи Docker.
Для этого нужно в настройках агента (при добавлении нового агента) сменить `Способ запуска` на `Запустить агент, подключив его к контроллеру`. При таком подходе мы избегаем лишних сложностей с настройкой SSH между агентом и контроллером.
После создания агента такого типа, в его интерфейсе появятся подсказки - это команды, которые необходимо ввести из места, где мы планируем развернуть агента:
![[jenkins_pipeline_11.png]]
Агент станет доступен сразу после развертывания.
Важно понимать, что с таким агентом появляются определенные проблемы, т.к. формально он может управлять всей системой, в которой развернут (т.е. куда дотянутся его права).
Однако есть и определенные преимущества, например, мне не нужно устанавливать в него Docker и проч. (как мне нужно было бы делать, если агент был бы развернут в контейнере), т.к. они все уже установлены в системе, где я развернул агента и агент может ими пользоваться. Однако он будет пользоваться ими от лица пользователя `jenkins`, следовательно, у этого пользователя должны быть права на выполнение некоторых действий.
Например, я хочу, чтобы мой `Jenkinsfile` выполнял сборку контейнера Docker. В этом случае мне нужно выдать пользователю `jenkins` права на запуск контейнеров - добавить его в группу `docker`.
Теперь мне нужно просто передать имя моего агента в `Jenkinsfile`:
```Groovy
pipeline {
    agent { label 'debian' }
    stages {
        stage('hello') {
            steps {
		    	echo 'hello world!'
            }
        }
    }
}
```
И пайплайн запустится прямо на нем.
