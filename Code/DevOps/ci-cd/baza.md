**CI (Continuous Integration)** - постоянная интеграция. Подразумевает интеграцию нового кода в проект, например, в проект на GitHub + проверку тестами проекта с обновленным кодом на работоспособность.
**CD (Continuous Delivery)** - постоянная доставка. Подразумевает полноценный деплой протестированного кода в продакшн.
**CD (Continuous Depoloyment)** - постоянное развертывание. Второй вариант трактовки аббревиатуры подразумевает автоматическое развертывание проекта после его проверки.
Механизм CI-CD подразумевает, что при любом пуше в удаленный репозиторий готового кода будет происходить процесс проверки этого кода, а после - процесс деплоя нового кода на "боевые сервера".
На практике термин описывает довольно размытый круг вещей. Например, проверки кода могут происходить не при каждом пуше, а только при каждом пуше в ветку для которой уже сделан MR (merge request) и т.д. Но общая схема всегда примерно одна - термин "CI-CD" описывает процесс автоматических проверок и деплоя нового кода.
**pipline** - это термин, который обозначает автоматический "конвеер" между разработчиком и задеплоенным кодом. Пайплайн представляет собой жесткую последовательность шагов, которая будет выполнятся для любого разработчика в этом проекте.
Процедура CI/CD востребована, поэтому для ее реализации существуют свои сервисы. Например, у GitLab есть такой и у GitHub есть такой. Сервис от GitLab, как и сам GitLab можно развернуть локально - в этом его большое преимущество. Однако в плане CI/CD я предпочитаю выбрать Jenkins.
