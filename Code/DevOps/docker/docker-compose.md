Допустим у меня есть приложение на FastAPI, но еще я хочу докрутить к этому приложению БД. Не очень удобно заранее разворачивать БД на целевом сервере, потому что это делает все приложение не самым мобильным.
Лучше, чтобы БД сама по себе являлась контейнером, который так же, как и мое приложение, можно развернуть. Не трудно заметить, что проблема мобильности все еще остается - да, теперь не нужно запариваться с перемещением БД с системы на систему (из-за контейнера), но перемещать два контейнера отдельно друг от друга все еще не очень круто.
Чтобы справиться с трудностями была придумана **оркестрация** - автоматическое управление сразу несколькими контейнерами одновременно из одного файла.
Этот один файл называется `docker-compose.yml`. Он пишется в формате `YAML`.
# начало
Я начал с того, что сделал стандартный `docker-compose.yml` для моего приложения на FastAPI:
```yaml
services:
  blg_1:
    build: .
    networks:
      - blg_1_net
    ports:
      - "8000:8000"
```
Подробнее:
- `services` - это тег, который содержит набор контейнеров. Эти контейнеры мы и должны запускать.
- `blg_1` - имя первого контейнера (мое приложение).
- `build` - указатель на место, где лежит файл `Dockerfile`.
- `networks` - сеть контейнера (расскажу позже).
- `ports` - все точно так же как в `Dockerfile`: первый порт - это порт для обращения с хоста, а второй - это порт внутри контейнера.
После того как файл создан можно сразу "поднять его":
```Bash
docker compose up
```
Пойдет стандартный процесс запуска контейнера.
# добавляю БД
Теперь я хочу добавить второй контейнер - мою базу данных. Так как PostgresSQL является стандартным и популярным приложением, то для него уже существует готовый образ, на основании которого Docker может сделать контейнер. Логика такая, что мы позаимствуем этот образ и на его основе соберем для себя новенькую базу.
```yaml
services:
  ...
  database:
    image: postgres:latest
    container_name: postgres_blg_1
    environment:
      POSTGRES_USER: ${POSTGRES_DB_USER}
      POSTGRES_PASSWORD: ${POSTGRES_DB_PASS}
      POSTGRES_DB: ${POSTGRES_DB_NAME}
    networks:
      - blg_1_net
    ports:
      - "127.0.0.1:${POSTGRES_DB_PORT_HOST}:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_DB_USER} -d ${POSTGRES_DB_NAME}"]
      interval: 30s
      timeout: 10s
      retries: 5
```
Подробнее:
- `database` - название сервиса (такое же, как и `blg_1`).
- `image` - имя образа, который мы заберем с реестра Docker.
- `container_name` - имя контейнера.
>[!important] `container_name` так же будет являться именем хоста в ссылке на обращение к БД!!!
- `environment` - здесь прописываются данные для аутентификации в БД.
	- Конструкция `&{<что-то>}` - это обращение к константам в файле `.env` (он лежит в той же директории, что и `docker-compose.yml`).
- `networks` - контейнер будет принадлежать к той же сети, что и приложение на FastAPI.
- `ports` - здесь указываются порты для коннекта в БД. Как обычно - сначала порт по которому БД доступна с хоста, а потом порт, на котором слушает БД внутри контейнера.
- `healthcheck` - это по сути функция, которая дергает БД и проверяет ее на готовность с помощью командной строки внутри контейнера.
После того как БД добавлена мне еще нужно "присоединить" этот контейнер к предыдущему контейнеру, чтобы они запускались в определенной последовательности.
Это поможет сделать параметр `depends_on`:
```yaml
services:
  blg_1:
    depends_on:
      database:
        condition: service_healthy
    build: .
    networks:
      - blg_1_net
    ports:
      - "8000:8000"
```
Теперь при строительстве контейнеров сначала сделается контейнер с БД, затем проверится его доступность (`condition`) и только потом начнет строится контейнер с приложением.
По сути это все.
# volumes
Контейнер хранит данные только внутри своей сессии. Это означает, что при отключении контейнера все данные пропадут. Для решения этой проблемы для любого сервиса в списке сервисов так же можно добавить "хранилище" - `volumes`. Этот параметр определяет какую часть данных докер может сохранить даже в случае удаления контейнера.
В моем примере с БД эта фича кажется очень полезной, потому что по сути мы создаем хранилище данных, которое не зависит от состояния контейнера с БД.
`volume` (том) - это сущность, равная по статусу `image` (образу) и `container` (контейнеру), поэтому посмотреть список `volumes` в докере можно так:
```Bash
docker volume ls
```
Теперь я добавлю к нашему контейнеру с БД новый том:
```YAML
services:
...
database:
	image: postgres:latest
	container_name: postgres_blg_1
	environment:
	POSTGRES_USER: ${POSTGRES_DB_USER}
	POSTGRES_PASSWORD: ${POSTGRES_DB_PASS}
	POSTGRES_DB: ${POSTGRES_DB_NAME}
	networks:
	- blg_1_net
	volumes:  # добавляю том
		- postgres_data:/var/lib/postgresql/data
	ports:
	- "127.0.0.1:${POSTGRES_DB_PORT_HOST}:5432"
	healthcheck:
		test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_DB_USER} -d ${POSTGRES_DB_NAME}"]
		interval: 30s
		timeout: 10s
		retries: 5

volumes:  # регистрирую том
	postgres_data:
```
Запись тома строится следующим образом:
1) `postgres_data` - имя тома.
2) `/var/lib/postgresql/data` - это директория, которую том будет хранить.
За актуализацией и поддержкой данных в томе следит Docker. Это означает, что данные могут поддерживаться в актуальном состоянии отдельным от контейнера сервисом. Так мы сохраним информацию в БД, даже если контейнер вдруг остановит работу.
# как спрятать `.env`?
Файл `.env` в моем проекте нужен был мне сразу для того чтобы заполнить секретные значения в `docker-compose` + чтобы использовать переменные окружения в самом FastAPI приложении. Учитывая, что я хотел опубликовать образ, на основании которого я буду делать контейнер, мне нужен был способ убрать из образа `.env` секреты.
Убрать их не сложно - нужно просто добавить строчку в `.dockerignore`. Но как после этого сообщить приложению внутри контейнера нужные ключи из `.env`?
Чтобы это сделать нужно добавить в сервисы, которым нужен `.env`, строку, которая указывает на местонахождение `.env` на хосте:
```YAML
services:
  blg_1:
    image: lipkerton/blog_1:v1.1
    depends_on:
      database:
        condition: service_healthy
    networks:
      - blg_1_net
    ports:
      - "8000:8000"
    env_file: ".env"  # тут.
```
Как проверить?
Когда `.env` не находился в файле `.dockerignore`, то его можно найти и прочитать в итоговом контейнере с помощью:
```
docker exec <имя контейнера> cat .env
```
Когда он в `.dockerignore`, то такой фокус больше не прокатит - файл не добавится в образ при создании => его нельзя будет найти. Однако я воспользовался параметром `env_file` - это значит, что локальный файл `.env` будет доступен внутри контейнера.
