Докер - это такая штука, которая была придумана, чтобы решить проблему транспортировки целого приложения с одной системы на другую.
Например, я сделал приложение на Ubuntu и хочу перенести его без проблем на какой-нибудь Arch. Для этого отлично подойдет докер.
Докер запаковывает приложение в контейнер + он делает слепок системы, на которой приложение работало и кладет эту систему так же в контейнер. Этот контейнер мы можем передать на другую систему и "активировать его" = внутри контейнера развернется система и наше приложение. Удобно, круто, классно, а развернуть можно везде.
# компоненты
Вокруг докер-контейнера крутится несколько важных понятий:
- демон
- образ
- контейнер
Под "демоном" имеется в виду программа докера, которая постоянно должна работать в фоновом режиме в системе, чтобы управлять контейнерами.
**Образ (*image*)** - это что-то вроде инструкции для сборки контейнера. Всегда перед тем как создать контейнер нужно создать образ (или скачать готовый образ с открытого реестра образов). Образ является пассивным организмом, т.е. нельзя "запустить", "эксплуатировать" образ. Его можно использовать **только** для создания контейнера.
**Контейнер (*container*)** - активная часть докера. Контейнер создается из образа, может быть запущен, перезапущен, удален и т.д.
# создать контейнер
## Dockerfile
Самым первым шагом в процессе создания контейнера является создание `Dockerfile`. Это файл-инструкция для создания образа.
Допустим, у меня есть какой-то проект на Python, который я хочу перенести на свой удаленный сервер. Как мне запаковать этот проект в `Dockerfile`?
Я создам в корне проекта `Dockerfile` и пропишу туда:
```Dockerfile
FROM python:3.12
WORKDIR /test
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["fastapi", "run", "main.py", "--port", "80"]
```
Что тут происходит?
1) `FROM python:3.12` - внутри контейнера у меня будет работать питоновский код. Т.к. контейнер по сути является отдельной системой, то внутри этой системы мне понадобится питон. Поэтому данная строка просит докер поставить питон 12-й версии внутрь контейнера - докер возьмет версию из собственного реестра и развернет внутри.
2) `WORKDIR /test` - это указатель на папку с программой **внутри** контейнера.
3) `COPY requirements.txt ./` - здесь я копирую файл `requirements.txt` в рабочую папку на контейнере.
4) `RUN pip install --no-cache-dir -r requirements.txt` - запускаем установку зависимостей внутри контейнера.
5) `COPY . .` - копируем содержимое корня проекта на нашем компьютере в рабочую папку на контейнере.
6) `CMD ["fastapi", "run", "main.py", "--port", "80"]` - прописываю команду для командной строки внутри контейнера. По сути это команда: `fastapi run main.py --port 80`, только она написана в формате, который необходим для `Dockerfile`.
Каждая команда в `Dockerfile` представляет из себя уровень. Уровни кэшируются докером, чтобы, при повторном построении контейнера, стройка не заняла так много времени. А еще уровни таким образом не зависят друг от друга - изменения одного уровня не требуют создания нового образа.
В основе каждого образа лежит базовый образ - образ операционной системы. Образ операционной системы берется тупо из той ОС, где я произвожу настройку, если не указан другой тип системы.
## образ
Чтобы построить образ на основании `Dockerfile` нужно ввести команду:
```Bash
docker build -t test123 .
```
Подробнее:
- `build` - собираем образ.
- `-t` - параметр позволяет ввести имя для образа.
- Точка в конце указывает на место, где нужно искать `Dockerfile`. В данном случае имеется в виду текущая директория.
После команды произойдет процесс сборки образа. А затем результат можно будет проверить так:
```Bash
docker image ls
```
Удалить получившийся образ можно с помощью его имени:
```Bash
docker image rm test123
```
Что будет если я попытаюсь внести изменения в образ? По идее докер должен закешировать уровни и при изменении одного использовать кэши для всех других. Логика правильная, но на практике не совсем так: докер будет использовать кэши только тех уровней, которые следовали **до** измененного уровня.
Например:
```Dockerfile
FROM python:3.12
WORKDIR /test
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY . ./work  # поменял здесь.
CMD ["fastapi", "run", "main.py", "--port", "80"]
```
Все команды до измененной будут идти через кэш, а команды после нет.
Я это понял, когда поменял месторасположение рабочую директорию в контейнере:
```Dockerfile
FROM python:3.12
WORKDIR /work  # вот тут.
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["fastapi", "run", "main.py", "--port", "80"]
```
Зависимости начали устанавливаться снова.
## контейнер
Итак у меня есть образ. Теперь пора собрать из него рабочий контейнер.
Так как в `Dockerfile` указано, что наш контейнер должен запускать сервер FastAPI, то на выходе после запуска контейнера у нас должен запуститься именно такой сервер.
Контейнер создается командой `run`:
```bash
docker run --name test123 -p 80:80 test123
```
Подробнее:
- `--name` - имя будущего контейнера.
- `-p` - указываем `<номер порта на основном компьютере>:<имя порта внутри контейнера>`. То есть наши обращения к порту 80 будут переадресовываться на 80 порт внутри контейнера.
- В конце имя образа, на основе которого будет создан контейнер.
После того как контейнер создан демон будет поддерживать его работу, т.е. наш сервер будет работать в фоновом режиме. Чтобы остановить работу контейнера нужно:
```Bash
docker stop test123
```
Чтобы снова его запустить:
```Bash
docker start test123
```
Сразу после команды `run` высвечиваются логи, которые можно потерять, после комбинации `Control + C`. Чтобы их снова найти:
```Bash
docker logs test123
```
Что более важно - нам теперь доступны обращения к нашему серверу внутри контейнера через интерфейс `localhost` и `127.0.0.1`:
```Bash
curl http://localhost
```
## немного допов
Так как контейнер по сути является отдельной системой внутри системы (= "гостевая система"), у него есть свои собственные порты.
Именно поэтому при инициализации контейнера можно указать внешний порт, обращения к которому будут переадресованы в конкретный внутренний порт контейнера.
В примерах выше я использовал одинаковые порты, но я могу остановить контейнер, удалить его и запустить заново с другой связкой портов:
```Bash
docker stop test123
docker container rm test123
docker run --name test123 -p 8000:80 test123
```
Теперь извне контейнера нужно писать запросы к интерфейсам `localhost:8000` или `127.0.0.1:8000`. Внутри контейнера они будут маршрутизироваться на порт 80.
# подробнее про команды
`WORKDIR` - это команда, которая похожа по смыслу на `cd`. Мы указываем для нее в качестве параметра директорию, а команда сразу переходит в эту директорию. Если такой директории нет, то она будет создана.
`COPY` - это команда, которая копирует файлы из точки A в точку B. A - это директория, где хранится `Dockerfile` на хосте, а B - это директория внутри образа. Если целевой директории в образе нет, то `COPY` создаст ее.
Когда в `Dockerfile` прописано что-то такое:
```Dockerfile
WORKDIR /app
COPY . .
```
То это означает, что Docker 1) создаст папку `/app`, затем перейдет в нее (команда `WORKDIR`) 2) скопирует содержимое директории с `Dockerfile` с хоста и вставит в папку `/app`.
*Вопрос*: почему инструкция `COPY` в моем `Dockerfile` применяется дважды - один раз для всех файлов и второй раз только для файла `requirements.txt`? *Ответ*: это как раз сделано потому, что у `Dockerfile` многослойная структура.
`RUN` - это команда, которая выполняет терминальные команды внутри контейнера. **Она выполняется только при сборке контейнера!**
`CMD` - это команда, которая точно так же выполняет терминальные команды, как и `RUN`, но она делает это **при запуске контейнера**.
