это заметка-конспект по книжке *operating systems: three easy systems*. по главе *introduction to operating systems*.
запуск программы = выполнение инструкций. миллионов и сейчас уже миллиардов инструкций в секунду.
процессор *цепляет (fetches)* инструкцию из памяти, *декодирует (decodes)* инструкцию (т.е. пытается понять какая именно это инструкция) и *исполняет (executes)* ее.
я сначала не понял зачем процессор декодирует инструкцию, если на предыдущем шаге он ее вытягивает из памяти. на самом деле, похоже, что процессор вытягивает код инструкции, а не саму инструкцию, поэтому ему еще нужно декодировать этот код.
механизм *fetch-decode-execute* звучит просто, но на самом деле под капотом вместе со всеми этими процессами происходит еще куча всего, чтобы сделать систему *easy-to-use*. существует тело ПО (*body of software*), которое ответственно, например, за запуск нескольких программ одновременно, за распределение памяти для нескольких программ, за связь между программами и устройствами и т.д. *the body of software* = *operating system (OS)*
главный инструмент, которым пользуется OS для реализации *easy-to-use system*, - **виртуализация (virtualization)**. в общем случае виртуализация - это когда OS берет физический ресурс в системе (процессор, память или диск) и превращает его в более общую (general), мощную (powerful), easy-to-use виртуальную форму самого себя. поэтому иногда мы обращаемся к оперативной системе, как к **виртуальной машине (virtual machine)**.
чтобы дать пользователям возможность пользоваться оперативной системой виртуальной машины (запускать программы, отдавать команды и т.д.) так же реализуются *интерфейсы (APIs)*. пользователи обращаются к интерфейсам (*to make a call*) - пользователь управляет программой, программа обращается к интерфейсу, чтобы создать **системный вызов (system call)**.
>[!important] Набор системных вызовов - это часть ядра операционной системы. Набор системных вызовов - это машинный интерфейс ядра.

так как OS предоставляет сотни таких системных вызовов для программ, чтобы те получали доступ к памяти, процессору и дискам, то принято говорить, что OS предоставляет **стандартную библиотеку (standard library)** для программ.
>[!important] Стандартная библиотека - это высокоуровневый набор функций для обращения к системным вызовам.

виртуализация позволяет множеству программ работать одновременно деля ресурсы системы. поэтому именно OS выполняет работу менеджера по ресурсам (*resource manager*) - она решает когда и как давать доступ программам к реальным физическим ресурсам системы.

# memory roll
модель физической памяти компьютера очень простая. память это просто последовательность байт. чтобы получить доступ к данным нужно указать адрес данных в памяти. что писать или обновлять память нужно указать данные, которые будут в нее записаны.
память всегда используется программой, когда программа работает. программа хранит все свои структуры данных в памяти. к ним она получает доступ через разные инструкции. инструкции, кстати, так же хранятся в памяти, поэтому при выполнении любой инструкции открывается доступ  памяти.
я запущу программу:
```C
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
#include <assert.h>
#include "common.h"

int main(int argc, char *argv[]) {
    int *p = malloc(sizeof(int));
    assert(p != NULL);
    printf(" (%d) ardress of p: %08x\n", getpid(), (unsigned) p);
    *p = 0;
    while (1) {
        Spin(1);
        *p = *p + 1;
        printf("(%d) p: %d\n", getpid(), *p);
    }
    return 0;
}
```
первая же строчка функции занимает память в heap - четыре байта. указатель `p` теперь показывает на это место в памяти.

## stack-smashing attacks
этот тип атак так же известен как *stack buffer overflow*. такая атака полагается на уязвимость, которая возникает, когда stack компьютерной программы или операционной системы насильно перегружают (*forced to overflow*). это может привести к некорректной работе программы и ее вылету.
stack - это область памяти, которая работает по принципу *first-in last-out (FILO)* (это та самая тема с тарелками). она не бесконечна и имеет определенный размер. основной принцип *stack smashing* заключается в заполнении stack большим количеством данных, чем он может вместить.
допустим у меня есть функция, которая по какой-то причине не проверяет размер приходящих в нее данных. когда происходит вызов такой функции в stack остается запись о том куда мне следует вернуться после завершения функции - адрес возврата. чтобы хранить данные новая функция создает свою локальную переменную. эта переменная так же хранится в stack. теперь злоумышленнику осталось только понять на каком конкретно байте памяти программы лежит адрес возврата. другими словами, ему нужно передать в функцию большое количество мусорных данных, чтобы память stack переполнилась, а затем вместе с этими данными подсунуть новый адрес функции ровно в ту ячейку памяти, где хранился старый адрес.
так, находясь в функции можно заставить программу вернуться не к изначальной вызывающей функции, а к другому участку памяти - к другому коду.

# concurrency
третья программа:
```C
#include <stdio.h>
#include <stdlib.h>
#include "common.h"

volatile int counter = 0;
int loops;

void *worker(void *arg) {
	for (int i = 0; i < loops; i++) {
		counter++;
	}
	return NULL;
}

int main(int argc, char *argv[]) {
	if (argc != 2) {
		fprintf(stderr, "usage: threads <value>\n");
		exit(1);
	}
	loops = atoi(argv[1]);
	pthread_t p1, p2;
	printf("Initial value: %d\n", counter);
	
	Pthread_create(&p1, NULL, worker, NULL);
	Pthread_create(&p2, NULL, worker, NULL);
	Pthread_join(p1, NULL);
	Pthread_join(p2, NULL);
	printf("Final value: %d\n", counter);
	return 0;
}
```
я не могу полностью ее репрезентировать на своей машине, потому что здесь используется заголовочный файл `common.h`, который использует реализованные в другом исполняемом файле функции `Pthread_create()`, `Pthread_join()` + тип `pthread_t`.
смысл же программы прост - она создает два потока (*threads*), используя `Pthread_create()`. поток - это функция, которая работает в том же пространстве памяти, что и другие функции, причем несколько таких функций могут быть активны одновременно. насколько я понимаю функция `Pthread_create()`, создает поток, который начинается с вызова функции `worker()`.
В примере из книги создается сразу два потока, каждый из которых выполняет код функции `worker()`. Для каждого потока в памяти существует собственный stack, однако они разделяют глобальные переменные, например, `counter`.
