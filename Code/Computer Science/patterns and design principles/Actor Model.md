*Actor Model* - модель, которая описывает правила взаимодействия и поведения системных компонентов, при их конкурентной работе.
*actor* - минимальная единица любого компьютерного вычисления. актор должен получать сообщение и выполнять какие-то вычисления на его основе. принцип очень похож на вызов метода в OOP - актор получает аргумент (сообщение) и выполняет логику.
главное отличие в том, что акторы должны быть абсолютно изолированными друг от друга и они не должны делить между собой память. так же актор должен иметь свое приватное состояние, которое не может быть изменено другим актором.
один актор не является актором, акторы всегда действуют в рамках системы, которая содержит других акторов.
удобно представить актора, как чувака с почтовым ящиком, которому приходит сообщение на обработку. важно понимать, что три чувака могут работать параллельно над запросами, но у них так же есть собственные очереди запросов, которые уже работают в синхронном режиме. поэтому *mailbox* является общим хранилищем для всех сообщений для актора. при этом все сообщения актору могут поступать в асинхронном режиме.

что актор делает, когда получает сообщение?
- создает еще акторов
- отправляет сообщение другим акторам
- определяет как будет выглядеть его состояние для следующего сообщения
последний пункт интересен. акторы не меняют свои внутренние состояния, они их репрезентируют для следующих сообщений иначе. т.е., сохраняя приватное состояние, они меняют внешний вид этого состояния для следующих сообщений.
например, актор получает сообщение `add(1)`, при этом его изначальное состояние равно нулю. вместо того, чтобы изменить изначальное состояние он просто определит, что для следующего получаемого сообщение его изначальное состояние будет репрезентировано, как единица.

>[!important] вообще, акторная модель чаще всего используется вместе с языком `Erlang`, где акторы называются процессами.  

 полная изоляция акторов друг от друга + наличие приватного состояния, дают всей модели отказоустойчивость (*fault tolerance*). например, у нас может быть предусмотрен супервизор в системе (который так же является актором), который при падении одного из акторов будет выполнять какие-то действия по его лечению (одна из возможных стратегий при падении актора - сбросить его до изначального состояния).
 так как любой агент по сути является отдельным организмом с почтовым ящиком, не важно где этот агент будет развернут, т.е. на одном устройстве с другими агентами или на другом устройстве. главное, чтобы в его почтовый ящик можно было класть сообщения.
