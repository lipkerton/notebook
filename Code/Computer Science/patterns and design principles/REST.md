Это набор принципов, по которым организуется сервис. Это не протокол или четкое правило - это архитектура, которая должна соблюдаться, чтобы сервис выдавал какое-то определенное свойство. В данном случае речь идет преимущественно о масштабируемости.
Принципов всего шесть.
### клиент-серверная архитектура
это первый принцип - сервис должен делегировать часть ответственности клиенту. Сервер отвечает только за функциональную составляющую приложения - он выполняет сложную логику, следит за запросами к БД, создает ответы и направляет их клиенту. Такой подход помогает вносить изменения в функциональную часть проекта, не затрагивая при этом клиентскую сторону. Однако это накладывает и ограничения. Ограничение называется "единая точка отказа". В данном случае такой точкой является сервер, т.к. если что-то случиться с ним, то весь функционал приложения для всех пользователей окажется недоступен.
### stateless
это принцип согласно которому сервер не должен хранить состояния. На практике это означает, что сервер не хранит в БД контекст общения с клиентом. Если бы у нас был statefull сервер, то мы, наоборот, открывали бы для каждого пользователя сессию и хранили бы данные, которые присылает пользователь с запросами, чтобы иметь в виду контекст общения с пользователем при создании ответов. Плюсом системы stateless является то, что состояния не нужно хранить в БД, а значит под эту информацию не потребуется пространство + не нужно тратить время на обращение к этой информации + это облегчает масштабируемость, т.к. новые сервера БД, которые мы добавляем к сервису, могут не обращаться лишний раз к старым серверам, чтобы узнать не было ли уже какого-то контекста в диалоге с данным пользователем.
### кэширование
пользователи могут одновременно обращаться к одним и тем же данным. Серверу придется им всем слать одни и те же ответы. Чтобы не тратить ресурсы сервера на создание одинаковых ответов мы можем сохранить изначальный ответ на сервере. Такое сохранение часто называют "кэшем". Например, это может быть применено, когда сервер запрашивает данные с метеостанции, чтобы ответить на запрос пользователя: чтобы не слать запросы на метеостанцию для каждого пользователя сервер закэширует первый ответ метеостанции и будет брать ответы из кэша для всех остальных пользователей. Такой подход сохраняет ресурсы сервера - можно не генерировать одинаковые ответы постоянно.
