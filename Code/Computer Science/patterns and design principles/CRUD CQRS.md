# CRUD
CRUD = **Create Read Update Delete**
![[single-model.png]]
четыре фундаментальные для приложений, которые работают с данными, операции. фундаментальные они потому, что от каждого приложения, которое получает или отдает данные, мы ждем, что оно может создавать данные в неком хранилище данных, читать данные из хранилища, обновлять данные в хранилище и удалять данные из хранилища.
это в общем все.
далее идут несколько очевидных параметров CRUD.
например, в RESTful архитектурах HTTP-методы являются прямым отражением операций CRUD - POST (= CREATE), READ (= GET), UPDATE (= PUT/PATCH), DELETE (= DELETE).
# CQRS
ресурсы:
https://martinfowler.com/bliki/CQRS.html
https://learn.microsoft.com/en-us/azure/architecture/patterns/cqrs
CRUD - это круто, но когда приложениям нужна более сложная логика, то сверху CRUD наворачивают много других интересных вещей. например, данные могут не просто записываться в БД, но еще и валидироваться перед записью или запись может происходить по более сложной логике.
однако проблемы начинаются, когда мы пытаемся раздуть простую модель тяжелыми механиками - теряется форма модели и сама архитектура.
CQRS придуман, чтобы решить эту проблему.
изначальная идея состоит в том, чтобы разделить ответственность между принципами обновления и отображения данных.
поэтому здесь на изображении можно найти сразу две модели:
![[cqrs.png]]
две модели в данном случае может означать и разделение кода на одном устройстве, и обработку операций двумя разными устройствами. смысл в том, что вся работа с отображением данных проходит через `query model`, а вся работа с обновлением и вставкой данных в хранилища происходит в `command model`. если пользователь заходит на ресурс, то `query model` берет данные, чтобы показать пользователю, а если пользователь делает какое-то изменение в данных, то изменение перенаправляется в `command model`.
так же здесь могут быть вариации. например, обе модели могут общаться с одной базой данных и в этом случае БД выступает посредником между ними, или может существовать сразу две БД для каждой модели. в последнем случае БД для `query model` может являться репликой или разрешать только функции чтения из себя, что облегчит нагрузку на основную БД для `command model`.
