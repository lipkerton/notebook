# MVC
MVC = **Model View Controller**
аббревиатура содержит перечисление трех основных компонентов архитектуры
**Model** - модель это часть архитектуры, которая должна содержать бизнес-логику приложения, данные приложения и правила манипуляции данными.
**View** - представление бизнес-логики и прочего пользователям в удобном формате (XML, HTML и прч.). *View никак не должен быть связан с Model*.
**Controller** - посредник между *View* и *Model*. именно через него проходят входящие в приложения реквесты пользователей, чтобы затем контроллер распарсил и перенаправил эти реквесты в *Model* на обработку (например, чтобы *Model* каким-то образом сохранил данные). посредником он выступает потому, что берет обработанные в *Model* данные и выбирает подходящие функции в *View* для их отображения или просто отдает в *View* для отображения.
обычный сценарий для такой модели:
пользователь высылает запрос к приложению на архитектуре *MVC*. запрос попадает в *Controller*. *Controller* парсит запрос, отправляет его в *Model*. *Model* делает с данными что-то, возвращает их в *Controller*. *Controller* отдает данные в *View* и *View* часть приложения отображает данные для пользователя.
плюсы:
- *разделение ответственности* - приложение модульное. каждый модуль = своя линия разработки.
- *удобство тестирования* - каждый компонент может быть протестирован автономно.
- *параллельная разработка* - разработка каждого модуля может проходить независимо.
недостатки:
- *сложность для простых задач* - архитектура немного занудная, т.к. все действия всегда связаны контроллером, хотя, очевидно, что если бы данные попадали от *Model* в *View* напрямую, то все было бы проще.
- *раздутый контроллер* - в контроллер частично может попасть логика *Model*. это частая проблема. теряется граница между контроллером и *Model*.
# MVP
MVP = **Model View Presenter**
это усовершенствованный MVC.
**Model** здесь по-прежнему содержит всю бизнес-логику.
**View** отличается - он больше не содержит *никакой* логики. теперь это просто пассивный интерфейс. он вообще ничего не делает.
**Presenter** ловит все действия, которые делает пользователь на внешнем интерфейсе *View*, обрабатывает с помощью *Model*, готовит их к выводу и отправляет обратно на вывод в *View*.
в этой модели компонент контроллера поменялся на компонент *Presenter*.
обычно между *Presenter* и *View* существует интерфейс, через который *Presenter* управляет *View*. этот интерфейс определен именно в *Presenter*.
обычный сценарий для такой модели:
пользователь кликает по какому-то элементу или как-то еще взаимодействует с *View*. *View* не знает, что делать, поэтому посылает действие в *Presenter*. *Presenter* обращается к *Model* за выполнением бизнес-логики, готовит ответ и обновляет *View* в соответствии с ответом.
ключевое отличие от MVP:
здесь *View* абсолютно тупая - она не знает о существовании *Model* и подчиняется интерфейсу, который реализован на стороне *Presenter*.
плюсы:
- *тестируемость* - *Model* и *Presenter* не нуждаются в *View* для проведения тестов. все тестирование логики может быть проведено без *View*.
- *устранение любой связи между Model и View* - делает компоненты архитектуры еще более независимыми.
минусы:
- *раздутый Presenter* - так как *Presenter* здесь является центром системы, имеется огромный шанс его раздуть или смешать *Model* и *Presenter*.
- *низкая реюзабельность Presenter* - так как интерфейс для управления *View* прописан в *Presenter*, это означает, что *Presenter* может общаться только с конкретным *View*.
# MVVM
MVVM = **Model View View-Model**
**Model** - все так же выполняет бизнес-логику.
**View** - пользовательский интерфейс, имеет ссылку на *View-Model*.
**View-Model** - реализует всю логику пользовательского интерфейса.
фишка модели в том, что она полностью автоматизирует процесс общения между компонентами. здесь *View* получает данные, *View-Model* подхватывает данные, обновляет состояние каких-то объектов, отправляет в *Model* вещи для обработки. *View* видит, что данные готовы и объекты обновились, берет это состояние и несет пользователю обратно.
если в MVP и MVC модели были *Presenter* и *Controller*, которые отвечали за управление *View* и *Model*, то здесь между *View* и *View-Model* установлены здоровые отношения. *View* не зависит от *View-Model*, а *View-Model* не зависит от *View*, следовательно, одна и та же *View-Model* может работать с разными *View*, потому что она просто реализует обновление собственных состояний.
стандартный сценарий:
пользователь печатает что-то на стороне *View*. из-за плотной связки между *View* и *View-Model*, на стороне *View-Model* сразу обновляется состояние - *View-Model* получает данные от пользователя. если необходимо, данные уходят на *Model*, а *View-Model* обновляет все необходимые объекты, которые будет читать *View*. *View* видит обновление и сразу "забирает" его для отображения пользователю.
