что такое виртуализация в общем случае?
представим, что в моем компьютере есть один процессор. пользоваться ресурсами процессора хотят несколько программ одновременно. как дать им возможность это делать?
ОС для каждой программы создает виртуальную версию процессора. на самом деле ресурс процессора все еще остается тем же, но каждая программа думает, что у нее он свой приватный. фишка тут в том, что программа не всегда использует процессор на полную мощность, поэтому ОС способна брать ресурс процессора и передавать между несколькими программами очень быстро. OC делит с помощью таких передач ресурс между всеми программами в компьютере.
# the abstraction: process
главная абстракция, которую дает операционная система - процесс.
*процесс* - это запущенная программа.
программа сама по себе это безжизненная штука - операционная система заставляет ее работать. но операционная система хочет запускать не одну, а несколько программ одновременно.
поэтому задача операционной системы заставить каждую программу думать, что у нее есть собственный ресурс *CPU*. то есть сделать виртуальную копию процессора для каждой программы. ОС запускает один процесс, останавливает его, запускает другой процесс, чтобы создать иллюзию существования нескольких процессоров. эта базовая техника называется *time sharing of CPU* и она позволяет пользователям запускать несколько программ конкурентно; взамен скорость работы программ может замедляться, т.к. чем больше программ работает, тем больше ресурсов процессора нужно распределить.
чтобы реализовать виртуализацию CPU используются низкоуровневые (*low-level machinery*) и высокоуровневые (*high-level intelligence*) штуки. низкоуровневые штуки называются *mechanisms* - это низкоуровневые методы и протоколы, которые имплементируют требуемую функциональность в систему. высокоуровневые штуки называются *policies* - это алгоритмы, которые нужны ОС, чтобы принимать решения. например, "какую программу должна выполнять система, если у нее N ресурсов процессора?" - такое решение в операционной системе принимает *scheduling policy*.
как было сказано ранее, процесс является запущенной программой, поэтому, чтобы понять что такое процесс можно суммировать все те действия, которые выполняет программа, пока работает. чтобы понять, из чего складывается процесс нужно разобраться в его *machine state*: что программа может читать или обновлять пока работает.
память - самый очевидный компонент машинного состояния. сама программа (ее инструкции) лежит в памяти и она же использует память для чтения и обновления. поэтому та память, к которой программа может получить доступ (*address space*), является частью машинного состояния.
так же регистры являются частью машинного состояния. *регистр* - это сверхбыстрая ячейка памяти, к которой процессор имеет доступ. там он хранит компоненты, которые ему нужны для выполнения инструкций, например, *programm counter (PC) (instruction pointer (PC))* говорит о том какая инструкция программы выполняется в текущий момент; *stack pointer* хранит адрес вершины стека.
программы так же имеют доступ к постоянной памяти.
# process api
какие действия должна выполнять каждая операционная система?
точнее *какие действия должна ОС позволять совершать пользователям?* другими словами - нужно понимать какие основные механики есть у любой ОС.
*Create* - каждая операционная система должна иметь метод для создания нового процесса. такой метод вызывается, например, при двойном клике на иконку приложения или при вводе команды в оболочку.
*Destroy* - если есть интерфейс создания процесса, то должен быть и интерфейс его уничтожения. ОС не берет на себя ответственность за каждое завершение программы, т.к. обычно программа сама завершается на каком-то этапе, но ОС имеет средства для вынужденного завершения процесса.
*Wait* - метод приостановки процесса.
*Miscellaneous Control* - ОС вместо силового завершения процесса может позволить произвести с ним еще ряд каких-то действий, например, предоставить метод для временной приостановки и метод для продолжения работы.
*Status* - почти всегда есть интерфейс, который позволяет посмотреть статус процесса (характеристики процесса).
# process creation
а как вообще программы превращаются в процессы? как именно текст программы становится запущенным процессом?
первым делом операционная система должна загрузить весь код программы и любые статические данные (я так понимаю тут речь идет о переменных окружения, инициализированных и неинициализированных глобальных переменных) в память, в *address space* процесса. программы изначально лежат на диске в каком-то *исполняемом формате*, поэтому процесс загрузки программы и статических данных в память состоит из чтения ОС байтов программы с диска и размещения их в памяти.
в ранних операционных системах процесс загрузки программы делался *eagerly* (резво + резко?), т.е. все шаги выполнялись сразу вместе, перед запуском программы. современная операционная система делает это лениво (*lazily*) - она загружает кусочки кода или данных только тогда, когда они нужны во время исполнения программы. чтобы полностью понять как это работает нужно сначала освоить механики (machinery) *paging* и *swapping*.
после того как текст программы и статические данные загружены в память, нужно так же выделить место в памяти под *run-time stack* (или просто *stack*) программы. stack используется (в С) для хранения локальных переменных, параметров функции и адресов возврата. так же ОС скорее всего (зависит от программы) инициализирует stack с уже готовыми аргументами для функции-точки-входа в программу `main()` - `argc` (количество передаваемых параметров в программу) и `argv` (сами аргументы).
операционная система так же создает память для *heap* программы. heap память в программах на С используется для *excplicitly requested dynamically-allocated data*. программы запрашивают такое пространство через вызовы функций `malloc()` и `free()`. heap память не очень велика изначально, но программа увеличивает ее размер через `malloc()`; эта же функция заставляет ОС вмешиваться в ход программы, чтобы удовлетворить просьбу на увеличение памяти.
операционная система так же делает несколько инциализирующих задач, которые связаны с *I/O*. например, в UNIX системах каждый процесс имеет по умолчанию три открытых *file descriptors*, для стандартного потока вывода, стандартного потока ввода и потока вывода ошибок.
после всего этого ОС находит точку входа в программу - функцию `main()` (этот переход к функции вообще принято называть прыжком + он совершается с помощью специального механизма, о котором я поговорю позднее). начиная работу функции ОС передает контроль над CPU созданному процессу и программа начинает выполнять инструкции.

# process states
какие вообще могут быть состояния у процесса?
процесс может находится в определенный момент времени в одном из трех состояний:
- *running* - в *running* состоянии процесс выполняет свою работу на процессоре.
- *ready* - в этом состоянии процесс готов к выполнению инструкций на процессоре, но по какой-то причине ОС решает не запускать его в этот момент.
- *blocked* - в заблокированном состоянии процесс выполняет какую-то операцию, что делает его не готовым к запуску, пока не произойдет определенное событие. например, когда процесс выполняет запись на диск, то он становится заблокированным и другие процессы могут занять процессор.

# data structures
несмотря на то, что ОС - это особенная программа, она все еще является программой, поэтому у нее есть какие-то структуры данных для отслеживания разных кусочков информации.
чтобы отслеживать состояние каждого процесса у ОС существует *process list* - коллекция  всех текущих процессов в состоянии готовности, в состоянии работы и в заблокированном состоянии.
для хранения всех подобных вещей ОС использует структуры.
пример структуры для учебной ОС *xv6*:
```C
struct context {
	int eip;
	int esp;
	int ebx;
	int ecx;
	int edx;
	int esi;
	int edi;
	int edp;
}
```
каждая из этих строк описывает параметры с которыми процесс останавливается и с которыми он должен быть перезапущен затем. ОС записывает и сохраняет эти параметры для корректного возобновления работы процесса. так же этот набор параметров является набором регистров процессора, т.е. набор ключевых значений, к которым процессор должен получить доступ сразу же, как только ему это будет необходимо.
```C
enum proc_state {
	UNUSED, EMBRYO, SLEEPING, RUNNABLE, RUNNING, ZOMBIE
}
```
коллекция из возможных состояний процесса:
- `UNUSED` - слот в таблице процессов свободен (его можно занять другим процессом).
- `EMBRYO` - процесс только что создан, но еще не готов к выполнению.
- `SLEEPING` - процесс заблокирован и ждет какого-то события.
- `RUNNABLE` - процесс готов к выполнению и ждет, когда планировщик выделит ему CPU.
- `ZOMBIE` - процесс завершился, но запись из таблицы процессов еще не удалена.
```C
struc proc {
	char *mem;    // указатель на начало памяти процесса (виртуальный адрес).
	uint sz;      // размер памяти процесса в байтах.
	char *kstack; // указатель на дно стека ядра для этого процесса.
	enum proc_state state; // текущее состояние процесса.
	int pid;               // идентификатор процесса.
	struct proc *parent;   // указатель на родительский процесс.
	void *chan;            // если процесс спит, то на каком канале (адрес процесса).
	int killed;            // флаг процесс был убит (потому что ему был послан сигнал).
	struct file *ofile[NOFILE]; // массив открытых файлов.
	struct inode *cwd;          // текущая рабочая директория.
	struct context context;     // сохраненный контекст регистров для переключения.
	struct trapframe *tf        // указатель на кадр прерывания.
}
```
эта структура отражает параметры, которые ОС *xv6* отслеживает для каждого процесса.
здесь есть пара интересных моментов:
во-первых, указатель *kstack* показывает не дно пользовательского stack, а дно stack, который принадлежит ядру и является полностью автономной сущностью. отдельный stack нужен для безопасности - когда код выполняется в режиме ядра, то процесс пользуется отдельным stack ядра, чтобы случайно не выполнить какой-нибудь вредоносный код из пользовательского stack.
во-вторых, прямо тут указаны переменные для хранения информации о работе с файлами. например, `cwd` указывает на текущую рабочую директорию процесса, что поможет, если программа вдруг захочет использовать относительный, а не абсолютный путь.
в-третьих, стоит обратить внимание на `ZOMBIE` статус процесса. он используется, например, когда процесс завершился, но другим процессам нужно проанализировать его код возврата. если анализ больше не нужен, то родительский процесс (указатель на который лежит в `parent`) сообщает методом `wait()`, чтобы подождать пока ОС завершит и очистит завершенный процесс.
