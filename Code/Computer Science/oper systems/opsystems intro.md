это заметка-конспект по книжке *operating systems: three easy systems*. по главе *introduction to operating systems*.
запуск программы = выполнение инструкций. миллионов и сейчас уже миллиардов инструкций в секунду.
процессор *цепляет (fetches)* инструкцию из памяти, *декодирует (decodes)* инструкцию (т.е. пытается понять какая именно это инструкция) и *исполняет (executes)* ее.
я сначала не понял зачем процессор декодирует инструкцию, если на предыдущем шаге он ее вытягивает из памяти. на самом деле, похоже, что процессор вытягивает код инструкции, а не саму инструкцию, поэтому ему еще нужно декодировать этот код.
механизм *fetch-decode-execute* звучит просто, но на самом деле под капотом вместе со всеми этими процессами происходит еще куча всего, чтобы сделать систему *easy-to-use*. существует тело ПО (*body of software*), которое ответственно, например, за запуск нескольких программ одновременно, за распределение памяти для нескольких программ, за связь между программами и устройствами и т.д. *the body of software* = *operating system (OS)*
главный инструмент, которым пользуется OS для реализации *easy-to-use system*, - **виртуализация (virtualization)**. в общем случае виртуализация - это когда OS берет физический ресурс в системе (процессор, память или диск) и превращает его в более общую (general), мощную (powerful), easy-to-use виртуальную форму самого себя. поэтому иногда мы обращаемся к оперативной системе, как к **виртуальной машине (virtual machine)**.
чтобы дать пользователям возможность пользоваться оперативной системой виртуальной машины (запускать программы, отдавать команды и т.д.) так же реализуются *интерфейсы (APIs)*. пользователи обращаются к интерфейсам (*to make a call*) - пользователь управляет программой, программа обращается к интерфейсу, чтобы создать **системный вызов (system call)**.
>[!important] Набор системных вызовов - это часть ядра операционной системы. Набор системных вызовов - это машинный интерфейс ядра.

так как OS предоставляет сотни таких системных вызовов для программ, чтобы те получали доступ к памяти, процессору и дискам, то принято говорить, что OS предоставляет **стандартную библиотеку (standard library)** для программ.
>[!important] Стандартная библиотека - это высокоуровневый набор функций для обращения к системным вызовам.

виртуализация позволяет множеству программ работать одновременно деля ресурсы системы. поэтому именно OS выполняет работу менеджера по ресурсам (*resource manager*) - она решает когда и как давать доступ программам к реальным физическим ресурсам системы.

# memory roll
модель физической памяти компьютера очень простая. память это просто последовательность байт. чтобы получить доступ к данным нужно указать адрес данных в памяти. что писать или обновлять память нужно указать данные, которые будут в нее записаны.
память всегда используется программой, когда программа работает. программа хранит все свои структуры данных в памяти. к ним она получает доступ через разные инструкции. инструкции, кстати, так же хранятся в памяти, поэтому при выполнении любой инструкции открывается доступ  памяти.
я запущу программу:
```C
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
#include <assert.h>
#include "common.h"

int main(int argc, char *argv[]) {
    int *p = malloc(sizeof(int));
    assert(p != NULL);
    printf(" (%d) ardress of p: %08x\n", getpid(), (unsigned) p);
    *p = 0;
    while (1) {
        Spin(1);
        *p = *p + 1;
        printf("(%d) p: %d\n", getpid(), *p);
    }
    return 0;
}
```
первая же строчка функции занимает память в heap - четыре байта. указатель `p` теперь показывает на это место в памяти.

## stack-smashing attacks
этот тип атак так же известен как *stack buffer overflow*. такая атака полагается на уязвимость, которая возникает, когда stack компьютерной программы или операционной системы насильно перегружают (*forced to overflow*). это может привести к некорректной работе программы и ее вылету.
stack - это область памяти, которая работает по принципу *first-in last-out (FILO)* (это та самая тема с тарелками). она не бесконечна и имеет определенный размер. основной принцип *stack smashing* заключается в заполнении stack большим количеством данных, чем он может вместить.
допустим у меня есть функция, которая по какой-то причине не проверяет размер приходящих в нее данных. когда происходит вызов такой функции в stack остается запись о том куда мне следует вернуться после завершения функции - адрес возврата. чтобы хранить данные новая функция создает свою локальную переменную. эта переменная так же хранится в stack. теперь злоумышленнику осталось только понять на каком конкретно байте памяти программы лежит адрес возврата. другими словами, ему нужно передать в функцию большое количество мусорных данных, чтобы память stack переполнилась, а затем вместе с этими данными подсунуть новый адрес функции ровно в ту ячейку памяти, где хранился старый адрес.
так, находясь в функции можно заставить программу вернуться не к изначальной вызывающей функции, а к другому участку памяти - к другому коду.

# concurrency
третья программа:
```C
#include <stdio.h>
#include <stdlib.h>
#include "common.h"

volatile int counter = 0;
int loops;

void *worker(void *arg) {
	for (int i = 0; i < loops; i++) {
		counter++;
	}
	return NULL;
}

int main(int argc, char *argv[]) {
	if (argc != 2) {
		fprintf(stderr, "usage: threads <value>\n");
		exit(1);
	}
	loops = atoi(argv[1]);
	pthread_t p1, p2;
	printf("Initial value: %d\n", counter);
	
	Pthread_create(&p1, NULL, worker, NULL);
	Pthread_create(&p2, NULL, worker, NULL);
	Pthread_join(p1, NULL);
	Pthread_join(p2, NULL);
	printf("Final value: %d\n", counter);
	return 0;
}
```
я не могу полностью ее репрезентировать на своей машине, потому что здесь используется заголовочный файл `common.h`, который использует реализованные в другом исполняемом файле функции `Pthread_create()`, `Pthread_join()` + тип `pthread_t`.
смысл же программы прост - она создает два потока (*threads*), используя `Pthread_create()`. поток - это функция, которая работает в том же пространстве памяти, что и другие функции, причем несколько таких функций могут быть активны одновременно. насколько я понимаю функция `Pthread_create()`, создает поток, который начинается с вызова функции `worker()`.
В примере из книги создается сразу два потока, каждый из которых выполняет код функции `worker()`. Для каждого потока в памяти существует собственный stack, однако они разделяют глобальные переменные, например, `counter`.

спустя время я написал примерно похожую программу. и удивительно то, что она каждый раз возвращает разные результаты. но еще более удивительно, что раз в век она *может* вернуть корректный результат.
на самом деле причиной такого странного поведения является то, что программа, чтобы увеличить счетчик выполняет сразу три инструкции: берет из памяти текущее значение счетчика в регистр, обновляет значение, сохраняет новое значение в памяти. операция инкремента не является атомарной, поэтому, когда два потока обновляют одну и ту же область памяти, какая-то операция увеличения может потеряться.
как?
поток A берет значение `counter` из памяти и допустим оно равно $5$. в этот момент планировщик (что такое планировщик?) останавливает работу потока А и передает управление потоку Б, который делает все свои три операции - значение `counter` увеличивается на единицу. затем планировщик возвращается к потоку А, который продолжает работу с места, на котором остановился, делая следующим шагом обновление переменной. проблема в том, что поток А забрал из памяти значение $5$, а поток Б уже увеличил в памяти переменную на единицу. следовательно, когда А прибавит единицу и положит новое значение в память получится, что он положил в память число $6$, точно так же как сделал поток Б. так потеряется единица.
это называется **race condition**. два потока соревнуются за один ресурс.
# persistence
*persistence = постоянство*.
в системной памяти данные могут запросто потеряться, так как устройства (такие как DRAM) хранят данные в энергозависимой (*volatile*) манере, т.е. только пока есть электричество. поэтому нужно ПО и железо, которые будут хранить данные постоянно.
железо для этой задачи выступает в формате *input-output* устройства (*I/O*) - *hard drive* и *solid-state drives*.
ОС не создает приватную виртуальную версию жесткого диска для каждой программы. такое поведение намеренно - часто мы находимся в ситуации, когда необходимо делиться данными в файлах. например, когда я пишу программу на C, я затем передаю ее текст в компилятор `gcc`, а тот возвращает мне исполняемый файл программы. все эти передачи возможны только благодаря тому, что жесткий диск делиться данными с многими программами.
еще одна программа:
```C
#include <unistd.h>
#include <assert.h>
#include <fcntl.h>
#include <sys/stat.h>

int main(int argc, char *argv[]) {
	int fd = open("/tmp/file", O_WRONLY | O_CREAT | O_TRUNC, S_IRWXU);
	assert(fd > -1);
	int rc = write(fd, "hello world\n", 13);
	assert(rc == 13);
	close(fd);
	return 0;
}
```
в ней существует три функции, которые выполняют системные вызовы: `open()`, `write()`, `close()`. каждая из этих функций обращается к части операционной системы, которая называется *file system*.
каким конкретно образом система делает такую работу - совершенно другой разговор. и это большой + страшный разговор. он напрямую затрагивает тему *device drivers*. драйвер - это код, который написан специально для взаимодействия ОС и конкретного устройства.
# kernel и user
во всех существующих системах существует разделение на область ядра и пользовательскую область. оно появилось из простой проблемы - операционная система это программа, которая может управлять устройствами и запускать программы. программы могут пользоваться кодом операционной системы свободно, следовательно, каждая программа может получить доступ к любому устройству и любым данным в системе.
так появилась идея системного вызова - если ранее можно было просто вызывать функции из библиотеки ОС, чтобы управлять системой, то теперь этот переход от программы к управлению системой было решено сделать более формальным.
ключевое отличие нового подхода в том, что системный вызов передает управление ОС, повышая уровень аппаратных привилегий (уровень доступа программы) - *hardware privilege level*. программы же живут в пользовательском режиме - *user mode*. в пользовательском режиме они не могут самостоятельно взаимодействовать с железом компьютера. например, они не могут получить доступ к физической памяти или отправить пакет в сеть.
пример:
у меня есть программа. сначала я прошу у ОС запустить программу. ОС выполняет подготовительные действия, переключает процессор в пользовательский режим и передает управление точке входа в программу. программа начинает автономную работу - процессор выполняет ее инструкции. если встречается какая-то функция, которая требует сделать системный вызов, то программа просит у ОС сделать его; "просьба" = вызов *trap*. после вызова управление переходит к ОС, а уровень прав повышается до *kernel mode* и процессор в этом режиме может выполнять системный вызов. после выполнения ОС вызывает *return-from-trap*, привилегии понижаются, процессор снова выполняет инструкции программы в *user mode*.
у меня возник вопрос: если процессор в один момент работы функционирует в *user mode*, а затем переходит в *kernel mode*, чтобы выполнять другие инструкции, то как быть с другими программами? что с ними происходит?
ответ простой - на самом деле процессор может функционировать только в одном режиме, но он так быстро переключается, чтобы использовать все свои ресурсы, что несколько программ могут работать одновременно. это идея *multiprogramming* появилась потому, что процесс *I/0* (записи данных на устройства) медленный, поэтому пока выполняется запись процессору бы лучше заняться какими-то другими делами.
из-за того, что процессор может выполнять инструкции нескольких программ появилась проблема *memory protection* - нельзя, чтобы программы получали доступ к памяти друг друга.
