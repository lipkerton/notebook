здесь будет обсуждаться процесс создания процесса в UNIX системе.
во всем этом будут участвовать три функции - `fork()`, `create()`, `wait()`. UNIX предоставляет эти системные вызовы специально для создания отдельных процессов.
# the `fork()` system call
системный вызов `fork()` используется, чтобы создать новый процесс.
это самая *странная* рутина, которая когда-либо использовалась мной.
я напишу программу:
```C
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

int main(int argc, char *arhv[]) {
    printf("Hello world (pid: %d)\n", (int) getpid());
    int rc = fork();
    if (rc < 0) {
        fprintf(stderr, "fork failed\n");
        exit(1);
    } else if (rc == 0) {
        printf("Hello, I am a child (pid: %d)\n", (int) getpid());
    } else {
        printf("Hello, I am parent of %d (pid: %d)\n", rc, (int) getpid());
    }
    return 0;
}
```
первая строка функции `main()` печатает в терминал номер текущего процесса, который возвращает функция `getpid()`. затем я использую функцию `fork()` и записываю ее результат в переменную.
что вернется в переменную?
в переменную вернется одно из чисел - `-1, 0`. `-1` сообщает об ошибке создания потока, поэтому в первом условии `if` я проверяю, что поток был создан корректно. `0` сообщает, что мы находимся в дочернем потоке уже, поэтому следующее условие должно при успехе сообщить мне номер этого потока.
если ни одно из условий не подходит, то я делаю вывод (в конструкции `else`), что я нахожусь в родительском потоке.
вывод программы будет таким:
![[cpu.3.pnh.PNG]]
во-первых, следует сразу отметить, что `rc` при нахождении в родительском потоке содержит PID дочернего процесса.
во-вторых, я могу понять из кода, что системный вызов `fork()` создает процесс, который является почти точной копией родительского процесса. с точки зрения операционной системы теперь запущенны две копии программы и обе сейчас вернуться из системного вызова `fork()`.
как можно заметить, сообщение `Hello world...` печатается в терминале единожды, что означает, что дочерний процесс не выполняет функцию `main()` с самого начала, а продолжает работать с того места, где был создан. магия такова, что после вызова метода `fork()` появляется две новые ветви - родительская и дочерняя; обе возвращаются из функции `fork()` так, будто бы вызвали ее сами, и продолжают работу.
я сказал, что была создана *почти* полная копия процесса. в чем они различаются?
следует всегда помнить, что для нового процесса создается отдельное пространство в памяти, отдельные регистры (в т.ч. отдельный *instruction counter*) и т.д.
если бы у меня в компьютере стоял процессор с одним CPU, то вывод программы был бы непостоянным, т.к. CPU может выполнять только один процесс в стандартном потоке бы чередовались строки от запуска к запуску. такое поведение определяется планировщиком CPU - он решает когда запустить одну программу, а когда другую.
>[!important] хотя я называю это *планировщиком CPU* - это все еще часть OC, т.к. именно она решает, когда останавливать одни программы и запускать другие.
# adding `wait()` system call
довольно часто бывает полезно, чтобы родитель дождался выполнения работы дочерним процессом. заставить родительский процесс ждать можно с помощью функции `wait()`.
```C
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/wait.h>

int main(int argc, char *argv[]) {
	printf("Hello, world (pid: %d)\n", (int) getpid());
	int rc = fork();
	if (rc < 0) {
		fprintf(stderr, "fork failed\n");
		exit(1);
	} else if (rc == 0) {
		printf("Hello, world, I am a child (pid: %d)\n", getpid());
	} else {
		int wc = wait(NULL);  // здесь я жду завершения дочернего потока.
		printf("Hello, I am a parent of %d (wc: %d) (pid: %d)\n", rc, wc, getpid());
	}
	return 0;
}
```
я вызываю `wait()`, чтобы отложить выполнение родительского процесса до момента, пока дочерний не будет выполнен. когда дочерний завершится, то `wait()` вернет его идентификатор в родительский процесс.
вызов метода `wait()` делает вывод предетерминированным - даже если я запускал бы программу на одном процессоре, дочерний поток все равно выводил бы свое сообщение раньше конечного сообщения, т.к. родительский поток всегда первой инструкцией ждет завершения дочернего.
# finally `exec()` system call
рутина `exec()` помогает вызвать другие программы внутри текущей.
я написал вторую маленькую программу:
```C
#include <stdio.h>
#include <unistd.h>

int main() {
    printf("Привет, я другая программа (pid: %d)\n", getpid());
    return 0;
}
```
и теперь я должен внутри кода основной программы вызвать эту программу с помощью `exec()`. проблема только в том, что существует целое семейство рутин `exec()`.
я буду использовать `execvp()`. буквы `vp` в конце означают *vector* и *path*:
- *vector* - аргументы внутрь вызываемой программы передаются в виде массива строк.
- *path* - функция будет искать программу, используя переменную окружения `PATH` и, как я понял, это поможет ей найти программу, если я передам на вход не абсолютный, а относительный путь.
все это означает, что мне нужен массив с именем моей новой маленькой программы:
```C
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <sys/wait.h>

int main(int argc, char *arhv[]) {
    printf("Hello world (pid: %d)\n", (int) getpid());
    int rc = fork();
    if (rc < 0) {
        fprintf(stderr, "fork failed\n");
        exit(1);
    } else if (rc == 0) {
        printf("Hello, I am a child (pid: %d)\n", (int) getpid());
        char *args[2];                   // объявляю массив.
        args[0] = strdup("./side_kick"); // передаю в массив указатель на строку.
        args[1] = NULL;                  // передаю в массив NULL - аргументов не будет.
        execvp(args[0], args);           // запускаю программу с переданным массивом.
        printf("this shouldn't print out\n");
    } else {
        int wc = wait(NULL);
        printf("Hello, I am parent of %d (wc: %d) (pid: %d)\n", rc, wc, (int) getpid());
    }
    return 0;
}
```
появилась функция `strdup()` - она принимает на вход строку, применяет функцию `malloc()` и возвращает указатель на пространство в heap памяти, где была размещена строка.
этот указатель я передаю первым параметром в массив. вторым параметром в массив я передаю `NULL`, т.к. моя программа не принимает никакие аргументы на вход.
после создания массива, я передаю его на вход функции `execvp()`.
тут есть странность - я храню в массиве имя программы и `NULL`, но зачем-то мне нужно явно указать в качестве первого аргумента первый элемент массива (относительный путь до программы) и только после передать сам массив. зачем я делаю так? а все потому, что речь идет не про массив, а про указатель на начало массива и функция не предусматривает автоматическую распаковку.
но даже если я сделаю так:
```C
execvp(args[0], args[1]);
```
то компилятор начнет издеваться:
![[cpu4.png]]
а все потому, что `execvp()` вторым аргументом принимает указатель на массив, а не конкретный аргумент для запускаемой программы.
когда я передам все корректно, то вывод будет таким:
![[cpu5.png]]
национальная гордость - программа_1 запускает программу_2.
кстати.
здесь есть немного странная особенность.
программа_2 запускается внутри программы_1 с тем же PID, который имеет программа_1. то есть формально запускается только программа_1.
как оказывается функция `exec()` более делулу, чем `fork()`.
как она работает?
чтобы это понять следует обратить внимание на этот кусок скрипта:
```C
printf("this shouldn't print out\n");
```
в выводе выше он и не был отражен. это правильно. но как именно это произошло?
дело в том, что функция `exec()` никогда ничего не возвращает, потому что на самом деле она полностью перезаписывает код вызывающей программы кодом вызываемой программы в памяти. так же она обнуляет heap и stack память, заменяет глобальные и статические переменные. то есть она буквально превращает программу_1 в программу_2.
как видно из вывода выше, не поменялся только PID, равно как и то, что новая программа все еще является дочерней по отношению к изначально вызвавшей ее программе.
# why so?
почему это вообще так странно работает?
казалось бы, чтобы создать новый процесс (другую работающую программу), нам не нужно делить методы `fork()` и `exec()` - достаточно сделать единый метод, который будет сразу и создавать отдельный процесс, и запускать в нем программу.
однако такой подход менее гибкий - во1 пришлось бы передавать слишком много параметров в одну единственную функцию, во2 нельзя было бы выполнять код между созданием нового процесса и до активации другой программы (то есть после `fork()` и до `exec()`).
в UNIX системах разделение `fork()` и `exec()` является основообразующим для многих программ, потому, что через это разделение программа *shell* может выполнять большое количество интересных махинаций.
*оболочка (shell)* сама является программой. она показывает пользователю строку ввода (*prompt*), пользователь может ввести команду, *shell* в большинстве случаев найдет исполняемый файл, на который ссылается команда, сделает отдельный процесс с помощью `fork()`, затем вызовет какой-то вариант `exec()` и подождет результата с помощью `wait()`.
однако!
лучшим примером фокуса в оболочке с разделением `fork()` и `exec()` является механизм записи в файл:
```Bash
echo "hello world!" > text.txt
```
результат работы программы `echo` будет перенаправлен в файл `text.txt`.
однако на практике выполняется такая последовательность - создается процесс с помощью `fork()`, стандартный поток вывода меняется на вывод в файл и выполняется команда `exec()`. то есть оболочка просто заменила стандартный вывод на вывод в файл в отдельном процессе.
